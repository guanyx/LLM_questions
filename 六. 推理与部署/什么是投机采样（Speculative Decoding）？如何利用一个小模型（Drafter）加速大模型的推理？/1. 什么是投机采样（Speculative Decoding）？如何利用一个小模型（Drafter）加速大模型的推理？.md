# 什么是投机采样（Speculative Decoding）？如何利用一个小模型（Drafter）加速大模型的推理？

在使用各种大语言模型（LLM）产品时，你是否感觉到了“等待”的焦灼？看着屏幕上的字像老式打字机一样一个一个蹦出来，尽管背后的模型由于参数量巨大而显得“智慧无穷”，但这种逐字生成的龟速体验，始终是制约大模型落地的一道坎。

为什么大模型推理这么慢？有没有办法让它快起来？

一种名为**投机采样（Speculative Decoding）**的技术给出了答案。它巧妙地引入了一个“小助手”，在保证输出质量完全不变的前提下，实现了推理速度的成倍提升。

本文将剥离复杂的数学公式和代码实现，带你从原理到本质，彻底看懂这项技术的精妙之处。

---

## 一、 为什么大模型推理这么慢？

要理解投机采样，首先得理解大模型生成的本质瓶颈。

大语言模型的生成方式是**自回归（Autoregressive）**的。简单说，就是“接龙”。模型必须先写出第一个字，才能基于第一个字写第二个字，基于前两个字写第三个字……

这就好比你在过一条独木桥，每一步都必须踩实了，才能迈出下一步。你无法同时迈出五步。

对于计算机而言，这种模式带来了一个巨大的浪费：**内存带宽（Memory Bandwidth）**。

想象一下，大模型的参数（Weights）就像是存放在仓库（显存）里的几百吨原材料。每次生成一个字，工厂（GPU 计算单元）都需要把这几百吨原材料从仓库搬到车间，仅仅是为了生产一颗螺丝钉（一个 Token）。生产完这颗螺丝钉，原材料又得搬回去或者原地待命，等待下一个指令。

- **搬运时间（读取权重）**：很长。
- **加工时间（计算）**：很短。

GPU 强大的并行计算能力在这种“搬运一次，计算一次”的模式下无处施展。它大部分时间都在“等货”，而不是在“干活”。这就是著名的**内存墙（Memory Wall）**问题。

## 二、 投机采样的直觉：主编与实习生

既然“搬运一次”成本很高，那能不能“搬运一次，生产一批”呢？

但在自回归模式下，不知道前一个字，就不知道后一个字，怎么批量生产？

这时候，**投机采样**的思路来了。它引入了一个新角色：**小模型（Drafter / Small Model）**。

我们可以把大模型（Target Model）比作一位**才华横溢但下笔缓慢的资深主编**，把小模型比作一位**手速极快但水平一般的实习生**。

传统的推理模式是：

> **主编**亲自执笔，深思熟虑，写完第一个词，再想第二个词。质量顶尖，但速度感人。

投机采样的模式是：

1.  **实习生（Drafting）**：实习生先快速地写一个草稿，比如一口气猜了 5 个词：“今天、天气、真、不错、啊”。因为实习生“脑容量”小（参数少），跑得飞快。
2.  **主编审阅（Verification）**：主编把这 5 个词拿过来，**一眼扫过**（并行计算）。
    - 主编心想：“‘今天天气真’这三个词写得对，跟我设想的一样。但‘不错’这个词太俗了，应该用‘晴朗’。”
3.  **结果输出**：于是，系统直接输出了“今天天气真”这三个词，并把第四个词改为“晴朗”。

**关键点在于：**
主编“审阅”5 个词的时间，和他“亲自写”1 个词的时间，其实是差不多的！

因为对于 GPU 来说，把几百吨参数搬到车间后，顺手计算 5 个词的概率，和计算 1 个词的概率，在计算量上虽然增加了，但在时间成本（主要是搬运成本）上几乎可以忽略不计。

**这就是投机采样的核心魔法：用小模型的“速度”换取大模型的“并行度”。**

## 三、 深入技术细节：它是如何运作的？

让我们把视角稍微拉近一点，看看这个过程在技术上是如何严丝合缝地运转的。

### 1. 两个角色的博弈

- **Drafter（草稿模型/策略）**：它的任务不是“完美”，而是“像”。它可以是一个参数量极小的小模型，也可以是基于 N-gram 的查找算法，甚至是模型自身的非自回归头。
- **Target（目标大模型）**：这是我们真正想要使用的模型。它拥有最终解释权。

### 2. “验证”的并行化：为什么校验比生成快？

这是最反直觉的地方，也是你疑惑的核心：**“校验不也是生成吗？为什么不需要像生成那样一个个搬运权重？”**

其实，校验的过程更像是**“批改作业”**，而不是“做作业”。

- **普通生成（做作业）**：
  你不知道下一题的答案是什么。你必须先做完第一题，根据第一题的结果，才能去做第二题。这是一个**串行**的过程。

  - 生成 $Token_1$ -> **加载一次权重** -> 得到 $Token_1$。
  - 生成 $Token_2$ -> **再加载一次权重** -> 得到 $Token_2$。
  - ...
  - 生成 5 个词，就要**加载 5 次权重**。

- **并行验证（改作业）**：
  实习生已经把 5 个词 $[T_1, T_2, T_3, T_4, T_5]$ 全写在纸上了。
  Transformer 架构有一个神奇的特性（也就是训练时的 Teacher Forcing 模式）：**只要输入是已知的，它就可以一次性并行计算出所有位置的概率。**

  大模型只需要**加载 1 次权重**，把这 5 个词作为一个**矩阵（Batch）** 输入进去：

  - 它能同时计算：
    - 看到 $Input$，觉得 $T_1$ 对不对？
    - 看到 $Input + T_1$，觉得 $T_2$ 对不对？
    - ...
    - 看到 $Input + ... + T_4$，觉得 $T_5$ 对不对？

  **关键点：** GPU 搬运几百吨的权重非常慢，但一旦搬到了计算单元，它的计算能力是极其恐怖的。处理“1 个词”和处理“5 个词”，对 GPU 来说，**计算时间的差异微乎其微**（因为权重加载才是大头）。

  所以，投机采样把 **“搬 5 次家，算 5 次”** 变成了 **“搬 1 次家，算 5 次”**。这就省下了 4 次搬家的时间。

### 3. 拒绝与修正（Rejection & Correction）

如果实习生猜错了怎么办？
投机采样拥有一套严格的数学机制（基于拒绝采样 Rejection Sampling）。
大模型会逐个检查实习生的草稿。一旦发现某个词的概率分布不符合大模型的标准（比如实习生写了“苹果是蓝色的”，而大模型认为“红色”概率更高），大模型就会在那个位置**截断**，丢弃后面所有实习生生成的词，并亲自生成正确的词。

**最重要的一点：**
**投机采样是无损的（Lossless）。**
这听起来可能有点反直觉：**既然有一部分是“水平一般”的实习生写的，为什么最终结果和大模型“亲自写”的完全一样？**

这是因为大模型拥有“一票否决权”和“修正权”。
投机采样并不是简单地保留实习生的草稿。它使用了一种严谨的数学方法（拒绝采样）。简单来说：

1.  实习生提出一个词。
2.  大模型计算这个词在自己心中的概率。
3.  如果这个词在大模型看来概率也很高，就保留；如果概率很低（不符合大模型的分布），就**拒绝**，并根据大模型自己的概率重新生成一个新的词。

**结果就是：** 虽然草稿是实习生打的，但所有“不合适”的词都被大模型剔除或修正了。最终留下的文字，从统计分布上来看，就像是大模型亲自写的一样。用户根本感觉不到实习生的存在，除了速度变快了。

## 四、 独到见解：算力换时间的经济学

投机采样不仅仅是一个技术技巧，它折射出大模型时代的一种新的计算经济学。

### 1. 只有“猜得准”才划算（由盈转亏的临界点）

你可能会问：**“如果实习生第一个词就猜错了，那是不是比平时更慢？”**
**是的，你的直觉非常敏锐。**

- **猜对了**：实习生花 1ms 猜词，主编花 10ms 验证，输出了 5 个词。平均每个词 2ms。 -> **大赚**。
- **猜错了**：实习生花 1ms 猜词，主编花 10ms 验证，发现第一个词就是错的，全部推翻重写。总耗时 11ms，只产出了 1 个词。而平时主编自己写只需要 10ms。 -> **亏本**。

这就是投机采样的**额外开销（Overhead）**。如果小模型和大模型极其不合拍（接受率极低），投机采样反而会拖慢推理速度。
所以，投机采样是一场“赌博”，赌注是“小模型的推理时间”，赢面是“接受率”。

### 2. 从“算力瓶颈”到“带宽瓶颈”

在深度学习早期，我们总担心 GPU 算力（FLOPS）不够。但在大模型推理阶段，真正的瓶颈往往是显存带宽（Bandwidth）。GPU 里的计算单元经常在“摸鱼”，等待数据从显存送过来。
投机采样实际上是**给闲置的计算单元找活干**。既然都要搬运一次权重，不如顺便多验算几个词。这是一种对硬件资源极致压榨的智慧。

### 3. 小模型的选择至关重要

投机采样并非万能。它的加速效果取决于**接受率（Acceptance Rate）**，即实习生猜对的比例。
这也是为什么在 **RAG（检索增强生成）** 或 **代码补全** 场景下，**Prompt Lookup Decoding** 这种简单的 N-gram 查找方法特别有效——因为答案往往就藏在输入文档或之前的代码里，“抄”比“猜”更准。

### 4. 实战中的限制与挑战

虽然投机采样听起来很完美，但在实际落地中（比如使用 vLLM 等框架），也面临一些挑战：

- **词表一致性（Vocabulary）**：独立的 Drafter 模型必须与 Target 大模型共享完全相同的词表（Tokenizer）。很多时候，哪怕两个模型架构相似，词表微小的差异也会导致无法配合。
- **高并发下的副作用**：在极高并发（High QPS）的场景下，显存带宽本身就已经被塞满了。这时候再引入额外的 Drafter 或者进行多 Token 验证，可能会反而加重带宽负担，导致整体吞吐量（Throughput）下降。投机采样最适合的是 **Latency Sensitive（对延迟敏感）** 的低并发场景（如个人助手、本地部署）。
- **上下文长度限制**：目前的 EAGLE 等模型通常有上下文长度上限（如 2048），超过这个长度后，Drafter 的预测能力可能会下降。

### 5. 工业界的黄金搭档：如何选型？

你说到了点子上：**“词表一致性”是投机采样最大的拦路虎。**
因此，在实际工程选型中，我们通常不会随意拼凑模型，而是选择**“同系列、同词表”**的大小模型组合。

以下是目前（2024-2025）业界最主流的几种“黄金搭档”：

- **Llama 系列（Meta）**：

  - **大模型**：Llama-3-70B
  - **小模型**：Llama-3-8B
  - _点评_：最经典的组合。官方原生支持，词表完全一致，配合度极高。

- **Qwen 系列（通义千问）**：

  - **大模型**：Qwen1.5-72B / Qwen2-72B
  - **小模型**：Qwen1.5-7B / Qwen2-7B（甚至更小的 1.8B/0.5B）
  - _点评_：Qwen 系列提供了非常丰富的尺寸梯度（0.5B -> 72B），非常适合做投机采样。

- **专用的“蒸馏”小模型**：
  - 有些社区（如 HuggingFace H4 团队）会专门训练一些 `Distil` 版本的小模型，专门用来给大模型打草稿。
  - 或者使用 **EAGLE** 权重的专用 Drafter（通常是在原模型基础上训练的一个极小的 Head），这种方式不需要加载完整的 7B 小模型，显存占用极低。

**工程建议：**
如果你刚开始尝试，**首选 vLLM 框架 + 同系列大小模型**（例如 Qwen-72B 配 Qwen-7B）。这是最稳妥、坑最少的起步方案。

### 6. 手把手教你：在 vLLM 中开启投机采样

光说不练假把式。**“我该写什么代码来开启它？”**
在业界主流的 `vLLM` 框架中，开启投机采样非常简单，只需要添加几个参数。

**场景一：使用独立的小模型（Draft Model）**
如果你显存够用，可以加载一个独立的小模型来加速。

```python
from vllm import LLM, SamplingParams

# 初始化大模型（Target），同时指定小模型（Speculative）
llm = LLM(
    model="Qwen/Qwen2-72B-Instruct",
    speculative_model="Qwen/Qwen2-7B-Instruct", # 指定小模型
    num_speculative_tokens=5, # 每次让小模型猜 5 个词
)

prompts = ["介绍一下投机采样的原理"]
sampling_params = SamplingParams(temperature=0.8, top_p=0.95)
outputs = llm.generate(prompts, sampling_params)
```

**场景二：使用 N-gram 查找（Prompt Lookup）**
如果你显存紧张，或者在做 RAG/总结任务，可以使用零显存开销的 Prompt Lookup。

```python
llm = LLM(
    model="meta-llama/Meta-Llama-3-70B-Instruct",
    speculative_model="[ngram]", # 魔法参数：启用 N-gram 查找
    num_speculative_tokens=5,
    ngram_prompt_lookup_max=4, # 查找最大 4-gram 的匹配
)
```

_看，只需要改动几行代码，你的推理速度可能就会直接翻倍！_

### 7. 端侧推理的救星

在云端 GPU 上，投机采样是“锦上添花”；但在手机、笔记本等**端侧设备（On-Device AI）**上，它几乎是“雪中送炭”。
端侧设备的显存带宽通常非常有限（Unified Memory），读取一次权重极其缓慢。像 Apple Intelligence 这样的端侧智能系统，大量使用了投机采样技术，利用端侧的小算力（NPU）快速生成草稿，从而让在手机上流畅运行大模型成为可能。

### 8. 未来的演进：投机采样的“百花齐放”

随着技术的发展，人们发现“找一个独立的小模型”并不是唯一的出路。现在的投机采样已经演化出了更多高效的变体，甚至成为了 vLLM 等主流推理框架的标配。

- **无模型投机（Model-Free）：Prompt Lookup Decoding**
  这是一种极简的思路。如果你的输入（Prompt）里已经包含了答案的片段（比如在翻译、文档摘要、代码修改场景中），为什么还需要一个小模型来猜呢？
  **Prompt Lookup Decoding** 直接在当前的上下文里寻找类似的 N-gram（词组片段）作为草稿。这种方法**零成本**（不需要额外显存加载小模型），但在特定场景下（如 RAG、代码编辑）效果惊人。

- **特征层面的“透视”：EAGLE (Extrapolation Algorithm for Greater Language-model Efficiency)**
  这是目前（2024-2025）最前沿的技术之一。
  传统的投机采样是在“词（Token）”的层面上猜，而 EAGLE 选择在更底层的“特征（Feature）”层面上进行外推。它利用大模型中间层的特征向量来预测未来，就像是直接读取了主编脑电波的波动，而不是等他写出字来再猜。
  EAGLE 及其后续版本（EAGLE-2）通过构建动态的“草稿树（Draft Trees）”，比传统的 Medusa 方法更准、更快，是目前追求极致推理速度的首选方案之一。

- **自我投机（Self-Speculative）与 Medusa**
  如果不想要独立小模型，也不想只靠 N-gram，还可以让大模型“自己猜”。
  **Medusa（美杜莎）** 架构在模型末尾加了几个额外的“头”，一次性预测未来好几个词。这相当于主编长了三头六臂，写第一个字的同时，其他手已经在草稿纸上把后几个字写好了。

## 结语

投机采样（Speculative Decoding）已经不仅仅是一个算法技巧，它正在重塑大模型推理的工业标准。从最初的“大带小”，到现在的 EAGLE、Prompt Lookup，技术的演进始终围绕着一个核心：**打破内存墙的物理限制，用更聪明的算力调度换取更流畅的用户体验。**

它告诉我们：在庞大的算力巨兽面前，有时候，一个灵巧的先锋（Drafter），加上一个并行的统帅（Target），往往比统帅单打独斗要走得更快、更远。
