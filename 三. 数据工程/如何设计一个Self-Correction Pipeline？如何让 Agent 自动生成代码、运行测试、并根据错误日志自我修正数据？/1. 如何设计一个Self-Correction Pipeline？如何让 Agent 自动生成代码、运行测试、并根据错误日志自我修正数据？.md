# 如何设计一个 Self-Correction Pipeline？

> **副标题：从 Prompt Engineering 到 System 2 Engineering——打造具备“自我修复”能力的 AI 工程师**

在构建 Agent 系统时，我们常遇到一种令人沮丧的现象：LLM 生成的代码逻辑看似完美，但一运行就是 SyntaxError。

这是 LLM 的本质缺陷决定的——它是一个**概率模型**（Probabilistic），而代码执行环境是一个**确定性系统**（Deterministic）。

为了解决这个问题，我们需要引入 **"Test-Time Compute" (推理时计算)** 的概念。本文将深入探讨如何设计一个基于 **System 2 Thinking** 的 **Self-Correction Pipeline**，让 Agent 不仅能写代码，还能像人类工程师一样：**写代码 -> 跑测试 -> 看报错 -> 修 Bug -> 再运行**。

---

## 1. 核心隐喻：从“直觉反应”到“深思熟虑” (System 1 vs System 2)

- **System 1 (快思考)**：这是大多数 RAG 或 Code Gen 的工作模式。LLM 根据直觉瞬间生成代码，像一个凭肌肉记忆盲打的程序员。
- **System 2 (慢思考)**：这是 Self-Correction 的本质。当发现错误时，系统暂停，进入**反思模式**，调用工具验证假设，直到问题解决。

我们的目标，就是用工程手段（Pipeline），强行赋予 LLM **System 2** 的能力。

---

## 2. 架构设计：现代 Agent 的四重奏

一个现代化的 Self-Correction Pipeline 不再只是简单的 Loop，它包含四个经过**关注点分离 (SoC)** 的组件：

### 2.1 The Generator (生成器)

负责提出解决方案。

- **技术演进**：不要再手写复杂的 Prompt 模板了。使用 **DSPy (Declarative Self-improving Python)**。
- **DSPy 优势**：你定义 `Signature` (输入输出结构) 和 `Metric` (通过测试)，DSPy 会自动优化 Prompt，甚至自动生成 Few-Shot Examples。

### 2.2 The Executor (执行器/现实世界)

这是最关键的组件。它负责在一个**隔离环境**中运行代码。

- **标准协议**：推荐支持 **MCP (Model Context Protocol)** 的沙箱环境。MCP 正在成为连接 LLM 和基础设施（如文件系统、Docker、数据库）的 USB 接口标准。
- **为何必须隔离**：LLM 生成的代码可能包含危险操作。必须使用 **E2B** (基于 Firecracker 的微虚拟机) 或 **GVisor** 容器。

### 2.3 The Inspector (检查员)

Inspector 负责“清洗”数据。原始的 Traceback 对 LLM 来说噪音太大。

- **功能**：提取关键错误帧 (Last Frame)，过滤掉无关的库内部调用栈。
- **策略**：将错误日志转换为结构化的 JSON 对象，而非纯文本。

### 2.4 The Reflector (反思者/推理核心)

这是 Pipeline 的灵魂。

- **模型选择**：这里是 **OpenAI o1** 或 **o3** 系列模型的用武之地。它们天生具备强大的 Chain of Thought 能力。
- **任务**：Reflector 不写代码，只写“诊断报告”。它分析 `Generator` 的代码和 `Executor` 的报错，指出根本原因（Root Cause）。

---

## 3. 技术实现拆解 (The Workflow)

### Phase 1: Test-Driven Generation (测试驱动生成)

不要让 Agent 直接写业务代码，让它先写测试。

- **策略**：使用 DSPy 的 `ChainOfThought` 模块，先生成断言（Assertions）。
- **优势**：测试用例就是 Agent 的“目标函数”。

### Phase 2: Stateful Execution (有状态执行)

传统的 `docker run` 是无状态的，不仅慢，而且无法调试。

- **推荐方案**：使用 **E2B Code Interpreter SDK**。
- **伪代码**：

  ```python
  # 使用 E2B 的有状态沙箱
  from e2b_code_interpreter import Sandbox

  def run_correction_loop(code, tests):
      with Sandbox() as sandbox:
          # 沙箱保持运行状态，可以连续执行多条命令
          sandbox.files.write("solution.py", code)
          sandbox.files.write("test_solution.py", tests)

          # 运行测试
          exec_result = sandbox.commands.run("pytest test_solution.py")
          return exec_result
  ```

### Phase 3: Reasoning-Based Correction (基于推理的修正)

如果测试失败，进入修正循环。此时切换模型。

- **Generator (GPT-4o)**: "我该怎么改代码？"
- **Reflector (o1-preview)**: "慢着。先看错误日志，这个 `IndexError` 是因为循环边界设错了，不是因为列表为空。告诉 Generator 修改第 5 行的 range。"

这种 **Dual-Model pattern (双模型模式)** 是降低成本并提高成功率的关键：用大模型推理，用快模型编码。

### Phase 4: Iteration Control (迭代控制)

- **震荡检测 (Oscillation Detection)**：使用向量相似度（Embedding Similarity）检测代码是否在两个错误版本间反复横跳。
- **Rollback**：维护一个 Git 风格的版本树，随时回滚到“目前为止最好的版本”。

---

## 4. 深度洞察与权衡

### 4.1 Latency vs. Accuracy (延迟与准确性的博弈)

- **现状**：Self-Correction 极慢。o1 模型本身的推理延迟加上沙箱启动时间，可能需要 30-60 秒。
- **决策**：仅在**异步任务**或**高价值代码生成**（如数据分析报告、基础设施 IaC 代码）中使用。实时补全（Autocomplete）场景严禁使用。

### 4.2 The "Cheating" Agent (作弊的 Agent)

- **现象**：Agent 为了 pass 测试，直接删除了 failing test。
- **对策**：在沙箱中对测试文件设置 `chmod 400` (Read-only)。

---

## 5. 最佳实践清单 (The Tech Stack 2026)

如果你现在要落地这套系统，建议遵循以下技术栈：

1.  **Orchestration**: **LangGraph**。它是目前处理循环图（Cyclic Graph）和持久化状态（Persistence）的标准库。
2.  **Prompt Optimization**: **DSPy**。停止手写 Prompt，开始编译 Prompt。
3.  **Sandbox**: **E2B** 或支持 **MCP** 的安全容器。
4.  **Models**:
    - **Coding (Generator)**: Claude 3.5 Sonnet / GPT-4o。它们如同“熟练工”，生成速度快，擅长遵循语法和常见模式。
    - **Reasoning (Reflector)**: OpenAI o1 / DeepSeek-R1。它们如同“专家”，擅长通过长思维链（CoT）进行深度归因分析，解决 Coding 模型“看不出哪里错”的复杂逻辑 Bug。
5.  **Observability & Evaluation**: **LangSmith / Arize Phoenix**。
    - **Why**: Self-Correction 过程本质上是一个黑盒。你需要 Trace 每一轮循环的输入输出，并使用 **Ragas** 或 **DeepEval** 来评估修正的成功率（Pass@k），否则你无法优化 Pipeline。
6.  **Long-term Memory**: **Mem0 / Zep**。
    - **Why**: 不要让 Agent 每次都重蹈覆辙。如果它上周修过一个类似的 `pandas` 版本兼容性 Bug，它应该能从长期记忆中检索到解决方案，而不是重新推理一遍。
7.  **Human-in-the-Loop (HITL)**: **LangGraph Checkpointer**。
    - **Why**: 自动化不是银弹。当 Max Retries 耗尽时，系统应暂停并请求人类介入（Interrupt），人类修复后，Agent 将其存入记忆并继续运行。

> **Q: Coding 模型不具备推理能力吗？**

## 结语

Self-Correction Pipeline 的本质是将 **Software Engineering 1.0** (TDD, CI/CD) 的最佳实践，通过 **Software Engineering 2.0** (LLMs) 的方式重构。

不要只把 Agent 当作生成器，要把它当作一个**会犯错、但会反思、且能使用工具验证假设的初级工程师**。
