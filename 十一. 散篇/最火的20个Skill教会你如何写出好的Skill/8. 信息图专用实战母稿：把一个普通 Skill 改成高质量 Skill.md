# 信息图专用实战母稿：把一个普通 Skill 改成高质量 Skill

这份稿子不再讲“高质量 Skill 的定义”。它只回答一件事：你手里已经有一个能跑的 Skill，为什么它还是不触发、乱触发、越写越肿，以及怎么把它一步一步改到能上线。

为了把问题讲实，我只用一个例子贯穿全文：一个给前端项目做页面检查的 Skill。它的目标本来很简单，帮助 Agent 打开页面、核对布局、截图、发现明显问题。但它的第一版，几乎把大家常踩的坑踩了个遍。

## 先看一个“不算差，但很不稳”的 Skill

第一版长这样。

> 名称：网页助手  
> 描述：帮助用户浏览网页、收集信息、执行操作、检查页面问题。  
> 使用时机：当任务和网页、页面、网站、浏览器有关时。  
> 执行方式：打开页面，查看内容，按需操作，给出结果。  
> 其他：如有问题请重试。

这类写法很常见，因为它看起来没有明显错误：名字也能懂，描述也像那么回事，流程也不是完全没有。但它的问题是，每一行都写了，又几乎每一行都没写到点上。

如果你把这种 Skill 丢给 Agent，最容易出现 5 个结果。

- 该触发的时候不触发，因为“网页助手”这个名字太泛，`description` 也没有明确触发词。
- 不该触发的时候乱触发，因为只要任务里出现“网页”“网站”“浏览器”这类词，它都可能抢着进场。
- 真被触发后很容易跑偏，因为“打开页面，查看内容，按需操作”根本不是流程，只是一句空话。
- 一旦失败，问题很难排查，因为“如有问题请重试”不是失败信号，只是把问题往后推。
- 时间一长，维护者就会不断往里面塞例子、塞补丁、塞注意事项，最后 `SKILL.md` 越来越长，但命中率和稳定性没明显提升。

一句话说，这种 Skill 不是不能用，而是没有被设计成一个稳定的工作单元。

## 第一步，不要先加内容，先把任务缩窄

很多人修 Skill 的第一反应是继续补说明，补案例，补“注意事项”。这通常会让文件更长，不会让 Skill 更准。第一步应该反过来做：先把任务缩窄，先决定它到底只解决哪一类问题。

上面那个例子，真正想做的其实不是“网页助手”，而是“页面验证”。它关心的不是网页里有什么知识，而是页面有没有明显的布局、渲染和交互问题。

改完之后，名字和描述应该至少像这样：

> 名称：页面验证  
> 描述：打开本地或测试环境页面，检查布局错位、明显报错、关键按钮状态和截图结果；当任务涉及“验证页面是否正常”“检查改动有没有把页面弄坏”“对比页面表现”时使用。

这一步看起来只是改了两行，实际上完成了两件大事。

- 第一，它把“做什么”写实了，不再是什么都做。
- 第二，它把“什么时候用”写进去了，模型终于有机会在正确时机想到它。

一个经验判断是：如果你的 `description` 读完之后，还不能让人立刻知道“什么任务该触发它”，那它大概率还没写好。

## 第二步，最该补的不是流程，而是排除条件

很多 Skill 不稳定，不是因为不会做，而是因为出手太多。真正让误触发降下来的，往往不是 `Use When`，而是 `Don't Use When`。

继续看这个例子。页面验证 Skill 很容易被错误地用在这些任务上：

- 用户想抓取网页资料、做研究总结。
- 用户想批量采集站点内容。
- 用户想修前端代码，而不是检查页面表现。
- 用户想登录账号、下单、提交真实数据。

这些都和“网页”有关，但都不应该交给“页面验证”。

所以排除条件应该明确写出来，至少像这样：

> 不要在这些场景使用：  
> 需要长期浏览和整理网页资料时；  
> 需要采集大量页面内容时；  
> 任务重点是修改代码而不是验证页面结果时；  
> 需要真实登录、支付、提交敏感数据时。

很多人写 Skill 时不愿意把边界写窄，怕覆盖面变小。但对 Agent 来说，边界越虚，误触发越高；误触发越高，用户越不信任这个 Skill。先把适用面写窄，通常比一开始追求“大而全”更划算。

## 第三步，把“流程”从空话改成检查链

“打开页面，查看内容，给出结果”不叫流程。真正有用的流程，至少要让人知道它每一步看什么、在哪停、什么情况下继续。

页面验证这个例子，可以把流程压成 5 步检查链：

> 第 1 步：确认访问目标，是本地地址、测试环境还是指定 URL。  
> 第 2 步：等待页面稳定，不在首屏还没渲染完时下结论。  
> 第 3 步：检查是否有明显报错、空白区域、关键模块缺失。  
> 第 4 步：检查关键交互元素是否可见、可点、状态是否异常。  
> 第 5 步：截图并给出结论，结论必须包含“看到了什么问题”或“没有发现什么问题”。

这 5 步的关键，不在于它多复杂，而在于它终于可检查了。以后如果这个 Skill 老是跑偏，你能知道它偏在哪一步，而不是只能模糊地说“它这次又不太对”。

一个简单标准是：如果流程里没有检查点，就很难调试；如果流程里没有停止条件，就很容易误判。

## 第四步，给结果加上“完成定义”

很多 Skill 看起来执行完了，其实没有真正完成。因为它只“做了动作”，没有“完成定义”。

页面验证 Skill 的完成定义，不应该只是“输出检查结果”，而应该写得更具体：

> 满足以下条件才算完成：  
> 页面已成功打开；  
> 至少完成一轮首屏检查；  
> 对关键模块或关键按钮做了状态确认；  
> 产出一张截图；  
> 结论中明确写出“发现的问题”或“未发现明显问题”。

完成定义的价值是：它帮你把“做过了”变成“做完了”。没有这一层，完成率看起来很高，实际结果质量会很虚。

## 第五步，失败信号不要写成“重试一下”

最容易把 Skill 写废的一种方式，就是用兜底句子掩盖真实错误。比如：

- “如果失败请重试”
- “必要时采用其他方式”
- “如果页面异常，请灵活处理”

这种话不是容错，而是让错误失去形状。真正有用的失败信号，要让 Agent 知道什么时候该停、什么时候该追问、什么时候该换策略。

页面验证这个例子，失败信号至少应该有 4 类：

- 页面打不开：地址不可达、服务没启动、网络被拦截。
- 页面始终不稳定：长时间加载、反复重绘、关键元素迟迟不出现。
- 页面需要登录或敏感权限：没有凭证时不要硬闯。
- 当前任务已经不是验证页面，而是修代码、抓数据或提交真实操作。

比起一句“失败就重试”，这样的失败信号才真正有调试价值。

## 第六步，最后才谈权限和外置资料

很多人一开始就往 Skill 里堆工具说明、权限说明、背景资料，结果入口文件很快变得臃肿。更稳的顺序是：先把任务、边界、流程、完成定义写稳，再补权限和资料分层。

页面验证 Skill 的权限约束，应该直接写人话：

> 可以做：打开测试页面、点击不提交真实数据的界面元素、截图、读取前端报错。  
> 不要做：登录真实账号、执行支付、提交表单到生产环境、修改后端数据。  
> 需要时再追问：测试地址、账号权限、目标页面列表。

而那些长篇的选择器约定、页面清单、旧问题样例，不应该一股脑塞进入口。入口只保留最小流程和关键约束，重资料外置到参考文件。

这一步做完，Skill 才开始像“可维护资产”，而不是一次性提示词。

## 改完之后，怎么知道它真的变好了

不要凭感觉。最小验收法可以非常朴素：

- 找 10 个真的应该触发这个 Skill 的任务。
- 再找 10 个看起来像，但其实不该触发的任务。
- 看它会不会在第一组里稳定进场。
- 看它会不会在第二组里克制退出。
- 看它触发后，能不能稳定跑到完成定义。
- 看失败时，是不是能暴露真实原因，而不是继续瞎做。

如果一轮测完，你发现它还是经常抢错任务，那继续补 `description` 没用，应该回头改 `Use When` 和 `Don't Use When`。如果它总在执行中跑偏，那应该回头改流程和检查点。不要哪里都补一点，那样最容易把 Skill 补成一锅粥。

## 给一个可以直接套用的写法骨架

如果你要重新整理一个 Skill，最少可以按这个顺序填：

- 先写名字：只写一种核心动作，不要空泛。
- 再写描述：同时写清“做什么”和“什么时候用”。
- 再写触发条件：给出关键词和典型任务。
- 再写排除条件：把最容易误触发的相邻任务写出来。
- 再写输入输出：开始前需要什么，结束时必须交付什么。
- 再写执行流程：每一步看什么、在哪停、什么情况下继续。
- 再写失败信号：什么时候停止、追问、换策略。
- 最后补权限约束和外置资料。

这个顺序的意义是：先修命中，再修执行，再修治理。顺序反了，文件通常会越来越复杂，但质量不一定提升。

## 这份稿子最想传达的一句话

一个普通 Skill 变成高质量 Skill，不是靠加更多说明，而是靠把四件事写实：什么时候该触发，什么时候不该触发，触发后怎么稳定完成，失败时怎么尽早暴露。

继续往下做信息图，最值得画的不是“高质量 Skill 的定义”，而是下面这 4 张图：

- 一张图画“一个差 Skill 到底差在哪”。
- 一张图画“怎么把名字、描述、边界、流程改对”。
- 一张图画“什么叫完成定义、失败信号、权限约束”。
- 一张图画“怎么验收，以及怎么用一份骨架复用到下一个 Skill”。

一句话总结：高质量 Skill 不是写得更长，而是让命中、边界、流程和失败暴露都变得可检查。
