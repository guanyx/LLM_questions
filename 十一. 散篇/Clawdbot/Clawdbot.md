# Clawdbot / Clawd：从概念到落地使用

## 基本概念介绍

- **Clawd**：对 Claude/Claude Code 的黑话式称呼，用来调侃“把模型当效率怪物使”的状态。
- **Clawdbot**：一个你自己运行在本机/服务器上的个人 AI 助手。你通过常用聊天软件发消息，它不仅回复，还能在你的机器上跑工具、执行任务。
- **核心架构直觉**：Clawdbot 更像“本地控制平面（Gateway）+ 多端入口（CLI/Web/桌面/移动节点）”，模型只是其中一个可替换的组件。

## 核心亮点

- **长期记忆**：对话/偏好/上下文可以长期保留并被再次调用（这是它被当“个人助理”而不是“临时聊天”的基础）。
- **主动通知**：任务完成、监控命中、定时摘要等会主动推送到你常用的聊天渠道。
- **持续运行与自动化执行**：Gateway 常驻后台，支持定时任务与事件触发，把“今天该做什么”变成自动流程。
- **多渠道统一收件箱**：WhatsApp/Telegram/Slack/Discord/Google Chat/Signal/iMessage/Microsoft Teams/WebChat 等渠道统一接入（以及插件扩展的更多渠道）。
- **多端节点与可视化**：支持 macOS/iOS/Android 节点与 Canvas 这类可视化交互面，让它更接近“产品形态”的助理。
- **可控的安全默认值**：陌生私聊默认走配对码；近期版本还强化了对 slash/控制类命令的默认拒绝策略。
- **确定性工作流路线（Lobster）**：把多步工具调用收敛成可审计、可审批、可恢复的流水线，降低“全靠模型编排”的不确定性与 token 反复消耗。

## 工作流程

1. **本机/服务器启动 Gateway（常驻）**：它负责连接各聊天渠道、管理会话与工具执行。
2. **连接你的沟通渠道**：把 WhatsApp/Telegram/Discord/iMessage 等接入到 Gateway。
3. **消息进入 → 安全策略先过一遍**：
   - 陌生私聊默认只回配对码，未批准不处理消息内容。
   - 一些命令型输入（如 `/...`）需要显式授权才会被当成可执行命令。
4. **模型推理 + 工具执行**：根据需要调用 skills/工具，可能包含命令执行、文件修改、网页抓取等。
5. **结果回传与通知**：输出回到原聊天渠道；如果配置了 Cron/Webhooks，也会在条件满足时主动推送。

## 典型应用场景

- **信息获取/早晚报**：从邮件、日程、待办、系统监控里抽取关键信息，早上给你一段“电梯稿”。
- **聊天总结/降噪**：把你不得不在的群聊/频道做定时摘要（例如 iMessage 夜间总结）。
- **个人数据整理**：例如清理联系人、做数据去重、把散落的信息归档（社区分享的一个高频用途）。
- **远程开发/运维**：像“手机里的 Claude Code”一样做 DevOps 操作、跑测试、修构建、查看告警（但成本/可控性会成为关键约束）。
- **自动化触发**：用 Cron/Webhooks 把“看到某个事件就做某件事”跑成后台工作流。

## 已知缺点

- **安全风险天然更高**：它同时具备“收消息入口”和“本机执行能力”，prompt injection/社工风险比纯聊天更现实；社区普遍建议用 VM/独立机器隔离运行。
- **token/成本可能非常高**：让它探索陌生系统、从零摸 API、做复杂修复时，token 上涨很快。
- **可观测性/可控性争议**：部分用户不喜欢“看不清它到底跑了哪些命令、拿到了哪些输入输出”的体验（和传统 coding harness 的透明度相比）。
- **上手门槛偏高**：依赖较新的 Node 版本，Windows 更推荐 WSL2；远程机 OAuth 初始化也可能踩坑。
- **舆论噪声**：社区里存在对“营销感/同名代币”等现象的讨论，但这类更多是情绪与猜测，不属于官方信息。

## 如何使用

### 安装

方式 A：一键脚本

```bash
curl -fsSL https://clawd.bot/install.sh | bash
```

方式 B：npm 全局安装

```bash
npm i -g clawdbot@latest
```

### 首次配置（推荐路径）

1. 运行向导并安装后台服务：

```bash
clawdbot onboard --install-daemon
```

2. 打开本地控制台（浏览器 Control UI）：`http://127.0.0.1:18789/`。
3. 连接你的聊天渠道（WhatsApp/Telegram/Discord 等），并在私聊里完成配对批准。

### 常见排障要点

- 第一次发消息没反应：优先检查是否还没批准 pairing。
- 远程使用：更常见的做法是用 SSH tunnel 或 tailnet，把本地面板/WS 安全带出来，而不是直接公网暴露。
