# RAG 全流程拆解（Query → 检索 → 重排 → 生成）：每一步为什么必要？

目标：从用户的 Query 开始，串起检索增强生成（RAG）在一次请求里真正发生的链路：Query 规整与约束 → 向量化（embedding）→ 向量库召回（retrieval）→ 重排（rerank）→ prompt 组装 → 生成（generation）。每一步都同时回答“做了什么/为什么这么做”，并配一句“就像……”的类比，帮助建立直观的全局理解。

---

## 1. 输入与预处理（Query）

1. 接收 Query + 上下文/约束（可选）：拿到用户输入与当前对话上下文，并补上权限、时间范围、数据源范围等（为了明确“这次检索到底要解决什么”，也避免“搜到但不能给/版本不对”；就像你拿到便签写着“接口又挂了”，先问清楚“哪个接口、什么时间、什么症状”，再确认你查的是“今年版本”）
2. Query 规整（常见）：去噪、纠错、同义改写、抽取关键实体/术语（为了把“聊天式表达”变成“检索式表达”，减少检索偏航；就像把口语提问整理成可检索的关键词与条件）
3. 澄清与拆分（可选）：当 Query 过宽或前提不明时，先追问或拆成可检索的子问题（为了避免后面链路越跑越偏；就像你不知道对方要的是“修复方案”还是“根因分析”，最好先对齐目标）

---

## 2. Embedding 与索引准备（把知识变成“可被找出来”的形态）

1. 切片 + 元数据（离线常见）：把长文档拆成可检索片段，并保留段落边界/标题/时间等元数据（为了让检索有“抓手”，同时能追溯来源；就像把一本书拆成很多带页码与章节名的卡片）
2. 向量化 + 建索引（离线常见）：把每个片段变成向量并写入索引（为了让后续能用相似度快速召回；就像给每张卡片生成一枚“语义指纹”，再按指纹建目录）
3. Query 向量化 + 对齐检查（在线）：把 Query 变成向量，并确保它与文档片段在同一检索空间里可比（为了避免“看起来相似但其实错位”；就像把“我要找什么”投到同一张地图上，且坐标系一致）

---

## 3. 向量库召回（Retrieval）：先把候选证据“捞上来”

这一步的产物是：一批候选片段（通常是 top-N）+ 它们的元数据（来源、时间、相似度分数等）。

1. 近邻检索 + 规模控制：检索 top-N 片段，并把候选数量控制在后续能处理的范围（为了从海量内容里快速缩小范围、又不让后续被噪声淹没；就像先把可能相关的书搬到桌面，但桌面空间有限只能搬一摞）
2. 过滤与约束（常见）：基于权限、时间、数据源、文件类型等过滤（为了保证可用与合规；就像借书要看你有没有权限进特藏室）
3. 多路召回（可选）：向量召回 + 关键词/规则召回 + 结构化字段检索（为了互补：语义检索擅长“意思对”，关键词擅长“字面精确”；就像既用地图找附近，也用门牌号精准定位）

---

## 4. 重排（Rerank）：把“看起来相关”变成“真正能用来回答”

这一步的产物是：更少、更准的 top-K 证据片段（以及它们的排序与可用性判断）。

1. 精细相关性判断：对 top-N 候选逐个打分并重新排序（为了从“语义近”里挑出“真的能回答”的证据；就像桌面上先粗筛一摞书，再逐本翻目录挑最对题的几本）
2. 覆盖检查：优先选择覆盖 Query 关键实体、约束与子问题的片段（为了避免只选到“看起来像”的泛泛材料；就像你问的是“2025 年报销规则”，就别只拿“报销流程概述”）
3. 多目标重排（可选）：相关性 + 多样性 + 新鲜度 + 权威性（为了避免 top-K 全是同源复读或同一视角；就像你希望证据来自多个可信来源，而不是同一篇文章的不同段落）

---

## 5. Prompt 构造：把证据变成“模型会用”的输入

1. 证据清洗与裁剪：去重复、去无关、保留必要上下文（为了在有限上下文窗口里塞进“最能回答”的部分；就像登机箱容量有限，你得把最有用的东西装进去）
2. 证据组织 + 标注：按主题/子问题拼装片段，并附上来源、时间、标题、边界（为了减少互相打架、也方便追溯引用；就像把资料按主题贴到白板上，并给每条证据贴上“出自哪本书第几页”的标签）
3. 指令与输出约束：明确回答目标、格式、引用方式与拒答条件（为了降低跑题与幻觉；就像给写作任务列一个清晰的大纲与评分标准）

---

## 6. 生成（Generation）：把“证据包”转成自然语言答案

1. 模型阅读证据：把 Query 与证据对齐，形成“能回答”的内部表征（为了避免凭空发挥；就像先通读资料再动笔）
2. 组织答案：按问题结构输出，并在关键结论处对齐证据（为了让答案可验证；就像写论文时在结论旁边放引用）
3. 输出控制（可选）：格式、长度、语气、引用密度、是否给出不确定性提示（为了满足产品需求并降低误导风险；就像规定“只写要点、每点一句、必须带出处”）

---

## 7. 回答决策（在“能生成”之外，决定“是否该生成”）

1. 证据充足性判断（可选但很关键）：当证据不足或冲突时，选择拒答、追问或提示不确定性（为了避免“没证据也硬答”；就像你资料没找到，最好先承认缺口而不是乱写）
2. 触发二次检索（可选）：当首轮检索命中率低或 Query 太宽，改写 Query 或拆分子问题再检索（为了把“找不到/找不准”变成可恢复的流程；就像第一家书店没找到，你换关键词或换一家再找）
3. 结构化输出约束（可选）：要求回答按固定结构输出，并明确引用位置（为了让答案更可控、更易验；就像用模板写报告，结论、依据、风险分别放在固定位置）

---

## 8. 工程级 RAG 优化（检索侧）

1. Chunking 变得更“有上下文”：Late Chunking、Contextual Retrieval 等思路更常见（为了减少切片丢上下文与“中间遗失”；就像不再随便撕页，而是按章节结构做索引）
2. 检索更倾向于混合与多阶段：向量召回 + 关键词 + 多轮检索/迭代检索（为了兼顾语义与精确；就像先用地图找附近，再用门牌号精准定位）
3. Rerank 重要性上升：从“可选”变成“效果上限的关键阀门”（为了把“相似”转成“可回答”；就像第二轮面评决定你招到的是不是对的人）

---

## 9. 工程级 RAG 优化（系统侧）

1. 缓存与复用更普遍：Query/embedding 结果缓存、热门片段缓存、重排批处理（为了降低延迟与成本；就像常用资料做成工具包，下次直接取用）
2. 更重视可观测与评估：离线评测 + 在线监控 + 失败用例闭环（为了把 RAG 从“感觉还行”变成“可度量地变好”；就像把质量管理从拍脑袋变成指标与复盘）

一句话总结：RAG 不是给模型“外挂知识”，而是给生成模型配了一条“证据供应链”。你把每一步的产物与目的想清楚，就能在翻车时快速定位问题到底出在“找不到、找不准、还是用不好”。
