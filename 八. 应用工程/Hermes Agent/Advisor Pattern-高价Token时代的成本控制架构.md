# Advisor Pattern：高价 Token 时代，Agent 应该怎么把成本打下来

如果把 2024 年的 Agent 形容成“给一个强模型配一套工具”，那么 2026 年更像是在管理一支分工不同、价格差异巨大的模型团队。

问题已经不只是模型够不够强，而是**一轮任务到底值不值得一直烧贵模型**。

这件事之所以突然变得迫切，不是因为大家突然开始在意成本，而是因为 Agent 的调用链已经比过去长太多了。一次任务不再只有一问一答，而是会经历规划、查资料、调用工具、读结果、改计划、继续执行、最后再总结。每多一轮思考，每多一次长上下文拼接，每多一次高推理强度输出，Token 成本都会放大。

所以今天真正的问题不是“要不要用强模型”，而是：**强模型应该出现在链路的哪一段。**

这正是 **Advisor Pattern** 想解决的问题。

## 一、Advisor Pattern 到底是什么

最直白的定义是：

**让一个便宜、快、足够稳定的 executor 负责大部分执行步骤，只在关键判断点调用更贵、更强的 advisor。**

这里至少有两个角色：

- **executor**：负责主流程推进，读上下文、调工具、拿结果、继续向前走
- **advisor**：不负责全程跑流程，只在 executor 遇到难判断的问题时，给一个更高质量的决策意见

可以把它理解成一种“主治医生 + 会诊专家”的分工。

普通检查、常规问诊、流程推进，都由主治医生完成。只有遇到影像模糊、病灶复杂、方案分叉的时候，才拉专家会诊。真正昂贵的不是“专家存在”，而是“每件小事都让专家亲自做”。

也可以把它理解成“项目经理 + 高级顾问”。

项目经理负责大部分日常推进，但在需求冲突、架构分叉、关键上线决策这些地方，会去问更强的人。这不是为了显得复杂，而是为了让贵资源只出现在高价值节点。

## 二、为什么它会在现在变得重要

Advisor Pattern 这波升温，不是偶然的。

最近一段时间，公开的产品设计、Agent 框架讨论和开发者反馈，开始反复指向同一个现实：**模型能力已经明显分层，但很多 Agent 还在拿最贵的模型跑所有步骤。**

这个现实至少有四个背景变化。

第一，**Agent 的链路越来越长**。

过去一轮调用很短，现在一轮任务可能要连续跑很多步。成本不再只是“单次回复贵不贵”，而是“整条任务链累计下来贵不贵”。

第二，**强模型越来越像高价值决策器，而不是默认执行器**。

它们很擅长架构分叉、复杂规划、疑难判断、结果复核，但并不一定适合承担所有机械性推进工作。

第三，**输出 Token 和思考 Token 的压力都在变大**。

真正贵的任务，往往不是输入，而是长时间执行、反复思考、持续生成。越是长链路的 Agent workflow，这个问题越明显。

第四，**多模型协作开始从研究问题变成产品问题**。

开发者已经越来越不满意“只能手动切模型”或“整条链只能绑一家 provider”的工具形态。真正想要的是：共享上下文、自动升级、按步骤分配不同成本层级的模型。

所以 Advisor Pattern 并不是一个“更花哨的 Agent 术语”，它更像是一种成本现实逼出来的运行方式。

## 三、它真正节省成本的地方，不是“换便宜模型”，而是减少贵模型的驻场时间

很多人第一次听到 Advisor Pattern，会误以为它只是“高低搭配”。

这还不够准确。

它真正节省成本的关键，不是简单把某一部分切到便宜模型，而是**把贵模型从长流程里抽离出来，只保留在高价值决策点上。**

这背后有三个很具体的成本机制。

### 1. 便宜执行器负责高频步骤

在一个真实 Agent 里，最频繁的动作往往是：

- 读工具返回结果
- 判断下一步调用哪个工具
- 做局部整理
- 继续追踪任务状态

这些动作很多，但不一定都需要最强模型。它们更像“执行密集型步骤”，适合由便宜、快的 executor 接住。

### 2. 昂贵模型只处理高不确定性问题

真正值得升级到 advisor 的，通常是：

- 规划已经出现多条分叉，不知道选哪条
- 工具返回的证据相互矛盾
- 代码修改已经触及架构边界
- 需要做一次高质量复核，避免低级错误放大

这类问题的共同点是：**错误代价高，判断质量比执行速度更重要。**

### 3. 长上下文不再每一步都喂给贵模型

这是很多人低估的一点。

如果让最贵的模型从头到尾驻场，它不仅每一步都要花更高单价，还要反复读取越来越长的上下文。Advisor Pattern 的本质，是让贵模型只在必要时拿到“压缩后的高价值上下文”，而不是被迫观看整场流水账。

所以它节省的不是一次调用，而是整条任务链里最昂贵的那部分驻场成本。

## 四、一个像样的 Advisor Pattern，至少要有这四个部件

如果只是“偶尔手动问一下大模型”，那还不算 Pattern。

一个真正可落地、而且适合做进 Agent 的 Advisor Pattern，至少要有四个部件。

### 1. executor

主执行器，负责流程推进。

它要便宜、稳定、速度快，而且工具调用可靠。它不一定最聪明，但要足够能干。

### 2. advisor trigger

升级条件，也就是：**什么时候值得请 advisor 进场。**

这是整个模式里最关键的一层。

如果没有 trigger，系统最后就会退化成两种极端：

- 要么几乎从不升级，advisor 形同虚设
- 要么逢事就升级，成本直接失控

常见 trigger 可以来自：

- 任务复杂度高
- 分支数超过阈值
- 工具结果冲突
- executor 置信度下降
- 进入最终交付或风险较高的步骤

### 3. advisor context

传给 advisor 的上下文，不能是整条流水账原样转发。

更合理的做法是只交付：

- 当前任务目标
- 已完成的关键步骤
- 当前分叉点
- 需要决策的问题
- 候选方案与风险

这一步决定 Advisor Pattern 是“省钱”还是“看起来省钱、其实更贵”。

### 4. return path

advisor 给出意见以后，执行链怎么回到 executor。

这一点经常被低估。很多系统会把 advisor 结果当成最终回答，结果把整个链路割裂了。更合理的方式是：

- advisor 给策略
- executor 继续执行
- 必要时再进入下一次升级

这样 advisor 才是“高价值判断节点”，而不是第二个常驻主流程模型。

## 五、最容易做错的，不是模型选型，而是升级条件

如果这篇文章只留一句最实用的话，那就是：

**Advisor Pattern 的成败，不取决于你选了哪两个模型，而取决于你怎么定义升级条件。**

很多失败案例都不是因为模型太弱，而是因为 trigger 设计得太粗糙。

最常见的三种错误是：

### 1. 把 advisor 当“更贵的备用脑子”

看起来像升级，实际上只是每一步都想再问一次更强模型。

这样做的结果通常是：

- 成本没有降下来
- 延迟明显变长
- executor 的存在意义变弱

### 2. 只按任务类型升级，不按不确定性升级

比如简单粗暴地规定：

- 写代码就用 advisor
- 做规划就用 advisor

问题在于，同样是“写代码”，有的只是改一行常量，有的却是在决定模块边界。真正该升级的不是任务名称，而是**当前决策的风险和歧义程度**。

### 3. 把完整上下文全量转给 advisor

这样虽然“信息最全”，但几乎等于把贵模型重新请回主流程。成本不会真正下来，反而可能因为重复阅读长上下文而更高。

所以一个成熟的 Advisor Pattern，核心不是“有没有 advisor”，而是“有没有像样的 gate”。

## 六、Advisor Pattern 和 Provider Router 会自然连起来

当系统开始认真做 Advisor Pattern，它几乎一定会碰到另一个问题：

**既然要升级，这次升级到底落到哪个 provider、哪个 model、哪个 endpoint？**

这就是为什么 Advisor Pattern 最终会和 **Provider Router** 连起来。

因为上层策略表达的往往不是：

- 这一步用 OpenAI
- 那一步用 Anthropic

而更像是：

- 这一步先交给 fast executor
- 这里升级成 strong advisor
- 最终结果再过一个 reviewer

这些都还是意图，不是运行时答案。

真正把它们落地成请求的，还是中间的路由层。它要继续决定：

- 这次 advisor 升级落到哪家 provider
- 当前该走哪种 `api_mode`
- 凭证和 endpoint 怎么绑定
- advisor 当前不可用时，是换 key、换同 provider、还是跨 provider fallback

所以 Advisor Pattern 本身不是完整方案。它更像上层的成本编排策略，而 Provider Router 是下层的运行时执行边界。

## 七、如果今天要做一个 Advisor Pattern，最小可用版本应该长什么样

如果从零开始，不建议一上来就做得很复杂。

最小可用版本其实只需要四个决策：

### 1. 先定义 executor 的职责边界

哪些步骤默认交给便宜模型，必须先写清楚。

### 2. 再定义 3 到 5 个清晰的 advisor trigger

不要一开始做十几条复杂规则。先抓最有价值的几种升级场景。

### 3. 给 advisor 单独做一份压缩上下文

不要把主流程消息直接整包抛过去。

### 4. 明确 advisor 的输出不是最终回答，而是执行建议

这样 executor 才能继续推进，整条链路也更容易控成本。

如果再往上走一步，比较成熟的版本会继续加上：

- budget guard
- provider routing
- fallback policy
- final reviewer

但那已经是第二阶段的问题，不是第一天必须具备的东西。

## 八、为什么这篇内容适合做成组图

因为 Advisor Pattern 不是纯观点，它天然带有明确的对象和阶段。

这篇文章最适合被拆成组图的部分有五类：

- **角色图**：executor、advisor、reviewer 的分工
- **成本图**：为什么贵模型驻场最花钱
- **触发图**：哪些情况应该升级，哪些不该
- **上下文图**：什么信息该交给 advisor，什么不该全量转发
- **系统图**：Advisor Pattern 和 Provider Router 怎么衔接

也就是说，它比“一个宏观判断”更适合画，因为它有对象、有门槛、有切换点、有返回路径。

## 九、最后一句话

如果过去的 Agent 问题是“怎么把模型接进流程”，那么现在更现实的问题已经变成了：

**怎么让贵模型只出现在真正值得它出现的地方。**

Advisor Pattern 的意义，不是让系统显得更聪明，而是让聪明这件事开始有成本边界。

继续往下学，最值得接上的路线通常是三条：

- 先学 **gating**：怎么定义升级条件
- 再学 **context compression**：怎么给 advisor 交付高价值上下文
- 最后学 **Provider Router**：怎么把“升级成 advisor”真正落到运行时请求上
