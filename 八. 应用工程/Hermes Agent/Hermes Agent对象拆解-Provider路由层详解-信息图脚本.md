# Hermes Agent 对象拆解：Provider 路由层详解组图脚本

## 一、组图定位

这不是一张长图脚本，而是一组从封面到分镜的连续组图脚本。

这组图要讲清楚的核心判断只有一句：

**Provider 路由层的价值，不是帮 Hermes 换模型，而是把变化很快的模型世界，压成一个上层还能稳定依赖的运行时。**

建议做成 **7 张图**：

- 第 1 张：封面
- 第 2 张：它到底在管什么
- 第 3 张：为什么不能写成一个大函数
- 第 4 张：Hermes 抓住的三个核心矛盾
- 第 5 张：Hermes 最关键的设计动作
- 第 6 张：出错以后怎么恢复
- 第 7 张：这套设计换来了什么

## 二、整组图统一设定

### 1）整组图主线

整组图按这条叙事线推进：

主线：问题出现 → 对象拆分 → 边界划清 → 运行时装配 → 分层恢复 → 收益与代价

### 2）视觉关键词

- 技术拆解感
- 系统中间层
- 路由与缓冲带
- 稳定上层 vs 变化底层
- 运行时装配，而不是字符串分发

### 3）配色建议

- 深蓝：主结构、Agent Runtime、稳定层
- 亮蓝：主路径、请求流、路由动作
- 绿色：便宜恢复、cheap route、成功回稳
- 橙色：错误、fallback、复杂度、代价
- 灰色：背景说明、次级分支、备选线路

### 4）每张图固定结构

- 顶部：本张标题
- 中部：主图结构
- 底部：一句结论金句
- 页尾：`Hermes Agent 对象拆解 / Provider Routing`

## 三、逐张组图脚本

### 第 1 张：封面

**标题建议：**

Hermes Agent 对象拆解  
Provider 路由层详解

**副标题建议：**

它不是在“选模型”，而是在吸收 provider、认证、协议和故障的变化

**这一张承载的信息：**

- 先把读者预期掰正
- Provider 路由层不是模型下拉框
- 它是 Hermes 维持上层稳定的关键运行时层

**画面结构建议：**

- 中间放一个大标题框：Provider Router
- 左边是“变化中的模型世界”
  - provider 会变
  - 认证会变
  - 接口会变
  - 模型名字会变
  - 报错方式会变
- 右边是“稳定的 Agent Runtime”
  - Agent Loop
  - Tool System
  - Gateway
  - Cron
  - ACP
- 中间用一条厚实的缓冲层把左右隔开

**适合直接放图上的一句话：**

上层想稳，底层却一直在变。Provider 路由层就是中间那道缓冲带。

**本张结论句：**

Hermes 不是直接面对模型世界，而是先用 Provider 路由层把它收住。

### 第 2 张：它到底在管什么

**标题建议：**

Hermes 把 provider 拆成了 4 类对象

**这一张承载的信息：**

- Hermes 没把 provider 当一个字符串一路传
- 它拆成 identity、ProviderDef、runtime provider、recovery policy
- 这样上层拿到的不是含糊名字，而是一整套运行时绑定结果

**画面结构建议：**

中间做一条从左到右的 4 段对象链路：

1. Provider Identity
2. ProviderDef
3. Runtime Provider
4. Recovery Policy

**每段建议文案：**

#### 1）Provider Identity

- 先把别名、历史名、口语名统一
- 先讲清楚“你说的到底是谁”

#### 2）ProviderDef

- 描述这个 provider 平时长什么样
- 底图来自元数据
- overlay 补本地规则
- 用户配置再改自己的导航点

#### 3）Runtime Provider

- base_url 是哪个
- api_key 用哪把
- auth_type 是什么
- credential pool 绑哪组
- api_mode 说哪种 API 语言

#### 4）Recovery Policy

- 先在 provider 内自救
- 再跨 provider fallback
- 下一轮再尝试把主线路拉回来

**适合上图的强调句：**

真正发请求的，从来不是一个 provider 名字，而是一整套运行时绑定结果。

**本张结论句：**

Provider 路由层不是“选谁”，而是在组装“这次到底怎么发”。

### 第 3 张：为什么不能写成一个大函数

**标题建议：**

为什么这事不能塞进一个大函数里

**这一张承载的信息：**

- 很多系统早期都会这么写
- 进来一个 provider，拼一个 client，失败了就切备用
- 一开始很快，后面会因为问题类型混在一起而失控

**画面结构建议：**

- 左边画一个“巨型函数黑盒”
- 黑盒输入：provider、model、base_url、key、错误码
- 黑盒输出：一堆缠在一起的判断分支
- 右边拆成 4 张问题卡片

**4 张问题卡片文案：**

#### 卡片 A：身份问题

- 用户说的 provider，不一定就是系统真正要认的 provider
- openai 可能指官方，也可能指聚合入口
- 本地 endpoint 看着像 provider，本质更像运行时入口

#### 卡片 B：运行时问题

- 逻辑上说“走某家”，不等于真的知道该连哪里
- base_url、key、credential pool 都得临场解析
- 优先级还要清楚：显式请求 → config.yaml → 环境变量 → auto

#### 卡片 C：协议问题

- 不是所有 provider 都在说同一种 API 语言
- chat_completions、codex_responses、anthropic_messages 不能硬包成一种协议

#### 卡片 D：恢复问题

- 报错以后到底该重试、换 key、换 provider，还是先保住这轮
- 这不是一句“失败就 fallback”能讲清楚的

**适合直接上图的一句话：**

一个大函数会把命名、协议、安全和恢复全搅在一起。

**本张结论句：**

短期少几个文件，长期多一堆耦合。

### 第 4 张：Hermes 抓住的三个核心矛盾

**标题建议：**

Hermes 为什么这样设计？因为它抓住了 3 个矛盾

**这一张承载的信息：**

- 这一层写得稳，不是因为分支多
- 而是因为它抓住了中间层真正要解决的矛盾

**画面结构建议：**

做成 3 个并列大卡片，每张卡片都用“矛盾 → Hermes 的回答”结构。

#### 矛盾 1：上层要稳，底下却一直在变

- Agent Loop 想要固定入口、稳定语义、可预测报错
- provider 世界给你的却是名字、接口、认证、错误全都不统一
- Hermes 的回答：中间一定要有一层先把差异吞掉

#### 矛盾 2：你想统一，但不能装作大家都一样

- 很多系统的问题不是抽象太少，而是抽象太假
- 嘴上说统一接口，实际上在假装所有 provider 都能塞进同一种壳
- Hermes 的回答：上层统一入口，底下保留事实

#### 矛盾 3：系统想继续跑，但又不能把错误藏起来

- 为了高可用，很多系统什么错都 fallback
- 表面看没挂，真实问题却一直埋着
- Hermes 的回答：先分辨错误，再在最便宜的一层止损

**适合直接上图的表达：**

好的路由层会用 fallback，但不会滥用 fallback。

**本张结论句：**

Hermes 的 Provider 路由层，不是更花哨，而是更克制。

---

### 第 5 张：Hermes 最关键的设计动作

**标题建议：**

Hermes 最关键的 5 个设计动作

**这一张承载的信息：**

- 把文章的核心方法论收成一页
- 让读者看到这层为什么能长期演进

**画面结构建议：**

做成从上到下的 5 级台阶，越往下越接近真实请求。

#### 动作 1：先统一身份

- 先认人，再谈路由
- 名字不稳，后面的缓存、计费和恢复都会跟着晃

#### 动作 2：把选择权放到运行时

- 配置不是最终答案
- base_url、key、api_mode 这些都要在请求发生时补齐

#### 动作 3：把协议模式单独拎出来

- 很多时候不是发错 provider，而是说错话
- URL、endpoint、模型特征都应该参与 api_mode 判定

#### 动作 4：把恢复拆层

- 先转 key
- 再试 fallback chain
- 下一轮再尝试回主线路

#### 动作 5：把 key 和 endpoint 绑定

- 不同 base_url 应该绑定不同语义的 key
- 请求发错地方最多是功能问题
- key 发错地方就是安全问题

**适合直接上图的表达：**

Hermes 不是在执行配置文本，而是在组装这次请求。

**本张结论句：**

它先定义对象，再划边界，再谈切换。

### 第 6 张：出错以后怎么恢复

**标题建议：**

请求失败了，Hermes 先在哪一层止损？

**这一张承载的信息：**

- 恢复机制不是“主线挂了就跳备线”
- Hermes 把恢复拆成了更像生产系统的一条分层链路

**画面结构建议：**

做成一条从左到右的“止损阶梯”：

1. 当前 provider 内检查
2. 当前 endpoint 内换 key
3. 判断是不是短时限流
4. 再决定要不要跨 provider fallback
5. 下一轮尝试把主线路拉回来

**中间插 3 个错误标签：**

- 401：更像凭证过期，先 refresh
- 429：更像短时限流，先别急着切线
- 402：更像额度耗尽，直接轮转下一把 key

**适合放在图里的场景文案：**

半夜跑修复任务时突然遇到 429，Hermes 不会立刻切 provider，而是先判断是不是当前 key 被打热了。

**适合直接上图的一句话：**

fallback 应该像止损，不应该像乱跳。

**本张结论句：**

先在最便宜的一层救，救不回来，再往上切。

### 第 7 张：这套设计换来了什么

**标题建议：**

这层设计的收益、代价和最后的判断

**这一张承载的信息：**

- 最后一张负责收束
- 既讲清楚这套设计的收益，也不回避它带来的复杂度

**画面结构建议：**

上半部分做收益卡片，下半部分做代价卡片，最底部放方法论总结。

#### 收益卡片

1. 上层终于不用懂 provider 世界的细枝末节
2. 新 provider 更容易接入
3. 恢复动作更细，不会把所有失败都混成一种
4. 安全边界更清楚，endpoint 和 key 更好绑定
5. 成本路由更容易复用，简单请求可以先走 cheap route

#### 代价卡片

1. 新人更难追链路
2. 状态更分散，文档和实现容易出现时差
3. 必须补 decision trace，不然排查会很痛
4. 测试更难写，因为你测的是一条运行时装配线

#### 底部方法论

如果你自己设计这一层，至少记住 4 句话：

- 不要把 provider 当字符串一路传
- 不要把统一接口误听成统一协议
- 不要滥用 fallback
- 尽早补 decision trace

**适合直接上图的总结金句：**

好的 Provider 路由层，不是替系统找更多模型；它是把模型世界的波动、异构和故障，压成一个上层还能稳稳依赖的运行时。

**本张结论句：**

这层不是在帮系统变花，而是在帮系统变稳。

## 五、适合整组图反复出现的短句

- 底层一直在变，上层却必须稳定
- 先认人，再定怎么连，再定说哪种 API 语言
- 真正发请求的，不是 provider 名字，而是 runtime provider
- 统一入口，不等于统一底层协议

- key 和 endpoint 绑定，本身就是安全边界
- 路由层的价值，不是换模型，而是帮一次请求走对路
- 先定义对象，再划边界，再谈切换
