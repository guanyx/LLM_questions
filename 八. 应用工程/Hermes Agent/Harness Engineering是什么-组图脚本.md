# 《Harness Engineering 是什么》组图脚本

## 一、这组图的主线

这组图不准备把 `Harness Engineering` 讲成一个抽象新名词，而是要把它讲成：

**当模型开始进入真实世界以后，模型外面那一圈负责约束、编排、观察、恢复和控成本的工程系统。**

整组图的重点不是反复强调“它很重要”，而是让读者真正看见：

- Harness 到底包住了什么
- 为什么 AI 工程的主战场会从“调模型”转向“搭系统”
- 如果今天真要落地一个 Agent，最该先看哪几层

所以这组图的叙事结构应该是：

- 封面：先把“模型被工程系统套住”的感觉立起来
- 第 2 张：只用 1 张解释“为什么突然变重要”
- 后面 6 张：直接拆最关键的工程切入点

这组图的真正落点不是“名词解释”，而是：

**Harness 不是模型旁边的配件，而是把不稳定智能变成可落地系统的那套外骨架。**

---

## 二、建议张数

建议做成 **8 张图**：

- 第 1 张：封面 / Harness 到底是什么
- 第 2 张：为什么 AI 工程的主战场变了
- 第 3 张：工具接入与权限边界
- 第 4 张：Context Engineering
- 第 5 张：Workflow 与多 Agent 编排
- 第 6 张：Memory 设计
- 第 7 张：Evals 与可观测性
- 第 8 张：成本、延迟与恢复机制

这个张数的优点是：

- 有完整主线
- 每张只讲一个切入点
- 不需要在每页再塞“怎么开始做”那一段
- 读者看完后，会形成一张比较完整的 Harness 心智地图

## 三、统一视觉语言

### 1）整体风格

- 视觉气质：Tech Editorial / System Anatomy / Runtime Control Plane
- 基调：白底、冷色、结构密、线条克制、像技术杂志里的系统拆解页
- 关键词：harness、control、boundary、context、workflow、memory、evals、recovery
- 禁止：齿轮流水线、机械工厂、科幻机甲驾驶舱、夸张机器人拟人表情

### 2）封面核心隐喻

`Harness` 本身就有马套、缰绳、控制带的意味，所以封面最值得利用的，不是去画一匹马，而是：

**只保留“缰绳 / 束带 / 套具”这层视觉语言，让它像模型外面那一圈工程系统。**

更具体地说：

- 中间是一个高亮、发光、近似“思考核心”的模型球体或模型引擎
- 外面不是金属壳，也不是笼子，而是一圈半透明的“缰绳式工程束带”
- 这些束带从不同方向扣住核心，并分别带着标签：
  - tools
  - permissions
  - context
  - workflow
  - memory
  - evals
  - recovery
- 束带不是压制模型，而是把模型导向一个稳定输出方向
- 画面右前方可以有一个“稳定交付”或“runtime system”出口，表示这不是束缚，而是导向

### 3）封面禁忌

- 不要直接画完整的马，否则读者会先理解成“马术”而不是“工程系统”
- 不要把缰绳画成暴力捆绑，不然会变成“压制智能”的叙事
- 不要做成监狱、笼子、锁链；Harness 的感觉应该是“可控、可导向、可稳定”

### 4）配色建议

- 深蓝：主标题、结构边界、系统轮廓
- 冰蓝：上下文流、控制线、工程束带
- 绿色：稳定输出、成功路径、可交付状态
- 橙色：风险、错误、越界、失控
- 灰色：背景网格、注释、辅助信息

### 5）叙事规则

- 每张图只讲一个判断
- 每张图都要有“模型核心”和“系统外围”的关系
- 不讲大而空的趋势口号，要讲清对象和边界
- 每张图只保留一句可以记住的话

## 四、逐张分镜

### 第 1 张：封面 / Harness 到底是什么

**标题建议：**

Harness Engineering 是什么

**副标题建议：**

模型外面那一圈把智能套住、导向、稳定下来的工程系统

**这一张承载的信息：**

- Harness 不是模型本身
- 它也不是单一工具
- 它是模型外部那一圈让系统真正可控的工程层
- 这一组图接下来要拆的，就是这圈“工程缰绳”

**适合直接上图的一句话：**

真正能落地的 Agent，不只是一个强模型，而是一套把模型稳稳套住的工程系统。

**画面结构建议：**

- 画面中央放一个高亮发光的“模型核心”，像半透明推理引擎或神经球体
- 模型外面不是壳，而是一圈从不同方向拉住它的“缰绳式束带”
- 每条束带上挂极短标签：
  - tools
  - permissions
  - context
  - workflow
  - memory
  - evals
  - recovery
- 左上角保留一点“原始智能”的不稳定感：粒子、噪声、发散线
- 右侧则是“稳定输出方向”：结构化文档、运行结果、交付面板
- 主视觉的核心关系要非常明确：
  - 中间是模型
  - 外圈是 harness
  - 外圈正在把中间这股能力导向可交付方向

**封面视觉重点：**

- 不画马
- 只画“缰绳 / 套具 / 导向带”
- 让人一眼看出“模型外面那圈系统把它管住了”

---

### 第 2 张：为什么 AI 工程的主战场变了

**标题建议：**

模型越来越强，为什么工程反而更重要了

**副标题建议：**

瓶颈已经从“会不会回答”，变成“能不能完成任务”

**这一张承载的信息：**

- 以前大家主要比模型回答能力
- 现在真实任务越来越像一个长链路系统
- 任务不只是“回答一句话”，而是“理解、检索、调用、执行、检查、交付”
- 所以问题中心从模型能力，转向系统完成度

**适合直接上图的一句话：**

AI 工程的新瓶颈，不再只是模型够不够聪明，而是系统能不能把任务跑完。

**可直接上图的文案块：**

- 旧问题：模型能不能回答
- 新问题：系统能不能完成任务
- 真实链路：理解任务 -> 找信息 -> 调工具 -> 执行动作 -> 检查结果 -> 安全交付
- Harness 负责的，就是这条链路的稳定性

**画面结构建议：**

- 左边一块是“旧世界”：
  - user question
  - model
  - answer
- 右边一块是“新世界”：
  - request
  - context
  - tools
  - workflow
  - memory
  - eval
  - delivery
- 左边画成单段线，右边画成多段系统链
- 中间用一个转向箭头强调：
  - from model output
  - to system completion

---

### 第 3 张：工具接入与权限边界

**标题建议：**

Agent 一旦开始“做事”，第一层问题就是工具和权限

**副标题建议：**

没有工具，它只会说；没有边界，它就会乱做

**这一张承载的信息：**

- Agent 落地一定会接触外部世界
- 工具不是越多越好，关键是边界是否清楚
- 真正的问题不是“能不能接工具”，而是“允许它做什么”
- 这一层决定的是系统会不会越界

**适合直接上图的一句话：**

Harness 的第一层，不是让 Agent 无所不能，而是让它只在允许的边界里行动。

**可直接上图的文案块：**

- 真实世界入口：
  - filesystem
  - browser
  - terminal
  - database
  - internal API
- 真正关键的不是接入，而是边界：
  - allowlist
  - permissions
  - approval
  - middleware guardrails

**画面结构建议：**

- 中央是一个 Agent 核心
- 左边围一圈工具卡片：
  - file
  - shell
  - browser
  - db
  - api
- 这些工具卡片外面再画一层“权限边框”
- 用绿色表示 allowed，用橙色表示 blocked，用黄色表示 approval required
- 让读者一眼看到：
  - 工具在外面
  - 权限在工具与 Agent 之间
  - Harness 像一道明确的控制层

---

### 第 4 张：Context Engineering

**标题建议：**

最贵的不是模型调用，而是错误的上下文

**副标题建议：**

很多看起来像“模型不行”的问题，本质上是上下文工程没做好

**这一张承载的信息：**

- 真实任务里，上下文不是越多越好
- 全塞进去会更慢、更贵、更散
- 太少又看不到关键条件
- 现在真正重要的是上下文预选、动态压缩和检索前置

**适合直接上图的一句话：**

今天很多 Agent 体验差，不是因为模型太笨，而是因为喂给它的上下文太脏。

**可直接上图的文案块：**

- 常见误区：history 越多越稳
- 实际后果：
  - token 膨胀
  - 注意力发散
  - 目标漂移
  - 成本飙升
- 现在更前沿的做法：
  - context preselection
  - dynamic compression
  - retrieval before injection
  - role-isolated context

**画面结构建议：**

- 左边是一大团混乱上下文：聊天历史、长文档、日志、工具结果
- 中间是一个“context shaping”层，像过滤器、压缩层、检索器的组合
- 右边才是模型入口
- 左侧用灰色和红色表示噪声，右侧用蓝色和绿色表示干净、短、关键的上下文
- 整体视觉要突出“不是塞进去，而是筛出来”

---

### 第 5 张：Workflow 与多 Agent 编排

**标题建议：**

真正该设计的不是“一个万能 Agent”，而是任务怎么被拆开

**副标题建议：**

多 Agent 不是目的，职责分离才是目的

**这一张承载的信息：**

- 复杂任务本来就不是一步完成
- 真正关键的是先把流程拆清楚
- 规划、执行、审查可以是不同角色
- 多 Agent 的价值来自隔离干扰，不是来自数量变多

**适合直接上图的一句话：**

Workflow 设计的核心，不是让更多 Agent 同时上场，而是让每一步只承担自己该承担的职责。

**可直接上图的文案块：**

- 一个常见错误：
  - planning
  - retrieval
  - execution
  - review
  全塞给同一个 Agent
- 更成熟的结构：
  - planner
  - executor
  - reviewer
- 多 Agent 的真正价值：
  - less interference
  - clearer state
  - easier eval

**画面结构建议：**

- 左边画一个“万能 Agent”大脑，连着很多混乱箭头
- 右边画成三段清晰流水：
  - planner
  - executor
  - reviewer
- 每个角色各自连自己的输入和输出
- 用颜色强调右边更干净、更稳定、更可检查

---

### 第 6 张：Memory 设计

**标题建议：**

记忆不是越多越好，而是越有边界越好

**副标题建议：**

真正成熟的记忆，不是堆聊天记录，而是分层保存可复用状态

**这一张承载的信息：**

- 长期任务一定会碰到记忆问题
- 但“全都记住”会带来污染、膨胀和混乱
- 更成熟的做法是把记忆分层、分类、隔离
- 记忆正在从聊天历史，变成系统状态的一部分

**适合直接上图的一句话：**

Harness 里的记忆，真正要保存的不是过去说过的话，而是未来还要用的状态。

**可直接上图的文案块：**

- 低级记忆：全部历史一起堆
- 常见后果：
  - irrelevant history
  - stale facts
  - privacy mixing
  - token inflation
- 更成熟的记忆分层：
  - user preferences
  - task state
  - reusable experience
  - long-term facts

**画面结构建议：**

- 左边是一整个杂乱记忆池，里面混着聊天、任务、偏好、旧结论
- 右边是分层记忆柜或分区记忆面板
- 记忆层之间有明确标签和读取边界
- 中间加一层“summarize / isolate / store”的处理层
- 画面要让人看出：
  - 记忆不是囤积
  - 记忆是结构化存储

---

### 第 7 张：Evals 与可观测性

**标题建议：**

没有评测的 Agent，几乎等于不可维护

**副标题建议：**

你必须知道它错在哪一步，而不只是知道它“这次没做好”

**这一张承载的信息：**

- Agent 不是一次输出，而是一串过程
- 所以只看最终结果不够
- 必须看到轨迹、工具调用、中间状态和失败原因
- Evals 和 observability 不是上线后的装饰，而是 Harness 的核心层

**适合直接上图的一句话：**

Harness 最容易被低估的一层，是让系统失败时仍然能被看见、被定位、被复现。

**可直接上图的文案块：**

- 只看结果，你只能知道它错了
- 看见轨迹，你才能知道它为什么错
- 关键观测层：
  - trace
  - replay
  - tool logs
  - latency
  - cost
  - step-level eval

**画面结构建议：**

- 顶部是一条任务链
- 每一步下面都挂着小型观测窗口：
  - input
  - tool call
  - model decision
  - output
- 右边再补一块失败分析板：
  - planning error
  - context error
  - tool error
  - recovery error
- 整张图的感觉应该像“飞行记录 + 系统回放台”

---

### 第 8 张：成本、延迟与恢复机制

**标题建议：**

真正能上线的 Harness，一定同时考虑成本、速度和失败恢复

**副标题建议：**

强模型只是上限，模型路由、缓存和恢复边界决定下限

**这一张承载的信息：**

- 真实系统不可能永远成功
- 也不可能每一步都用最贵模型
- 这一层关注的是：
  - 成本控制
  - 延迟控制
  - 失败恢复
- 这也是 Harness 最像“系统工程”的地方

**适合直接上图的一句话：**

成熟的 Harness 不是从不失败，而是失败时仍然可控、可恢复、可承受。

**可直接上图的文案块：**

- 运行现实：
  - not every step needs the strongest model
  - not every failure deserves a blind retry
- 关键机制：
  - model routing
  - caching
  - bounded retry
  - fallback path
  - recovery policy
- 最后的工程目标：
  - stable
  - affordable
  - fast enough

**画面结构建议：**

- 左边画一条长任务链，每一段用不同模型强度或成本标签
- 中间插入缓存节点和恢复分叉
- 某个节点故意显示失败，再通过 bounded retry 或 fallback 收回主线
- 右下角给一个很清晰的结果板：
  - cost controlled
  - latency bounded
  - recovery available
- 让整组图最后落到“能上线”的感觉，而不是停留在概念上

---

## 五、这一组图最后应该留下来的感觉

看完整组图以后，读者应该带走的不是一句“Harness 很重要”，而是下面这句话：

**当模型真正进入真实世界，决定成败的就不再只是模型智力，而是模型外面那一整圈把它管住、导向、观察、恢复并控成本的工程系统。**

如果只留一句适合当结尾的话，可以用：

**Harness Engineering 的本质，不是给模型加配件，而是把智能变成系统。**
