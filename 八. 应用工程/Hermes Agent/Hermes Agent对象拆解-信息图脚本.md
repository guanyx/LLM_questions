# 《Hermes Agent 对象拆解》信息图脚本

## 一、组图主线

这组图建议采用一条很清晰的主线：

**先告诉读者 Hermes 不是聊天壳，而是一套 Agent Runtime；再按“入口 → 主循环 → 上下文 → 模型 → 工具 → 执行 → 记忆 → 技能 → 网关 → 定时 → 安全 → 扩展”的顺序，把整个对象拆开。**

如果要把整组图压缩成一句适合首页的表达，可以用：

**Hermes Agent 的核心价值，不是会不会聊天，而是把记忆、技能、执行、安全和持续运行做成了一整套机器。**

这样做的好处是：

- 第一张先建立世界观，避免读者把 Hermes 理解成“另一个 AI 助手”
- 中间几张图逐层拆结构，读者能跟着系统链路往下走
- 后面几张图把 Hermes 最有差异化的能力讲透：记忆、技能、Cron、安全、扩展
- 整组图既适合技术读者，也适合非技术读者看懂“为什么它像个运行时”

## 三、统一视觉建议

### 1）整组图的视觉定位

- 风格：技术拆解感、系统架构感、轻工程蓝图风
- 气质：不是营销海报，而是“把一个系统掰开给你看”
- 关键词：模块、路由、循环、注入、执行、沉淀、隔离

### 2）配色建议

- 主色：深蓝 / 石墨黑，代表 runtime 与系统感
- 强调色 1：亮蓝，代表 Agent 主循环与数据流
- 强调色 2：绿色，代表记忆、技能、生长
- 强调色 3：橙色，代表风险、安全、审批
- 辅助色：灰白，承载解释性文字

### 3）整组图统一组件

- 中心节点：Hermes Agent
- 模块卡片：每张图最多 3 到 5 个核心卡片
- 箭头：统一用“输入 → 决策 → 执行 → 回写”逻辑
- 标签：固定使用这几类
  - 输入层
  - 控制层
  - 执行层
  - 记忆层
  - 保障层
  - 扩展层

### 4）每张图的固定结构

- 顶部：标题
- 中部：主图结构
- 底部：一句总结金句
- 页尾：页码 + 系列名

### 第 2 张：入口层

**标题建议：**

Hermes 从哪里接收任务？

**这一张承载的信息：**

- Hermes 有多个入口，不只是在终端里对话
- 主要入口包括 CLI、Gateway、ACP、API、Batch
- 不同入口只是不同表面，最后都汇入同一个 AIAgent 核心

**适合上图的表达：**

不同入口，只是不同的门；进门以后，走进的是同一个 Agent。

**画面结构建议：**

- 左边画五个入口：
  - CLI
  - Telegram / Slack / Discord 这类 Gateway 入口
  - ACP 编辑器入口
  - API 入口
  - Batch Runner
- 中间汇聚成一个节点：AIAgent
- 右侧标注三个共同动作：
  - 创建会话
  - 绑定上下文
  - 返回结果

**源码设计补充（建议直接融进图注）：**

- 设计亮点：CLI、Gateway、ACP、Batch 最后都复用同一个 `AIAgent` 核心，入口变多了，但大脑没有分叉
- 设计亮点：新增入口更多是在补协议适配、回调和会话桥接，而不是重写一套 Agent
- 设计代价：核心统一，但外围接入层会变厚，不同平台各自还有会话映射、状态回传、协议胶水代码
- 画图时要强调：不是“五个入口并列存在”，而是“五种表面 → 同一个运行时核心”

### 第 3 张：Agent Loop

**标题建议：**

Hermes 真正的大脑：Agent Loop

**这一张承载的信息：**

- Agent 不等于一次模型调用
- Hermes 的核心是回合式主循环
- 一轮任务会经历：接收输入、组装 prompt、调用模型、判断是否调工具、执行工具、回填结果、继续或收敛

**适合直接上图的一句话：**

聊天模型只负责生成；Agent Loop 负责把任务真的跑完。

**画面结构建议：**

- 做成一个闭环流程图：
  - 用户输入
  - 构造上下文
  - 模型判断
  - 工具执行
  - 回填结果
  - 再次判断
  - 最终输出
- 旁边加 3 个注释小标签：
  - 可中断
  - 可重试
  - 有预算上限

**源码设计补充（建议直接融进图注）：**

- 设计亮点：`run_agent.py` 里的 `AIAgent` 真正承担了回合控制，不只是调一次模型
- 设计亮点：主循环里同时处理工具执行、上下文压缩、失败重试、最大迭代数和会话持久化，是真正的 runtime loop
- 设计代价：主循环很强，但 `run_agent.py` 也因此很重，provider 兼容、流式输出、压缩恢复都堆在中枢里
- 画图时要强调：Hermes 的核心不是“生成答案”，而是“让任务在循环里收敛”

### 第 4 张：Prompt 组装层

**标题建议：**

Hermes 是怎么把“自己”组装出来的？

**这一张承载的信息：**

- Hermes 的系统提示词不是一段固定文案
- 它由人格、记忆、用户偏好、项目上下文、技能索引、工具说明共同拼装
- 上下文文件包括 `SOUL.md`、`MEMORY.md`、`USER.md`、`AGENTS.md`、`.hermes.md` 一类内容
- Hermes 采用冻结快照思路，避免中途频繁改变系统提示词

**适合直接上图的一句话：**

Hermes 每次开工前，都会先把“我是谁、我记得什么、当前项目是什么”拼成一张工作底稿。

**画面结构建议：**

- 中间画一个“系统提示词装配台”
- 左侧放输入材料：
  - SOUL
  - MEMORY
  - USER
  - AGENTS
  - Skills Index
  - Tool Instructions
- 右侧输出：
  - Stable System Prompt
- 底部强调：
  - 稳定
  - 可缓存
  - 不乱漂移

**源码设计补充（建议直接融进图注）：**

- 设计亮点：`prompt_builder.py` 不是拼一段死文案，而是在装配 `SOUL.md`、`AGENTS.md`、项目上下文、技能索引、工具说明
- 设计亮点：注入前会先扫上下文内容，防止把明显的 prompt injection 文本直接拼进系统提示词
- 设计亮点：技能提示词有 snapshot / cache 思路，说明它优先追求稳定和缓存命中
- 设计代价：来源越多，系统越强，但也越难追溯“这一轮行为到底受哪段上下文影响”
- 画图时要强调：这不是 prompt 文案，而是一条“零件装配线”

### 第 5 张：Provider 路由层

**标题建议：**

Hermes 怎么决定这次该用哪个模型？

**这一张承载的信息：**

- Hermes 支持多模型提供商
- 它不把自己绑死在一个模型 API 上
- 它会做 provider 解析、凭证管理、API 模式选择与失败切换
- 三种主要 API 形态：
  - OpenAI 兼容 chat
  - Responses / Codex
  - Anthropic Messages

**适合直接上图的一句话：**

模型是会换的，路由层才是稳定的。

**画面结构建议：**

- 左边：Hermes
- 中间：Provider Router
- 右边：多个模型提供商节点
- 每条线旁边写：
  - API mode
  - credentials
  - fallback

**源码设计补充（建议直接融进图注）：**

- 设计亮点：`AIAgent` 初始化时会结合 provider 和 base_url 自动判定 `api_mode`
- 设计亮点：既兼容 chat completions，也兼容 codex responses、anthropic messages，模型和接口差异被 runtime 吸收了
- 设计亮点：`smart_model_routing.py` 里还有 cheap route，简单请求可走更便宜的路径
- 设计代价：路由逻辑分散在 provider 解析、credential、runtime provider、多种 API 形态之间，排查调用链会更绕
- 画图时要强调：路由层的价值不是“换模型”，而是“把模型差异藏在中间层里”

### 第 6 张：Tool System

**标题建议：**

Hermes 是怎么从“会说”变成“会做”的？

**这一张承载的信息：**

- Tool System 是 Hermes 的执行总线
- 它负责工具注册、schema 收集、调用分发、结果包装和错误处理
- Hermes 的工具不是零散外挂，而是统一注册表里的能力
- 常见工具类型：
  - Web
  - Terminal
  - File
  - Browser
  - Memory
  - Todo
  - Delegate
  - Cronjob
  - MCP

**适合直接上图的一句话：**

工具系统决定 Agent 能不能落地执行，而不只是生成一段看起来很聪明的话。

**画面结构建议：**

- 中间一条总线：Tool Registry
- 四周辐射多个工具卡片
- 底部标一个重点提示：
  - 有些工具在改外部世界
  - 有些工具在改 Agent 自己

**源码设计补充（建议直接融进图注）：**

- 设计亮点：`tools/registry.py` 里的 `ToolRegistry` 把 schema、handler、toolset、check_fn 统一收口
- 设计亮点：对模型暴露工具时会先过滤可用性，真正执行时统一 `dispatch`，工具不再是零散外挂
- 设计亮点：像 memory、todo、delegate 这类工具并不只是外设，而是在改 Agent 自己的内部状态
- 设计代价：模块导入即注册虽然好用，但读代码时不够显式；统一错误包装也会让第一现场离底层异常更远
- 画图时要强调：Tool System 不是工具列表，而是一条“能力治理总线”

### 第 7 张：Execution Backends

**标题建议：**

为什么执行环境本身也是 Hermes 的一部分？

**这一张承载的信息：**

- 工具执行不只是一句函数调用，还需要真实运行环境
- Hermes 提供多种执行后端：
  - Local
  - Docker
  - SSH
  - Singularity
  - Modal
  - Daytona
- 这让 Hermes 可以在本机、远程主机、容器和云沙箱里工作

**适合直接上图的一句话：**

同样是执行命令，跑在本机、容器还是云沙箱，安全和能力完全不是一回事。

**画面结构建议：**

- 上层：Tool System
- 下层：六个地基模块
- 每个模块标一个关键词：
  - Local：最快
  - Docker：隔离
  - SSH：远程
  - Singularity：集群
  - Modal：无服务器
  - Daytona：持久工作区

**源码设计补充（建议直接融进图注）：**

- 设计亮点：`tools/environments/` 把执行环境抽成正式架构层，分别实现 local、docker、ssh、modal、daytona、singularity
- 设计亮点：同一类执行能力可以挂到不同环境后端，Hermes 因此能从本机延伸到远程、容器和云沙箱
- 设计代价：后端一多，能力边界就不可能完全一致，远程同步、凭证传递、生命周期清理都会变成额外复杂度
- 画图时要强调：这里画的不是“六个命令执行器”，而是“六种能力边界”

### 第 8 张：Memory & Session

**标题建议：**

Hermes 怎么记住你，又不把上下文撑爆？

**这一张承载的信息：**

- Hermes 的记忆不是“把所有对话都塞进 prompt”
- 它有两层结构：
  - 小而精的持久记忆：`MEMORY.md` 与 `USER.md`
  - 大而全的会话仓库：SQLite + FTS5 搜索
- 当前上下文太长时，还有压缩层负责折叠中段历史

**适合直接上图的一句话：**

真正的记忆系统，不是全记住，而是在正确的时候想起正确的事。

**画面结构建议：**

- 上层：Persistent Memory
  - 环境事实
  - 用户偏好
  - 项目约定
- 中层：Session Search
  - 历史对话搜索
  - 召回过去讨论
- 下层：Context Compression
  - 压缩中段
  - 保留最近消息

**源码设计补充（建议直接融进图注）：**

- 设计亮点：`hermes_state.py` 用 SQLite + FTS5 做会话仓库，`context_compressor.py` 负责长会话压缩，`memory_manager.py` 负责多层记忆协调
- 设计亮点：Hermes 把长期记忆、会话检索、上下文压缩拆成不同层，而不是把所有历史硬塞进 prompt
- 设计代价：记忆分散在文件、数据库、压缩摘要、外部 provider 之间，统一心智模型不算简单
- 设计代价：FTS5 搜索很实用，但它不是完整语义记忆，压缩摘要也天然会丢细节
- 画图时要强调：Hermes 的记忆是“分层记忆系统”，不是“一个大仓库”

### 第 9 张：Skills System

**标题建议：**

为什么技能是 Hermes 最像“学习型 Agent”的地方？

**这一张承载的信息：**

- Skills 不是普通提示词碎片
- 它是按需加载的知识文档，也是程序化记忆
- Hermes 可以在复杂任务完成后，把成功方法沉淀成新技能
- 技能还可以被修补、更新、从 Hub 安装、从外部目录加载

**适合直接上图的一句话：**

记忆让 Hermes 记住事实，技能让 Hermes 记住做事方法。

**画面结构建议：**

- 左边画“任务完成”
- 中间画“经验沉淀”
- 右边画“新技能写入 skill 库”
- 下方补一条技能使用链路：
  - 技能列表
  - 按需展开
  - 进入当前任务

**源码设计补充（建议直接融进图注）：**

- 设计亮点：`skill_utils.py`、`skills_tool.py`、`prompt_builder.py` 一起把技能做成可发现、可装配、可治理的程序化记忆
- 设计亮点：技能不是静态提示词，而是有目录、frontmatter、平台匹配、配置变量、加载条件的一整套系统能力
- 设计代价：技能会生长，就必须治理；否则很容易膨胀、重复、过时，反过来拖累 Agent 稳定性
- 画图时要强调：技能不是“模板库”，而是“方法库 + 生长机制”

### 第 10 张：Gateway & Delivery

**标题建议：**

Hermes 为什么能生活在 Telegram、Slack、Discord 里？

**这一张承载的信息：**

- Gateway 是 Hermes 的消息世界入口
- 它负责平台适配、会话路由、消息投递和回传
- 多平台支持的难点，不是接几个 Bot，而是把不同平台统一成同一套 Agent 行为

**适合直接上图的一句话：**

Gateway 不是“多开几个聊天窗口”，而是让同一个 Agent 穿过不同平台继续工作。

**画面结构建议：**

- 中间：Gateway Runner
- 左边：Telegram、Slack、Discord、WhatsApp、Signal、Email 等输入面
- 右边：AIAgent
- 底部：Delivery 返回各平台
- 补三个小标签：
  - authorize
  - route
  - deliver

**源码设计补充（建议直接融进图注）：**

- 设计亮点：`gateway/run.py`、`gateway/session.py`、`gateway/delivery.py` 共同组成统一消息运行时，不是单纯的 Bot 适配器集合
- 设计亮点：平台差异、会话路由、结果投递、状态追踪被网关层吸收，同一个 Agent 因此能跨表面延续
- 设计代价：多平台支持会显著放大边角复杂度，媒体处理、鉴权方式、会话模型都不统一
- 画图时要强调：Gateway 不是多开窗口，而是把多平台统一成同一条消息总线

### 第 11 张：Cron Scheduler

**标题建议：**

Hermes 为什么不只是“等你发消息”？

**这一张承载的信息：**

- Hermes 有自己的定时调度器
- Cron 跑的不是 shell 任务，而是完整的 Agent 任务
- 到点后会拉起一个新会话，按需要注入技能，跑完再把结果投递出去
- 这让 Hermes 从会话助手变成持续运行的系统

**适合直接上图的一句话：**

当 Agent 学会定时工作，它就开始像服务，而不只是像聊天对象。

**画面结构建议：**

- 时钟图标在顶部
- 流程图：
  - scheduler tick
  - load job
  - fresh agent session
  - inject skills
  - run prompt
  - deliver result
- 旁边补一个说明：
  - 可发回原平台
  - 可发到指定渠道
  - 可写本地输出

**源码设计补充（建议直接融进图注）：**

- 设计亮点：`cron/jobs.py` 负责定义任务和调度规则，`cron/scheduler.py` 负责真正执行和结果投递
- 设计亮点：Cron 跑的不是裸 shell，而是完整 Agent 任务，所以可以带 prompt、技能、上下文和目标渠道
- 设计代价：一旦有定时调度，Hermes 就更像持续运行服务，失败重试、重复投递、调度漂移都会变成系统级问题
- 画图时要强调：这不是定时器，而是“定时唤醒 Agent Runtime”

### 第 12 张：Security & Isolation

**标题建议：**

为什么 Agent 安全不是“内容审核”那么简单？

**这一张承载的信息：**

- Hermes 的安全是多层防线，不是一层过滤器
- 关键层次包括：
  - 用户授权
  - 危险命令审批
  - 容器隔离
  - 上下文文件注入扫描
  - 输入参数净化
  - 跨会话隔离
- 它要防的不是“说错话”而已，而是“真的做错事”

**适合直接上图的一句话：**

对 Agent 来说，真正的风险不是输出一句错话，而是拿着工具去执行一件错事。

**画面结构建议：**

- 画成三层护城河
  - 外层：谁能进来
  - 中层：什么能执行
  - 内层：执行时跑在哪、能碰到什么
- 旁边加一个风险箭头：
  - Prompt Injection
  - Dangerous Command
  - Unauthorized DM
  - Cross-session contamination

**源码设计补充（建议直接融进图注）：**

- 设计亮点：安全不是单点模块，而是分散在 `approval.py`、`terminal_tool.py`、`prompt_builder.py`、`skills_guard.py` 等多个环节
- 设计亮点：上下文进入前扫注入、命令执行前做危险审批、技能加载前做安全审计，再叠加 task_id 和环境隔离
- 设计代价：覆盖更全，但安全逻辑分散，阅读和维护时不容易一眼看全
- 设计代价：安全门越多，交互摩擦和误报成本也会上升
- 画图时要强调：安全不是一堵墙，而是一串门

### 第 13 张：Plugin / MCP / External Memory

**标题建议：**

Hermes 为什么还能继续长能力？

**这一张承载的信息：**

- Hermes 不把能力全部写死在核心里
- 它有插件机制、MCP 接入、外部记忆提供者
- 这让它既能接 GitHub、数据库、浏览器等外部工具，也能外挂 Honcho、Hindsight、Mem0 一类记忆层
- 这一层说明 Hermes 不只是产品，也是平台

**适合直接上图的一句话：**

核心负责稳定，扩展层负责让 Hermes 不断长出新器官。

**画面结构建议：**

- 中间：Hermes Core
- 外围三圈：
  - Plugins
  - MCP Servers
  - External Memory Providers
- 每圈连出几个例子：
  - GitHub
  - DB
  - Browser Stack
  - Honcho
  - Hindsight
  - Mem0

**源码设计补充（建议直接融进图注）：**

- 设计亮点：MCP、插件、外部记忆 provider 都被做成了可插拔扩展层，核心不必把所有能力写死
- 设计亮点：`plugins/memory/` 已经内置 Honcho、Hindsight、Mem0、RetainDB、Supermemory 等 provider，说明这层不是概念，而是真扩展接口
- 设计代价：扩展层越强，兼容性、依赖、可信边界、生态治理就越复杂
- 画图时要强调：这一层表达的不是“外挂更多”，而是“Hermes 已经开始平台化”

## 八、整组图可复用的金句

- Hermes 不是聊天壳，而是一套 Agent Runtime。
- 聊天模型负责生成，Agent Loop 负责跑完任务。
- Prompt 不是一句话，而是一套装配系统。
- 模型是会换的，路由层才是稳定的。
- 工具系统决定 Agent 能不能真正落地执行。
- 执行环境不是细节，而是能力边界。
- 真正的记忆，不是全记住，而是想起关键的事。
- 记忆让 Hermes 记住事实，技能让 Hermes 记住方法。
- Gateway 让同一个 Agent 穿过不同平台继续工作。
- 当 Agent 学会定时工作，它就更像服务。
- 对 Agent 来说，安全是运行时问题，不只是审核问题。
- 核心负责稳定，扩展层负责生长。
