# 极简定义
Hermes Agent 不是一个“会调用几个工具的聊天壳”，而是一个把长期记忆、技能沉淀、工具执行、消息网关、定时自动化和安全隔离绑在一起的 Agent 运行时。它想解决的核心问题，不是让模型回答得更像人，而是让一个 Agent 能持续工作、持续学习、跨会话延续、跨平台接入，并且在复杂任务里逐步长出自己的方法论。

# 为什么 Hermes 值得单独拆
- 如果把普通聊天助手比作“一次性顾问”，Hermes 更像“会持续上班的数字同事”。
- 它的重点不是单轮推理，而是完整闭环：接收请求、构造上下文、调用工具、沉淀记忆、长出技能、定时运行、跨平台交付。
- 这也是它适合做成信息图系列的原因：Hermes 不是一个大模块，而是一套层次很清楚的 Agent 机器。

# 封面总览：一个 Hermes Agent 由什么构成

中心放：
- Hermes Agent

# 最小运行链路
1. 用户输入进入系统，入口可能是终端、消息平台、编辑器或批处理任务。
2. Hermes 创建或恢复一个会话，把历史消息、人格设定、项目上下文、记忆和技能索引准备好。
3. Agent Loop 组装系统提示词，决定本轮该用哪个模型、哪种 API 形式。
4. 模型先做判断：直接回答，还是调用工具。
5. 如果要调用工具，Tool System 负责查注册表、做安全检查、执行并回填结果。
6. 如果任务很长，上下文压缩器会把中段历史折叠，但保留关键最近消息和工具对。
7. 如果过程中学到了稳定方法，Hermes 会把经验沉淀进记忆，或者进一步长成技能。
8. 如果输入来自 Gateway，结果会被发回 Telegram、Slack、Discord、WhatsApp 等平台。
9. 如果任务来自 Cron，结果会被定时送到目标渠道或本地输出目录。
10. 整个过程中，安全层负责审批危险操作、限制未授权用户、防止上下文文件把系统提示词带偏。

# 最小可运行形态
- `run_agent.py`：Agent 主循环，真正的“大脑中枢”。
- `cli.py`：终端交互入口，适合本地直接对话。
- `gateway/`：消息平台网关，负责把 Hermes 接到 Telegram、Slack、Discord 等外部世界。
- `agent/`：Prompt、压缩、缓存、记忆管理等内部子系统。
- `tools/`：所有工具的注册与执行层。
- `hermes_state.py`：会话与状态持久化，底层使用 SQLite 和 FTS5。
- `skills/` 与 `optional-skills/`：内置技能与可选技能目录。
- `cron/`：调度器，让 Agent 变成一个会定时做事的常驻系统。
- `acp_adapter/`：编辑器集成层，让 Hermes 能进入 VS Code、JetBrains、Zed 这类环境。
- `plugins/memory/`：外部记忆提供者扩展层，比如 Honcho、Hindsight、Mem0 一类能力。

# 子模块拆解
## 模块 1：入口层（CLI / Gateway / ACP / API / Batch）
- 解决的痛点：很多 Agent 只能待在单一界面里，一旦换渠道，状态、能力和行为就全断了。
- 核心职责：把不同入口统一收敛到同一个 AIAgent 核心，而不是为每个平台各写一套 Agent。
- 主要组成：终端交互入口、消息网关入口、编辑器 ACP 入口、API Server、批处理入口。
- 它负责什么：接收输入、创建会话、挂接回调、把最终结果送回各自表面。
- 为什么重要：这决定 Hermes 不是“某个聊天框里的助手”，而是“同一个 Agent 在多个表面工作”。
- 信息图建议：把 CLI、Telegram、Slack、JetBrains、Batch 都画成不同形状的入口箭头，汇聚到中心大脑。

## 模块 2：Agent Loop（任务主循环）
- 解决的痛点：普通聊天模型只负责生成文本，但 Agent 需要处理工具调用、回合控制、中断、失败重试和任务收敛。
- 核心职责：维护一次任务从“接收输入”到“返回结果”的完整循环。
- 主要组成：消息历史维护、模型调用、中断控制、工具执行循环、预算控制、回合结束判定。
- 它负责什么：决定当前是继续思考、继续调工具、压缩上下文，还是该停下来给出最终答案。
- 机制上最关键的一点：Hermes 的所有入口最后都走同一个 AIAgent 主循环，所以 CLI、Gateway、ACP 共享的是同一个核心，而不是几个平行实现。
- 信息图建议：画成一个环：用户输入 → Prompt 组装 → 模型判断 → 工具执行 → 结果回填 → 再次判断 → 最终输出。

## 模块 3：Prompt 组装层（人格 + 记忆 + 项目上下文）
- 解决的痛点：Agent 不稳定，常常不是模型不够强，而是上下文装配混乱，今天像助手，明天像脚本，后天像另一个人。
- 核心职责：把系统提示词稳定地组装出来，而且尽量不在会话中间频繁抖动。
- 主要组成：`SOUL.md`、`MEMORY.md`、`USER.md`、技能索引、项目上下文文件、工具说明、模型特定提示。
- 它负责什么：告诉 Hermes “你是谁、你记得什么、当前项目有什么约束、你手上有哪些能力、这轮任务该怎么做”。
- 一个很像工程系统的设计点：Hermes 采用“冻结快照”思路，会话开始时把记忆和上下文注入进去，中途不轻易改动，目的是保住前缀缓存和推理稳定性。
- 信息图建议：把这一层画成一个“提示词装配台”，不同来源的文档像零件一样拼到系统提示词里。

## 模块 4：Provider 路由层（模型与 API 模式选择器）
- 解决的痛点：不同模型提供商的接口格式、稳定性、价格和能力差异很大，Agent 不能把自己绑死在一个提供商上。
- 核心职责：根据模型、提供商和运行配置，决定这次请求该怎么发出去。
- 主要组成：provider 解析、凭证管理、API 模式选择、别名映射、失败切换。
- 它负责什么：把“想用哪个模型”翻译成“该走哪个 API、用什么凭证、出了错切到哪里”。
- 三种关键运行模式：兼容 OpenAI 的 chat 方式、OpenAI 的 Responses/Codex 方式、Anthropic 的 Messages 方式。
- 信息图建议：画成一个“模型路由器”，左边是 Hermes，右边是多个模型提供商，中间有一层策略开关。

## 模块 5：Tool System（工具注册与调度总线）
- 解决的痛点：Agent 真正难的不是会不会说“我要调用工具”，而是如何稳定找到工具、校验参数、执行并把结果正确喂回模型。
- 核心职责：提供统一工具注册表，让模型调用能力变成一套可管理、可扩展、可观察的系统。
- 主要组成：工具注册表、schema 收集、调用分发、结果包装、错误处理。
- 它负责什么：把 Web 搜索、文件读写、浏览器、记忆、待办、代理委派、Cron 管理、消息发送等能力暴露成统一接口。
- 一个重要细节：有些工具并不是普通外设，而是会直接改 Agent 自身状态，例如 memory、todo、session_search、delegate_task，这些能力更像“大脑内置器官”。
- 信息图建议：画成中心总线，向外连出 Web、File、Browser、Todo、Memory、Delegate、MCP 等能力卡片。

## 模块 6：Execution Backends（执行后端与运行环境）
- 解决的痛点：Agent 如果只能在当前机器裸跑，就很难兼顾安全、复现性和远程持续运行。
- 核心职责：给工具执行提供不同环境，不同任务可以跑在本地、容器、远程主机或云沙箱里。
- 主要组成：local、Docker、SSH、Singularity、Modal、Daytona 六类终端后端。
- 它负责什么：承接终端命令、文件操作、代码执行等需要“真环境”的工作。
- 为什么这一层特别重要：Hermes 不是把“工具”停留在函数调用层，而是把执行环境本身做成了可切换的模块，所以它天然适合云上常驻和远程自治。
- 信息图建议：把工具层下面画出六个地基：本机、容器、远程机器、无服务器云环境、持久远程工作区、高性能容器。

## 模块 7：Memory & Session（持久记忆、会话搜索、上下文压缩）
- 解决的痛点：Agent 如果只靠当前窗口的聊天历史，就会反复失忆；但如果什么都塞进 prompt，又会迅速膨胀、变慢、变贵。
- 核心职责：在“始终记得关键事实”和“不要把上下文撑爆”之间做平衡。
- 主要组成：`MEMORY.md`、`USER.md`、SQLite 会话库、FTS5 搜索、上下文压缩器。
- 它负责什么：保存长期有效的信息，搜索过去会话，在上下文快满时压缩中段历史，并保证关键最近消息不丢。
- Hermes 的设计很像“两层大脑”：一层是小而精的固定记忆，一层是几乎无限的会话档案库，需要时再搜索召回。
- 信息图建议：画成“双层记忆结构”：上层是短小精悍的长期记忆卡片，下层是巨大的会话档案仓库，中间有压缩和搜索通道。

## 模块 8：Skills System（程序化记忆与能力生长）
- 解决的痛点：如果 Agent 每次做复杂任务都从头试错，它就永远像一个“只会临场发挥的实习生”。
- 核心职责：把可复用的方法论保存成技能，并在以后按需加载。
- 主要组成：技能索引、`SKILL.md` 主文档、分层加载、技能管理工具、Skills Hub、外部技能目录。
- 它负责什么：让 Hermes 知道“我有哪些现成套路可用”，并在任务复杂时把一次成功经验沉淀下来。
- Hermes 在这里最像“学习型 Agent”：技能不是静态插件而已，它还是 Agent 的程序化记忆，复杂任务完成后可以长出新的技能，旧技能也能被修补。
- 信息图建议：把技能画成一个知识库抽屉柜，第一层是技能目录卡，第二层是按需展开的技能手册，第三层是新生长出来的技能文件。

## 模块 9：Gateway & Delivery（多平台消息网关）
- 解决的痛点：Agent 一旦跨平台，就会遇到会话路由、用户鉴权、消息格式、结果回传和平台差异的问题。
- 核心职责：把 Hermes 变成一个能生活在 Telegram、Slack、Discord、WhatsApp、Signal、Email、Matrix 等多个表面上的常驻实体。
- 主要组成：平台适配器、会话存储、消息投递、授权判断、配对系统、状态追踪。
- 它负责什么：接收平台事件、找到正确会话、把消息交给 AIAgent，再把执行结果投递回平台。
- 这一层的本质不是“加几个 Bot 适配器”，而是把平台世界统一成一个消息总线，让同一个 Agent 跨表面延续。
- 信息图建议：把 Hermes 画在中间，四周是 Telegram、Slack、Discord、WhatsApp 等平台，线条上标记“路由、授权、回传”。

## 模块 10：Cron Scheduler（定时自动化）
- 解决的痛点：很多 Agent 只能被动等人提问，不能像一个真正的系统那样按节奏主动工作。
- 核心职责：让 Hermes 能周期性运行任务，而且任务本身仍然是 Agent 任务，不是单纯 shell 脚本。
- 主要组成：jobs 存储、tick 调度器、运行元数据、技能附着、目标投递。
- 它负责什么：按自然语言或 cron 规则创建任务，定时启动一个新 Agent 会话，加载需要的技能，跑完后把结果送回指定渠道。
- 这个模块让 Hermes 从“会话式 Agent”变成“持续运行的 Agent 服务”。
- 信息图建议：画成时钟驱动的任务轮盘：时间到 → 拉起新会话 → 注入技能 → 运行 → 投递结果。

## 模块 11：Security & Isolation（安全护栏）
- 解决的痛点：一旦 Agent 拥有终端、文件、消息和自动化能力，它就不再只是一个会说话的模型，而是一个可能真的“做事”的系统。
- 核心职责：给 Hermes 装上多层刹车，防止未授权访问、危险命令、提示词注入和跨会话污染。
- 主要组成：危险命令审批、用户授权与 DM 配对、容器隔离、上下文文件扫描、输入参数净化、跨会话隔离。
- 它负责什么：在真正执行前做边界判定，在真正接入外部世界前先确认“谁能说话、能做到哪一步、哪些内容不能被当作可信上下文”。
- 很多系统把安全只理解为“输出审核”，Hermes 更像是在做 Agent Runtime 安全：输入、执行、记忆、渠道、容器几层同时设防。
- 信息图建议：画三层护城河：外层是用户授权，中层是命令审批，内层是容器与上下文隔离。

## 模块 12：Plugin / MCP / External Memory（扩展层）
- 解决的痛点：Agent 能力如果全写死在仓库里，扩展速度和场景适配都会很快撞墙。
- 核心职责：让 Hermes 在不改核心主循环的前提下接入外部工具、外部记忆和第三方生态。
- 主要组成：MCP 工具接入、插件发现机制、外部记忆提供者、第三方技能源。
- 它负责什么：把 GitHub、数据库、文件系统、浏览器栈、知识图谱式记忆和外部能力安全地挂到 Hermes 上。
- 这一层说明 Hermes 不只是一个产品，也是一套可增长的 Agent 平台。
- 信息图建议：画成外围生态接口层，外面连 MCP Server、Honcho、Hindsight、Mem0、Skills Hub 等外部节点。

# 这些模块之间是怎么协作的
如果要把整套 Hermes 讲清楚，最重要的是说明它不是“十几个功能点并排摆着”，而是一条连起来的链路：

- 入口层把请求送进来。
- Agent Loop 接手这次任务。
- Prompt 组装层把人格、记忆、技能和项目上下文拼起来。
- Provider 路由层负责找对模型和接口。
- Tool System 在需要行动时调工具。
- Execution Backends 提供真实执行环境。
- Memory 记录关键事实，Session Search 负责回忆历史。
- Skills 把成功流程沉淀成以后可复用的方法。
- Gateway 和 Cron 负责让这个 Agent 在更多场景下持续存在。
- Security 层始终横切全局，给所有链路上锁。

这一点特别适合做成封面图上的一句话：

- Hermes 不是“模型 + 工具”的二元组合，而是“会话、记忆、技能、执行、安全、调度”联动起来的完整 Agent Runtime。

# 常见误区
## 误区 1：Hermes 只是一个更强的命令行聊天工具
不完全对。CLI 只是它的一个入口。真正的核心是 AIAgent 主循环、网关、技能、记忆、Cron 和执行后端这一整套基础设施。

## 误区 2：Hermes 的记忆等于向量数据库
不对。Hermes 自带的是有边界的文件化持久记忆和基于 SQLite 的会话搜索；外部记忆提供者是外挂层，不是唯一核心。

## 误区 3：技能就是提示词模板
不够准确。技能在 Hermes 里更接近程序化记忆：按需加载、可复用、可被 Agent 自己创建和修补。

## 误区 4：多平台支持只是“多接几个机器人渠道”
也不够准确。Gateway 的难点不是接入多少平台，而是如何在不同平台之间保持统一会话路由、授权和交付行为。

# 现实取舍
- 模块越完整，Hermes 越像真正的运行时，但系统复杂度也越高。
- 冻结式记忆注入能保护缓存和稳定性，但代价是会话中途新增记忆不会立刻进入系统提示词。
- 工具和执行后端越强，自动化能力越高，但安全边界就必须更严格。
- 技能自动生长很有吸引力，但如果治理不好，技能库也可能膨胀、重复、过时。
- 多入口和多平台让 Hermes 更像常驻助手，但同时也把授权、状态同步和运维难度一起拉高。

- 重点：Hermes 如何在记忆、搜索和压缩之间做平衡

## 第 9 张：Skills System
- 重点：为什么技能是程序化记忆，而不是普通提示词片段

## 第 10 张：Gateway & Delivery
- 重点：为什么 Hermes 能跨 Telegram、Slack、Discord 等表面延续同一个 Agent

## 第 11 张：Cron Scheduler
- 重点：为什么 Hermes 可以从“会话助手”变成“持续运行的 Agent 服务”

## 第 12 张：Security & Isolation
- 重点：为什么 Agent 安全不是一个审核器，而是一整套运行时边界

## 第 13 张：Plugin / MCP / External Memory
- 重点：为什么 Hermes 不只是一个应用，而是一个还能继续长能力的平台

# 一句话总结
如果只看表面，Hermes 像一个能聊天、能调工具的 Agent；但把对象拆开看，它其实是一套由入口、主循环、提示词装配、模型路由、工具总线、执行后端、记忆系统、技能系统、消息网关、定时调度和安全护栏共同构成的学习型 Agent Runtime。
