# Claude Code 实战指南：把 AI 变成“可控的工程队友”

你可能已经体验过：同样是 Claude Code，有时候它像神助攻，几分钟把脏活累活清掉；有时候又像“过度热心的实习生”，一口气生成一堆文件、抽象、配置，最后你花更多时间收拾残局。问题往往不在模型“够不够聪明”，而在你有没有把它放进正确的工程流程里。

这篇文章把原文里最有用、最可复用的部分，整理成一套你可以直接照做的工作方法（来源见文末）。

---

## 一、先别打字：先“想清楚”，再让它写

Claude Code 的最大误区之一，是一进来就开始描述需求、让它开写。更稳的路径是：

- 先想清楚目标与边界：要交付什么？不做什么？成功标准是什么？
- 先想清楚结构（架构/模块/数据流）：你给的结构越明确，Claude 的“自由发挥空间”越小，输出越可控。
- 再进入实现：把“规划”和“执行”拆开，效率会明显提升。

原文提到一个很实用的习惯：在 Claude Code 里进入 plan mode（规划模式）花 5 分钟，往往能省下后面数小时的返工和 debug。

---

## 二、架构不是“宏大叙事”，而是把需求说具体

“帮我做个登录系统”这种话，对人类都太宽，对模型更是灾难。更好的写法是把“路径”也描述出来：

- 说清楚已有资产：复用哪个 User 模型/表？现有项目用什么框架？
- 说清楚关键决策：会话存 Redis 还是 JWT？过期策略？路由保护范围？
- 说清楚交付形态：改哪些文件？新增哪些 endpoint？需要哪些中间件？

你描述得越像一份可执行的技术方案，Claude 的输出就越像“按图施工”，而不是“脑补一个它认为合理的系统”。

---

## 三、把 CLAUDE.md 当成你的“杠杆点”

原文强调：每次启动 Claude Code，它会先读你的 CLAUDE.md（项目级指令/约束）。这意味着它是影响每一次输出的“总开关”。

写好 CLAUDE.md 的关键原则：

- **短**：模型可稳定遵循的指令数量有限；写成小说，它会开始随机忽略其中一部分，你还不知道忽略了哪条。
- **只写项目特有的**：别解释通用概念（比如什么是 components）；写那些“你不说它就不知道”的东西，比如真正要跑的脚本命令、约定俗成的目录职责、你们的发布/回滚流程。
- **写“为什么”**：只写“要做什么”不够。写清楚原因（例如“开启 TS strict 是因为线上出过 implicit any 的 bug”），模型在做权衡时会更像你。
- **持续更新**：当你发现自己第二次纠正同一件事，就把它写进 CLAUDE.md；让它变成你的“失忆自救笔记”，而不是新员工入职文档。

---

## 四、上下文会“提前劣化”：20%～ 40% 就可能开始变差

很多人以为 200k tokens 就是“能塞满再说”，但原文提醒：质量会在远没满的时候开始下降。结果就是你越聊越乱、越补充越差，最后只能 /compact 但也救不回来。

更稳的上下文策略：

- **一事一聊**：一个会话只做一个 feature/任务，别在同一个会话里既做登录又重构数据库层。
- **外部记忆**：把计划、进度、关键结论写到文件里（比如 plan/notes 一类），让它跨会话可读、可恢复。
- **复制粘贴重置法**：当上下文臃肿时，把关键终端输出/结论复制出来 → /compact 得到摘要 → /clear 清空 → 只粘贴“关键事实+目标+约束”再继续。
- **知道何时清空**：会话跑偏了就 /clear，通常比硬救更省时间；因为 CLAUDE.md 还在，你不是从零开始。

一个有用的心智模型：Claude 是“无状态”的，你不给它的，它就看不见。

---

## 五、提示词不是玄学：把“输入”工程化

原文给了几条很硬的工程建议，核心是把输入写成“可验收的任务单”：

- **具体**：越具体越好，尤其是接口、数据结构、约束、验收标准。
- **明确禁止项（告诉它别做什么）**：Claude（尤其某些模型）有“过度工程”的倾向。你要的是最小改动，就直接说：
  - “保持简单，不要额外抽象”
  - “尽量一个文件解决”
  - “不要引入我没要求的框架/模式”
- **永远交叉验证**：把 AI 当加速器，不是替代者。它会犯错，尤其会“自信地错”。你需要养成 review 输出的习惯，防止把技术债自动化。
- **给出动机与约束**：例如“这个逻辑跑在每次请求上，要优先性能”，或“这是一次性原型，可牺牲扩展性”，模型会做不同取舍。

原文一句话很尖锐但很真实：输出来自输入；如果输出很差，通常是输入不够清晰、不够可执行。

---

## 六、模型要分工：一个负责想，一个负责干

原文提到一个常见有效的工作流：

- 用更强推理的模型做：方案、取舍、架构、步骤拆分
- 用更快更便宜的模型做：按计划执行、写样板、重构落地、批量改动

关键不是“永远用最强”，而是把成本花在“决定怎么做”上，把执行交给更快的那一个。

---

## 七、别只会聊天：把 Claude 接进你的工作流

原文强调：最有价值的用法不是一次性问答，而是把 Claude 当作流程里的一个组件：

- **MCP（连接外部系统）**：如果你总在复制 Slack/GitHub/数据库信息进来，说明可以用连接器自动取数。
- **Hooks（自动运行动作）**：在 Claude 修改文件前/后自动跑格式化、typecheck、测试，把问题即时暴露，减少堆积的技术债。
- **自定义斜杠命令**：把高频任务（debug、review、deploy）封装成可重复调用的命令，减少你每次手写提示词的成本。
- **Headless 模式（-p）**：让 Claude 非交互运行，把它接进脚本、CI、自动化链路里（例如 PR review、工单回复、日志/文档更新），并且可审计、可迭代优化。

这会形成“飞轮”：它犯错 → 你从日志/结果里总结规则 → 改进指令/工具链 → 下次更稳。模型不变，系统会越来越强。

---

## 八、卡住时别硬刚：识别循环，立刻换打法

原文描述了一个常见症状：Claude 反复失败、反复尝试、或者一直在错误方向上自信输出。此时更有效的动作是“换方法”，而不是“加大解释力度”。

可用的止损动作：

- /clear：清掉累积误导上下文
- 拆小任务：先把最小闭环做对，再组合
- 给最小示例：用“展示成功长什么样”替代抽象描述
- 重构问题表述：换一种框架表达（例如“用状态机实现”而不是“处理这些转换”）
- 三次解释还不通：停止解释，改变策略

---

## 结语：你要构建的是“可控协作”，不是“神奇咒语”

把 Claude Code 用好，靠的不是更长的对话，而是更清晰的结构：先规划、再执行；用 CLAUDE.md 固化约束；把上下文当资源管理；把提示词当需求文档；把工具能力接进工作流。这样你会从“和工具搏斗”，变成“在流程里顺滑地放大产出”。
