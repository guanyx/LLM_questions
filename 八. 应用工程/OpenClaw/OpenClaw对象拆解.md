## 极简定义
OpenClaw 是一个基于“文件化工作区即系统状态”的开源自治代理运行时框架，主要用于解决 AI 助理跨会话持续运行、任务可恢复、能力可扩展与行为可审计的工程问题。

## 差异点
- 对比：无状态 Chat 助手 | 抓手：工作区契约文件 `AGENTS.md/SOUL.md/IDENTITY.md/TOOLS.md` 固化身份与执行边界 | 结果：行为可复现、可迁移，团队协作可审计  
- 对比：DB/向量库记忆型 Agent | 抓手：文件化状态机 `memory/active-tasks.md` + 日志 `memory/YYYY-MM-DD.md` + 精炼 `MEMORY.md/lessons.md` | 结果：重启先恢复而非重来，任务断点续跑成本显著降低  
- 对比：传统脚本/编排自动化 | 抓手：模型决策 + `skills/` 流程封装 + 渠道/集成适配器 | 结果：虽引入路由与安全治理负担，但换来“自然语言驱动的可组合自动化”与更快的任务覆盖扩展

## 最小运行链路
1. 输入进入 → Slack/Discord/Telegram 事件 → 渠道适配器 → gateway 进程接收  
2. 会话启动 → 读取 `memory/active-tasks.md` → 判定恢复优先级  
3. 【若存在 in-progress】恢复路径 → 装载 `AGENTS.md/SOUL.md/TOOLS.md` → 生成续跑计划 → 执行并更新 `active-tasks.md`  
4. 【否则】新请求路径 → 装载 `AGENTS.md/SOUL.md/USER.md/IDENTITY.md/TOOLS.md` → 生成行动计划与工具调用意图  
5. 路由执行 → 命中 `skills/` 中技能流程或直接集成调用 → 选用 GitHub/Jira/Google Workspace 等连接器  
6. 产出落地 → 结果写入 `outputs/`（或写回外部系统）→ 同步给渠道回复  
7. 记录沉淀 → 追加写入 `memory/YYYY-MM-DD.md` → 必要时更新 `lessons.md` 与 `MEMORY.md`  
8. 周期驱动 → HEARTBEAT 触发读取 `HEARTBEAT.md` → 自检任务停滞/上下文膨胀 → 归档与清理（可由 cron 驱动）  

## 最小可运行形态
- `workspace/`：单一工作区根目录，承载全部状态与产出  
- `AGENTS.md`：代理操作手册与权限边界，定义可自主/需审批事项  
- `SOUL.md`：稳定工作风格与输出规范，抑制行为漂移  
- `IDENTITY.md`：代理身份与角色定位，决定“我是谁/服务谁”  
- `TOOLS.md`：环境说明与命令约定，决定“怎么在本机做事”  
- `HEARTBEAT.md`：周期自检清单，驱动自治维护动作  
- `memory/active-tasks.md`：崩溃恢复入口与任务状态机  
- `memory/YYYY-MM-DD.md`：运行日志与决策轨迹，提供可追溯性  
- `skills/`：技能目录，封装可复用流程与路由入口  
- `outputs/`：产出物目录（文档/代码/导出物），用于交付与回放  
- `openclaw gateway`（进程/命令）：通道接入与模型调用的会话中枢  
- 环境变量（如 `ANTHROPIC_API_KEY`）：凭证载体，避免写入工作区文件  

## 子模块拆解
### 模块 1：工作区契约与身份配置（文件即系统状态入口）
- 解决的痛点：代理行为依赖临时上下文会漂移，迁移与协作不可控。  
- 核心组件：`AGENTS.md`、`SOUL.md`、`USER.md`、`IDENTITY.md`、`TOOLS.md`。  
- 生命周期/状态：启动读取 → 会话内作为硬约束 → 文件变更后重载/重启生效。  
- 性能/扩展性指标：规则覆盖率、变更生效延迟、跨项目复用成本。  

### 模块 2：运行时与网关（连接与编排中枢）
- 解决的痛点：模型调用、通道连接、会话管理分散会导致不稳定与不可恢复。  
- 核心组件：gateway 进程、provider 适配层、会话管理器。  
- 生命周期/状态：启动初始化 → 建立连接 → 请求处理 → 重连/重启/关闭。  
- 性能/扩展性指标：并发会话数、连接稳定性、失败重试率、端到端延迟。  

### 模块 3：渠道适配层（输入输出面）
- 解决的痛点：消息入口割裂导致上下文碎片化，结果分发不可统一治理。  
- 核心组件：Slack/Discord/Telegram 适配器、事件队列/消息通道。  
- 生命周期/状态：授权配置 → 事件接收 → 结果发送 → 失败降级/重连。  
- 性能/扩展性指标：消息延迟、重连频率、丢消息率、渠道覆盖面。  

### 模块 4：技能系统（流程复用与路由）
- 解决的痛点：每次临时规划导致结果不一致、成本攀升、难以规模化维护。  
- 核心组件：`skills/` 目录、技能路由规则、技能执行器。  
- 生命周期/状态：技能加载 → 命中触发 → 执行工具链 → 写回工作区与外部系统。  
- 性能/扩展性指标：命中率、执行成功率、技能数量增长下的可维护性。  

### 模块 5：任务状态与恢复（Crash Recovery 主开关）
- 解决的痛点：长任务中断后需要人肉对齐，自治循环易“失忆停摆”。  
- 核心组件：`memory/active-tasks.md`、任务状态约定（in-progress/blocked/completed）。  
- 生命周期/状态：更新任务状态 → 中断 → 启动优先读取 → 恢复续跑 → 状态迁移。  
- 性能/扩展性指标：恢复成功率、任务陈旧率（stale）、状态一致性错误率。  

### 模块 6：记忆分层与可追溯日志（从事件到经验）
- 解决的痛点：只保留对话上下文会丢历史决策，复盘与积累困难。  
- 核心组件：`memory/YYYY-MM-DD.md`、`MEMORY.md`、`lessons.md`。  
- 生命周期/状态：事件追加 → 周期精炼 → 形成长期记忆与经验规则 → 后续会话复用。  
- 性能/扩展性指标：噪声比例、可检索性、文件膨胀速度、复用命中率。  

### 模块 7：心跳与定时自动化（自治维护节拍器）
- 解决的痛点：无人值守运行会出现任务停滞、输出堆积、上下文膨胀且不可见。  
- 核心组件：`HEARTBEAT.md`、cron 作业、`metrics.json`（若启用）。  
- 生命周期/状态：周期触发 → 自检执行（续跑/归档/清理/测试）→ 记录指标与结果。  
- 性能/扩展性指标：自检耗时、误触发率、维护收益（stale 降幅）、运行成本。  

### 模块 8：安全姿态与权限边界（可控自治）
- 解决的痛点：代理具备工具与集成能力后，缺少边界会带来不可逆风险。  
- 核心组件：`AGENTS.md`（审批边界）、环境变量凭证、gateway token/auth 配置（若启用）。  
- 生命周期/状态：启动加载边界 → 执行前策略判定 → 违规阻断/降级 → 事后记录。  
- 性能/扩展性指标：敏感操作拦截率、误拦截率、凭证泄漏风险暴露面、最小权限覆盖率。  

## 边界与非目标
- 不提供模型能力本身，不解决训练/对齐/模型质量问题，默认依赖外部模型提供方与其 SLA。  
- 不内置企业级 RBAC、审计与合规闭环，默认由外部系统与 `AGENTS.md` 的审批边界补齐。  
- 不保证工具调用绝对安全，虽然文件化边界提升可控性，但外部输入与集成仍需按不可信处理。  
- 默认文件化记忆缺少加密与访问控制，敏感信息应放在环境变量或外部密钥系统而非工作区文件。  
- 不解决高可用与大规模分布式调度，水平扩展通常需要外部编排/队列/共享存储体系补齐。
