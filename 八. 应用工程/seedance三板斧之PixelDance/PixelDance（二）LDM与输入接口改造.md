# PixelDance（二）：LDM 是发动机，PixelDance 改的是“输入接口”

上一章解释了 PixelDance 的任务改写：从“向未来预测”变成“在首尾帧约束下做视频填空”。这一章把镜头推进到架构层，回答两个问题：

- 为什么它能算得动（LDM 的性能地基）
- 它到底改了哪些“输入口”，让模型更听话

## 1. 技术地基：从“画像素”到“编代码”（LDM）

LDM（Latent Diffusion Model）解决的是一个很工程的问题：**直接在像素空间生成视频太贵**。

- 一帧 1080P 是几百万像素，几十秒视频就是几百帧
- 如果把每个像素都当成生成对象，算力会被直接打爆

LDM 的策略是把生成发生在“压缩空间”里：

1.  **压缩（Encode）**：把像素帧压缩成 latent（可以理解成更小、更抽象的特征图）
2.  **生成（Diffusion）**：在 latent 上做去噪生成
3.  **还原（Decode）**：把生成好的 latent 解码回像素帧

把这三步换成常见组件名，就是：

- Encode/Decode：由 **VAE** 完成（像素帧 ↔ latent 帧）
- 去噪网络：通常是一个 **U-Net**（做视频时会扩成能看时间的版本）
- 文本条件：由 **Text Encoder（常见是 CLIP 文本编码器）**提供

## 2. 字节的魔改：重构输入接口

如果把标准文生视频 LDM 抽象成一个函数，它的输入往往只有文本：

```python
generate_video(prompt)
```

这类接口的上限很明确：它只能“理解意图”，很难“遵守边界条件”。

PixelDance 的做法是把输入接口扩展成“多模态约束入口”，使其同时接收：

- 文本：表达语义与风格
- 首尾帧：表达结构边界（起点与终点）
- 指令图（可选）：表达更强的动作或布局约束

## 3. 条件到底怎么进网络：两种注入方式

为了支持这个多模态接口，关键在于“条件注入”。可以把它拆成两种力度不同的方式：

### 3.1 软引导：Cross-Attention 注入（Text / Embedding）

- 条件被编码成 embedding（如 `c_text` 或 `c_img`）
- 在 U-Net 的中间层通过 Cross-Attention 影响去噪过程

它更像“强提示”：对结果影响很大，但不天然保证像素级对齐。

### 3.2 强约束：Latent Concat（把条件当成额外输入变量）

另一种方式是把首尾帧 latent（`z_start / z_end`）直接以 Concatenation 的形式拼到去噪网络输入（或特定层）的通道维度上。

- 它更像“额外输入变量”，通常约束更强
- 但是否接近“硬约束”取决于训练策略：条件是否加噪、是否 dropout、是否做显式重建等

## 4. 本篇小结：PixelDance 不是换引擎，而是把“方向盘和刹车”接上去

- LDM 提供了“能算得动”的底座（压缩空间生成）
- PixelDance 的核心改造是：把输入接口从“只有 prompt”变成“多模态边界条件 + 意图”，并配套设计条件注入方式
