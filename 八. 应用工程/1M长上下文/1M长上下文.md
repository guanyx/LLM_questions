# 把 DeepSeek-R1-Distill-Llama-70B 的输入上下文塞满，要多少显存和 FLOPs？

这次把示例模型换成 DeepSeek-R1-Distill-Llama-70B。

文章主线还是同一件事：

- 不看 API 售价
- 只看硬件资源这笔账
- 把模型可用的输入上下文塞满时，显存、prefill 和 decode 到底各要付出什么代价

这篇文章算的不是“怎么把吞吐做快”。
而是：

- 如果你真想把这台模型按最大输入长度去跑，机器到底要替你扛多少成本

先把结论压成四句话：

- DeepSeek-R1-Distill-Llama-70B 是一台典型的大型 dense decoder-only 模型，不是混合线性注意力架构
- 它的上下文长度是 131,072 token，也就是常说的 128K
- 在这个口径下，fp16 KV cache 约 42.95 GB，整轮 prefill 约 4.05 × 10^16 FLOPs
- 我这里继续拿 5090 做换算，不是因为它真是这台模型的部署目标硬件，而是因为大多数读者对消费级显卡更有体感；所以这些时间更适合拿来建立量级直觉

---

## 先把口径钉死

下面所有数字，都按这组假设来：

- 模型：DeepSeek-R1-Distill-Llama-70B
- 场景：纯文本推理
- 上下文：按模型支持的 131,072 token 计算
- FLOPs 口径：一次乘加记作 2 FLOPs
- 5090 口径：按约 419 TFLOPS FP16 Tensor 理论峰值、约 1.79 TB/s 显存带宽粗算

模型结构按公开配置取这一组参数：

- 总参数量：70B
- 层数：80
- hidden size：8192
- attention heads：64
- KV heads：8
- head dim：128
- FFN intermediate：28672

这里先说明一句。

DeepSeek-R1-Distill-Llama-70B 是基于 Llama 3.3 70B Instruct 蒸馏出来的 dense 模型。
也就是说，它这里不是“24 层线性注意力 + 8 层标准注意力”那种混合路线。
它本质上还是：

- 80 层 full attention

所以它的长上下文成本，会比前面那种混合架构更硬、更直白。

文里所有“折算到 5090 的时长”，都要读成：

- 面向读者体感的量级尺子
- 真实系统通常只会更慢

因为现实里还没把这些东西算进去：

- runtime buffer
- workspace
- allocator 碎片
- kernel 调度损耗
- 多卡通信
- 框架差异

---

## 第一笔账：显存先怎么炸

对 70B 模型来说，第一笔账甚至不是 KV cache。
而是：

- 权重本体就已经重得离谱

所以这一节必须把显存拆开看。

### 1. 模型权重本身

按 70B 参数粗算：

- fp16 / bf16：约 140 GB
- fp8：约 70 GB
- 4 bit：约 35 GB

这还只是权重本体。
还没算运行时 buffer、workspace 和框架额外占用。

也就是说，别说 5090 这种 32GB 显卡。
哪怕你把 KV cache 完全忽略掉：

- fp16 权重本体都已经远超单卡容量

### 2. 131,072 token 下的 KV cache

DeepSeek-R1-Distill-Llama-70B 是标准 dense attention。
所以 KV cache 直接按标准公式算：

- KV cache = 上下文长度 × 2 × 层数 × KV heads × head dim × 每元素字节数

代入：

- 上下文长度 = 131,072
- 层数 = 80
- KV heads = 8
- head dim = 128

先看每个 token 留下多少 KV：

- 2 × 80 × 8 × 128 = 163,840 个元素

于是整段 128K 上下文的 KV cache 大约是：

| KV 精度 | 131,072 token |
| --- | ---: |
| fp16 / bf16 | 42.95 GB |
| fp8 | 21.47 GB |
| 4 bit | 10.74 GB |

这个数字已经很有画面感了。

对于一台 70B dense 模型来说：

- 权重本体是第一座山
- 42.95 GB 的 fp16 KV cache，是第二座山

### 3. 这台模型已经在用 GQA，不然账会更难看

这台模型是：

- 64 个 Q heads
- 8 个 KV heads

也就是说，它已经做了 GQA。

如果不是 8 个 KV heads，而是 64 个 KV heads，全都一一对应，那么同样 131,072 token、同样 fp16 下，KV cache 会直接变成：

- 343.60 GB

而现在它是：

- 42.95 GB

也就是：

- 已经先缩到了 1/8

这点很重要。

不是因为 GQA 听起来高级。
而是因为对于 70B 这种大模型，不做 GQA，KV cache 根本难看得没法上线。

### 4. 权重和 KV 加在一起看

如果只是做一个很粗的静态账面加法：

| 组合 | 总占用 |
| --- | ---: |
| 权重 fp16 + KV fp16 | 182.95 GB |
| 权重 fp8 + KV fp8 | 91.47 GB |
| 权重 4 bit + KV fp8 | 56.47 GB |

所以这里拿 5090 去看，最关键的意义不是：

- “它能不能快”

而是：

- “让读者对这笔显存账先有一个消费级硬件尺度上的体感”

这也是为什么 70B dense 模型一谈长上下文，第一反应从来都该是：

- 先看显存账

---

## 第二笔账：prefill 到底要多少 FLOPs

现在再来看第二笔账。

你把 131,072 token 一次送进去，模型不是在“把这段文本浏览一遍”。
它是在：

- 80 层里反复做投影
- 80 层里反复做 FFN
- 80 层里对整段序列做 full attention

这就是 prefill 真正贵的地方。

### 1. 每层的线性项先有多大

对这台 70B 模型，单层每个 token 的主要线性开销，大约有：

- Q / K / V / O 投影：301,989,888 FLOPs
- FFN：1,409,286,144 FLOPs

合起来单层每个 token 大约：

- 1,711,276,032 FLOPs

把 80 层都算进去，就是每个 token：

- 1.37 × 10^11 FLOPs

再乘 131,072 token，得到 prefill 里线性项总计大约：

- 1.79 × 10^16 FLOPs

### 2. 最贵的还是 attention core

对 dense full attention 来说，真正吓人的主项还是：

- QKᵀ
- attention × V

按本文的 FLOPs 口径，这部分在 80 层、131,072 token 下大约是：

- 2.25 × 10^16 FLOPs

### 3. 合起来看整轮 prefill

于是整轮 prefill 大约是：

| 部分 | FLOPs |
| --- | ---: |
| 投影 + FFN | 1.79 × 10^16 |
| Attention core | 2.25 × 10^16 |
| 合计 | 4.05 × 10^16 |

折到 5090 的 419 TFLOPS 理论峰值下界：

- 4.05 × 10^16 / 4.19 × 10^14 ≈ 96.6 秒

也就是：

- 大约 1.61 分钟

这还只是理论下界。
这里继续拿 5090 做换算，不是为了说明“5090 就该拿来跑它”。
它只是告诉你：

- 就算完全不考虑显存不够这件事，纯算力账也已经到分钟级了

---

## 第三笔账：decode 每个 token 还要继续烧什么

很多人会以为，prefill 最重，prefill 一结束，后面就轻了。

对一半。

对在：

- decode 的 FLOPs 不再是 n² 级整段重算

错在：

- 每生成一个 token，模型还是要重新读整段 128K 历史的 KV cache

所以 decode 看三件事就够了：

- KV 继续常驻显存
- 每 token 继续消耗算力
- 每 token 继续消耗带宽

### 1. 每生成 1 个 token，要多少 FLOPs

把 80 层都算进去，单个 decode token 的 FLOPs 大约是：

| 部分 | 每 token FLOPs |
| --- | ---: |
| 投影 + FFN | 1.37 × 10^11 |
| Attention core | 3.44 × 10^11 |
| 合计 | 4.80 × 10^11 |

折到 5090 的理论算力下界：

- 4.80 × 10^11 / 4.19 × 10^14 ≈ 0.00115 秒

也就是：

- 约 1.15 毫秒 / token

如果只看这个数字，会觉得：

- decode 算力账虽然不轻，但还不至于夸张

真正拖垮机器的，往往不是这笔算力。
而是下一笔带宽。

### 2. 每生成 1 个 token，要读多少 KV

因为每一步都要读取整段历史 KV。

所以 131,072 token 下，每个新 token 大约要碰的 KV 数据量就是：

- fp16 / bf16：42.95 GB
- fp8：21.47 GB
- 4 bit：10.74 GB

如果拿 5090 的 1.79 TB/s 理论带宽粗算，下界大约是：

| KV 精度 | 带宽下界 |
| --- | ---: |
| fp16 / bf16 | 23.99 ms / token |
| fp8 | 12.00 ms / token |
| 4 bit | 6.00 ms / token |

这时候你就能看出层次差了。

同一个 decode token：

- 纯算力下界约 1.15 ms
- 但光读 fp16 KV 的带宽下界就接近 24 ms

也就是说，长上下文下的 70B dense 模型，decode 真正更容易卡死在：

- 显存带宽

### 3. 如果连续生成 1000 个 token

把上面的账乘 1000：

| 指标 | 数值 |
| --- | ---: |
| 总 FLOPs | 4.80 × 10^14 |
| 5090 算力下界 | 1.15 秒 |
| 读 fp16 KV 带宽下界 | 23.99 秒 |
| 读 fp8 KV 带宽下界 | 12.00 秒 |

这就是为什么长上下文系统常常会有一种体感：

- 首字已经慢
- 后面生成也还是不快

prefill 在烧算力。
decode 在持续烧带宽。

## 到这里，可以把这台 70B 模型的成本压成一句话

如果你真把 DeepSeek-R1-Distill-Llama-70B 的输入长度塞到 131,072 token：

- 权重本体大约是 140 GB fp16
- 长上下文 KV cache 大约是 42.95 GB fp16
- 整轮 prefill 大约是 4.05 × 10^16 FLOPs
- 每个 decode token 大约还要 4.80 × 10^11 FLOPs
- 但 decode 真正更像是在持续读几十 GB 的 KV，而不是持续撞算力墙

换成人话就是：

- 128K 长上下文不是“多塞一点历史”
- 而是“让一组 GPU 背着百 GB 级的权重和几十 GB 级的历史状态，先做一轮分钟级的 prefill，再在每一步生成里反复把这段历史搬出来读”

## 那怎么把这三笔成本降下来

虽然这篇主线不是讲吞吐优化，但你第二个问题本来就包含：

- 当前可行的技术手段，怎么把显存、prefill 和 decode 压下来

所以这里还是按“成本账”来讲，不按“跑分优化”来讲。

### 1. 缩短活跃上下文：这是砍 prefill 最狠的一刀

如果你不是非得把 131,072 token 全部保持活跃，而是只让 64K 甚至 32K 进入真正的 attention 窗口，那么成本会很明显地下去。

先看压到 64K：

| 指标 | 131,072 | 65,536 | 收益 |
| --- | ---: | ---: | ---: |
| fp16 KV cache | 42.95 GB | 21.47 GB | 2× 更小 |
| prefill | 4.05 × 10^16 | 1.46 × 10^16 | 2.77× 更低 |
| decode / token | 4.80 × 10^11 | 3.09 × 10^11 | 1.56× 更低 |

再看压到 32K：

| 指标 | 131,072 | 32,768 | 收益 |
| --- | ---: | ---: | ---: |
| fp16 KV cache | 42.95 GB | 10.74 GB | 4× 更小 |
| prefill | 4.05 × 10^16 | 5.89 × 10^15 | 6.87× 更低 |
| decode / token | 4.80 × 10^11 | 2.23 × 10^11 | 2.16× 更低 |

为什么 prefill 省得最狠。

因为：

- KV 是线性随 n 变
- prefill 的 full attention 主项是 n²

所以对 dense 70B 来说，真正最贵的不是“模型有多大”。
而是：

- 你是不是坚持让整段 128K 全活跃

这里最常见的做法，其实不是一句“把上下文变短”就完了。
而是分成三步：

- 先把原始材料切成块
- 再从里面挑出这一轮真正相关的块
- 最后只把被选中的那一小部分送进 attention

如果讲得再具体一点，工程上通常会有三种路径。

第一种是检索裁剪。
系统先把长文档切成很多 chunk，每个 chunk 预先做 embedding，用户提问时先做召回，再只把得分最高的那几段拼进 prompt。

这条路砍掉的是：

- 真正进入 attention 的活跃 token 数

也就是说，它不是在“优化 attention 内部”。
它是在 attention 开始之前，就把大多数不相关内容挡在外面。

第二种是摘要压缩。
不是把旧内容完全删掉，而是把前面的大段历史压成几段更短的摘要，再和最近的原文片段一起送进去。

这条路适合：

- 你还想保留长期语义，但不需要保留每一个原句细节

第三种是分层记忆。
最近几轮对话保留原文，较远的历史保留摘要，再远的历史只保留索引或标签。

你可以把它理解成：

- 热数据留原文
- 温数据留摘要
- 冷数据只留目录

这三条路的共同点都是：

- 不去碰 full attention 的公式本身
- 而是想办法让真正进入公式的 n 先变小

牺牲是什么：

- 召回链路更复杂
- 一旦召回错了，模型后面再强也没用
- 摘要压缩会丢掉原句级细节
- 分层记忆会让“跨很远的细节精确对齐”更难做

### 2. 滑动窗口或块稀疏：把 full attention 改成局部 attention

如果不是让每个 token 看完整个 128K，而是只看一个局部窗口，比如 16K，那么复杂度会更像：

- n × w

在 128K 下取：

- n = 131,072
- w = 16,000

单看 attention core，大约能省：

- 8.19×

但总账不会省到这么多，因为投影和 FFN 还在。

把整笔账一起看，一个比较实用的量级是：

| 指标 | 全局 attention | 16K 窗口 attention | 收益 |
| --- | ---: | ---: | ---: |
| 总 prefill | 4.05 × 10^16 | 2.07 × 10^16 | 1.96× 更低 |
| decode / token | 4.80 × 10^11 | 1.79 × 10^11 | 2.69× 更低 |
| 理想 fp16 热 KV | 42.95 GB | 5.24 GB | 8.19× 更小 |

这条路和“缩短活跃上下文”不完全一样。

前一条的思路是：

- 先减少送进模型的 token

这一条的思路是：

- token 还是送进来了
- 但不是每个 token 都能看到全部历史

如果画成一张图，原来的 full attention 是：

- 一整块 128K × 128K 的大矩阵

而滑动窗口更像是：

- 只保留主对角线附近的一条带状区域

也就是说，位置 t 的 token，不再去看全部 128K 历史。
它只看：

- 离自己最近的那一段窗口
- 再加少量全局 token 或摘要 token

块稀疏注意力则更进一步。
它不是一条连续窗口，而是把整个注意力矩阵分成很多 block，只保留少数需要计算的 block。

比如可以保留：

- 当前块对附近块
- 当前块对章节标题块
- 当前块对摘要块

这样系统就在显式地回答一个问题：

- 哪些位置之间的依赖值得保留
- 哪些位置之间的依赖可以不算

所以这条路省下来的本质不是“小优化”。
而是：

- 把“每个 token 看所有 token”改成“每个 token 只看少量必要 token”

牺牲是什么：

- 超窗口的精确逐 token 匹配能力下降
- 你必须设计哪些全局 token 能补远距离信息
- 窗口大小、全局块位置、稀疏规则都要调
- 任务一旦很依赖远距离逐字细节对齐，效果会特别敏感

### 3. Prefix cache：它不省第一次，但能省重复 prefill

如果你总是在同一份长材料上连续追问，最浪费的事情不是回答。
而是：

- 每轮都把前面那一长段不变前缀重新算一遍

Prefix cache 省的是这笔重复劳动。

举个直观例子：

- 已经缓存了 120,000 token 的前缀
- 这一轮只新增 11,072 token

那新增这段的 prefill 大约是：

- 5.16 × 10^15 FLOPs

而不是重新来一遍整段 131,072 token 的：

- 4.05 × 10^16 FLOPs

大约就是：

- 7.8× 的节省

Prefix cache 真正的工作方式，不是“压缩了模型”。
而是：

- 把已经算过的前缀直接复用

具体做法是：

- 第一次请求进来时，系统正常做 prefill，把前缀对应的 KV cache 建出来
- 下一轮如果前缀一字不变，就不再重算那段 token 的投影、attention 和 FFN
- 只对新增那一小段 token 补算，再把它们接到旧 KV 后面

也就是说，prefix cache 解决的不是：

- 一段超长上下文本身太贵

它解决的是：

- 你为什么还要把同一段超长上下文反复算很多遍

这在两种场景里特别值钱。

第一种是：

- 围着同一份长文档连续追问

第二种是：

- 多个用户共享同一个很长的系统前缀、知识前缀或工具说明前缀

这时候真正要做好的，不只是“缓存”两个字。
而是下面几件事：

- 怎么给前缀做稳定指纹，确保完全一样才复用
- 前缀中间只要改了一个 token，后面的 cache 怎样失效
- 多租户场景里，哪些 cache 可以共享，哪些绝不能共享

牺牲是什么：

- 只对“前缀不变、多轮追问”有效
- 不减少首轮最大显存占用
- cache 生命周期管理会更复杂
- 一旦 prompt 拼接策略不稳定，缓存命中率就会很差

### 4. 冷 KV 下放到 CPU 或更低层存储：省的是 GPU 容量，不是免费

还有一条思路是：

- 热 KV 留在 GPU
- 冷 KV 下放到 CPU 内存或更慢层级

它的价值是：

- 先解决“HBM 放不下”的问题

但它没有把成本变没。
它只是把成本从：

- GPU 显存容量

换成了：

- 更高延迟
- 更差带宽
- 更复杂的系统调度

所以它更像：

- 为了能放下
- 接受跑得慢

如果把它说得再具体一点，这类系统通常会先把 KV 分层：

- 最近一段、最常访问的热 KV 放在 HBM
- 较老但可能还会访问的温 KV 放在 CPU 内存
- 更冷的部分甚至放到 NVMe 或别的更慢层级

然后在 decode 时做一件很现实的事：

- 先猜这一步最可能会访问哪些 KV
- 尽量把这部分提前搬回更快的层

本质上，这已经不只是“模型推理”。
而是在做一套小型的分层存储系统。

所以这里最大的工程难点，不是公式。
而是：

- 如何判断哪些 KV 是热的
- 如何避免搬运本身把收益吃掉
- 如何让 GPU 计算和数据回传重叠起来

如果冷热判断错了，就会出现一种很难受的情况：

- 理论上模型放下了
- 实际上每一步都在等数据回来

也就是说，这条路主要解决的是：

- 能不能放下

而不是：

- 能不能便宜地跑

### 5. FlashAttention 很重要，但别把账算错

FlashAttention 当然重要。
但它主要省的是：

- HBM 读写
- 临时显存
- kernel 常数项

它通常不会把 dense full attention 的理论 FLOPs 主项从 n² 改掉。

所以它能让真实实现更接近“可用”。
但它不是把这篇文章里那笔大账直接砍没的魔法刀。

如果你把标准 attention 想成一个巨大的中间矩阵流水线，传统写法的问题是：

- 先把很大的 attention score 矩阵写到显存里
- 再把它读出来做 softmax
- 再把结果写回去
- 再读出来乘 V

FlashAttention 做的事更像是：

- 把 Q、K、V 分成很多小 tile
- 在片上 SRAM 里一块一块地算
- 尽量不把完整的大中间矩阵落回 HBM

它改变的是：

- 数据怎么流过 GPU

而不是：

- full attention 这个问题本身是不是 still O(n²)

所以它最值钱的地方是：

- 把 HBM 读写压下去
- 把临时显存占用压下去
- 把真实实现里的常数项压下去

换句话说，它更像是：

- 让你少交“实现税”

而不是：

- 让你免交“理论税”

---

## 最后把这件事压成一句工程话

如果你把 DeepSeek-R1-Distill-Llama-70B 的输入长度真塞到 131,072 token，首先遇到的问题不是“它够不够聪明”。
而是：

- 权重能不能先放下
- 42.95 GB 的 KV cache 能不能继续放下
- 4.05 × 10^16 FLOPs 的 prefill 你愿不愿意付
- decode 时每一步反复读几十 GB KV 的带宽，你能不能接受

所以这种 70B dense 模型的长上下文成本，本质上不是一个“支持 128K”这么轻飘飘的卖点。
它更像是在说：

- 你准备让一组 GPU 背着百 GB 级权重和几十 GB 级历史状态，先做一轮分钟级 prefill，再持续为后续生成支付带宽账

这才是“长上下文为什么贵”的真正答案。
