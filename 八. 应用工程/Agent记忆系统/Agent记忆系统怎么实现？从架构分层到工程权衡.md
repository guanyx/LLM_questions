# Agent记忆系统怎么实现？从架构分层到工程权衡

很多文章讲 Agent 记忆系统，喜欢先铺很多概念：短期记忆、长期记忆、情景记忆、语义记忆、程序记忆，讲完之后读者通常会得到一种“我好像懂了”的感觉。

但真正难的部分，从来不是“知道有这些层”，而是当你真的要把系统做出来时，马上会撞上的几个问题：

- 什么信息该写进去，什么不该写？
- 记忆应该放在 prompt、数据库、向量库，还是文件系统里？
- 历史会话是整段保存，还是要抽取成事实？
- 旧记忆和新记忆冲突时，谁说了算？
- 上下文越来越长时，是继续塞，还是压缩，还是直接遗忘？
- Skills、Session Search、Memory、Workspace Notes 这些东西到底怎么分工？

这篇文章不再重点解释“记忆是什么”，而是聚焦更有价值的部分：**一个 Agent 的记忆系统，应该如何落地实现，以及不同实现路径各自的权衡是什么。**

近一轮 Agent 热点里，这个方向其实已经很明显了。OpenAI 开始强调 long-running agents、skills、compaction；Cloudflare 在推 persistent sessions、sandbox、workspace filesystem；Hermes Agent 则把 persistent memory、session search、skills system 明确拆开。它们共同说明一件事：**记忆已经不是聊天记录的附属品，而是 Agent Runtime 的核心组成部分。**

## 一、先别急着做库，先定义你到底要解决什么问题

一个记忆系统值不值得做，不取决于你用了多少“高级存储”，而取决于它有没有解决下面四类问题。

第一类，是 **跨会话连续性**。

用户第二次来时，Agent 能不能记得这个人偏好什么风格、当前项目在做什么、哪些约束不能碰。

第二类，是 **长任务稳定性**。

任务跨了几十轮、调用了十几个工具、写了很多文件之后，Agent 能不能还知道自己做到哪一步，而不是上下文一长就开始飘。

第三类，是 **经验沉淀能力**。

同类问题第二次出现时，Agent 能不能更快地做对，而不是每次都像第一次见。这里真正想要的不是“能搜到历史”，而是“能把历史变成下次可复用的方法”。

第四类，是 **成本可控**。

如果你的记忆方案让每轮调用都要扫很多历史、做大量召回和重排，最终你会得到一个“很聪明但根本跑不起”的系统。

所以，设计记忆系统的第一步，不是选 Milvus、pgvector 还是 Neo4j，而是先把验收指标钉住：

- 长任务成功率是否更高
- 跨 Session 偏好一致性是否更好
- 重复犯错率是否下降
- 工具一次成功率是否提升
- 平均 prompt 成本和延迟是否仍可接受

如果不能映射到这些指标，所谓“记忆增强”大概率只是架构自嗨。

## 二、真正能落地的 Agent 记忆架构，通常不是一层，而是四层

如果只从实现角度看，一个实用的 Agent 记忆系统通常至少要拆成四层。这里不展开讲认知学概念，只讲工程分工。

### 1. 当前工作面

这是离模型最近的一层，也就是当前 prompt、当前线程状态、当前待办、当前工具输出的组合。它解决的问题只有一个：**让 Agent 在眼前这件事上不掉线。**

这一层通常放：

- 最近几轮消息
- 当前任务目标
- 子任务状态
- 本轮关键工具输出
- 当前工作区中最相关的短笔记

这一层不该放：

- 全量历史对话
- 大量长期档案
- 已经过期的中间结果

你可以把它理解成桌面，不是仓库。桌面的要求是“好操作”，不是“能囤东西”。

### 2. 事件层

这一层保存的是“发生过什么”。它更像历史日志，但不能只是原始日志堆积，而要有事件结构。

典型内容包括：

- 某次工具调用做了什么
- 任务在哪一步失败
- 哪个方案被验证通过
- 某次会话最后产出了什么结论

这一层的价值在于回放和检索。用户说“继续上次那个部署问题”，Agent 需要能把相关事件拉回来，而不是靠模糊印象硬猜。

### 3. 事实层

这一层保存的是“稳定成立的东西”。

比如：

- 用户偏好中文、偏好直截了当
- 项目使用 pnpm 而不是 npm
- 某个仓库使用 monorepo
- 某类回答必须避免营销口吻

这层信息和事件层最大的区别，是它们更稳定、更适合长期保留，也更适合小规模注入到 prompt 里。

### 4. 过程层

这是最容易被忽视、但最能拉开差距的一层。

它保存的不是“发生过什么”，也不是“什么是真的”，而是“以后类似问题怎么做更稳”。也就是：

- 一套排障流程
- 一个选题分析套路
- 某工具的可靠调用模式
- 某类任务的最佳工作顺序

这也是为什么 2026 年很多 Agent 产品开始重新强调 `skills`。因为真正的能力积累，往往不是记住更多事实，而是把成功路径固化成可复用流程。

## 三、实现时最核心的不是存储，而是写入链路

多数系统失败，不是因为“检索不出来”，而是因为一开始就“写错了东西”。

所以落地实现时，最该认真设计的是写入链路。

一个可用的写入链路，通常应该长这样：

`原始事件 -> 重要性判断 -> 分类路由 -> 持久化 -> 后台固化`

下面逐段拆开。

### 第一步：先记录原始事件，但不要把它直接当长期记忆

无论是用户消息、工具调用、网页读取、文件修改、错误日志，第一步都应该先进原始事件流。因为你需要一个可以回放的 source of truth。

这层最适合保存成结构化事件，而不是只留自然语言段落。常见字段会有：

- 时间
- 会话 ID
- 用户 ID
- 任务 ID
- 事件类型
- 输入摘要
- 输出摘要
- 是否成功
- 关联文件或工具

为什么这一步很重要？

因为你后面无论做压缩、抽取、冲突处理还是评测，都需要有原始依据。没有原始事件，你很快就会陷入“记忆到底是怎么来的”这类不可解释状态。

### 第二步：做重要性判断，而不是见啥存啥

真正的工程分水岭就在这里。

并不是每句话都该入长期记忆。很多信息只对当前会话有效，过了这个任务周期就应该自然消失。

实践上可以把写入判断拆成三个问题：

- 这条信息是不是跨会话仍然有价值？
- 这条信息是不是以后大概率会复用？
- 这条信息是不是值得占用 prompt 常驻空间？

如果三个问题都是否，那它最多进入事件层，不该进入事实层。

如果它是稳定偏好、环境事实、长期约束，就可以考虑写进事实层。

如果它是某次复杂任务中验证过的成功流程，就更适合进入过程层。

这里最常见的错误，是把“临时状态”写成“长期偏好”。例如用户某次说“这次先简单写”，系统就把“用户偏好简单写”永久化。这样做出来的记忆系统会越来越自信地误解用户。

### 第三步：把写入分成同步写入和异步固化

不是所有东西都该在主链路里写。

比较合理的做法是分成两类：

- 同步写入：必须立刻落盘的高价值信息
- 异步固化：先放入事件流，等后台任务再整理的候选信息

同步写入通常包括：

- 明确的用户偏好
- 环境事实变更
- 高代价错误和根因
- 关键任务状态快照

异步固化通常包括：

- 长对话中的中间尝试
- 工具调用轨迹
- 大段网页或文档阅读结果
- 低置信度的行为推断

这样做的好处是，主链路轻，系统更稳；同时你又没有丢掉后续整理的素材。

### 第四步：在后台做 consolidation，而不是把整理工作都压在对话当下

成熟系统几乎都会有一个后台固化阶段。

这个阶段可以理解成“夜间整理”，即使它不一定真的在夜里执行。它的任务通常包括：

- 把长对话压缩成高密度摘要
- 从事件流中抽取稳定事实
- 识别重复出现的用户偏好
- 识别值得沉淀为 skill 的成功流程
- 去重、合并、过期清理

这一步为什么重要？

因为对话时延要求高，而记忆整理是重活。你不应该在用户等响应的那几秒里，让系统又抽取事实、又做冲突消解、又写知识图谱、又更新技能索引。这样系统会又慢又脆。

## 四、召回链路比写入链路更容易被做坏

很多人以为只要存得好，后面自然能拿出来。实际上，召回链路往往才是用户真正感受到“像不像真记得”的地方。

一个实用的召回链路，建议至少拆成四步：

`任务判定 -> 候选召回 -> 重排过滤 -> 上下文打包`

### 第一步：先判断当前任务需要哪类记忆

不是每轮都需要搜历史。

比如：

- 问“今天星期几”这种请求，不需要任何长期记忆
- 问“继续那个部署问题”，就需要事件层
- 问“按我平时风格写”，就需要事实层
- 问“像上次那样排查”，就需要过程层

如果每轮都无脑召回全部记忆，你得到的不是更强的 Agent，而是更高的成本和更混乱的 prompt。

所以第一步不是搜，而是判断“这次搜不搜、搜哪层、搜多少”。

### 第二步：候选召回最好是混合检索，不要只靠向量相似度

事件层适合时间、关键词、标签和语义一起用。

事实层适合基于键值、标签、最近更新时间、置信度来取。

过程层适合按任务类型、工具类型、依赖环境、成功率来找最相关的套路。

只靠向量相似度的典型问题是：

- 语义近，但时间上已经失效
- 表述近，但场景不同
- 文本像，但前置条件不成立

所以更稳的做法通常是“先粗召回，再带约束过滤”。例如先根据任务关键词和标签缩小范围，再做语义检索和重排。

### 第三步：重排阶段要解决两个问题，相关性和可信度

召回回来的候选项，不应该直接塞进上下文。

你至少要重排掉两类东西：

- 看起来相关、其实已过期的内容
- 看起来像经验、其实从未验证过的内容

一个实用的重排分数，通常会同时考虑：

- 语义相关性
- 时间衰减
- 来源可信度
- 是否成功验证过
- 是否与当前环境匹配

这也是为什么很多做得更深的记忆系统，会引入时间感知、版本信息和环境约束，而不是只做“Top-K 最像文本”。

### 第四步：最后不是“把文档贴进去”，而是“把状态拼进去”

上下文打包是最后一道关，也是最容易被忽略的一道关。

真正有效的做法不是把候选记忆原文大段贴回 prompt，而是按模型真正需要的形式组织出来。常见形式包括：

- 当前任务必须知道的稳定事实
- 和当前问题最相关的 2 到 3 条历史事件
- 一条已经验证过的处理流程
- 一个简短的“目前已知 / 已排除 / 下一步”状态摘要

换句话说，召回的终点不是原文，而是 **可行动状态**。

如果系统每次召回完都往 prompt 里堆三四千字历史，实际上是在把检索问题变回上下文膨胀问题。

## 五、记忆该放哪里，不同存储各有边界

Agent 记忆系统一旦开始实现，团队很快就会问：到底该用什么存？

这里最怕的是“单库崇拜”。现实里，没有一种存储能把所有记忆类型都优雅解决。

### 1. 文件系统

文件系统很适合保存：

- 工作笔记
- 状态快照
- 阶段总结
- 可读可改的记忆文件

它的优点是简单、透明、可审计、很适合和 Agent Runtime 融合。尤其是对 Coding Agent 来说，工作区文件本身就是记忆的一部分。

它的缺点是结构化查询弱，不适合做大规模召回和复杂过滤。

### 2. 关系型数据库

关系型数据库很适合保存：

- 结构化事件
- 用户偏好
- 记忆元数据
- 时间戳、标签、置信度、版本信息

它的优点是事务和过滤能力强，适合做系统主账本。缺点是对模糊语义召回不够强，通常要配合其他检索层。

### 3. 向量索引

向量索引最适合做的是：

- 历史会话的语义召回
- 长文材料的相关片段查找
- 模糊表述下的近似匹配

它不适合独立承担：

- 时序判断
- 冲突更新
- 强结构关系表达

很多团队把“记忆系统”直接等同于“向量库”，问题就出在这里。向量库是召回层，不是完整记忆层。

### 4. 图存储

图存储适合：

- 多实体关系
- 依赖传播
- 权限和组织结构
- 随时间演变的事实网络

它的优势在于关系可解释，但建模和维护成本更高。对多数团队来说，不应该上来就全图化，除非你的场景天然依赖关系推理。

### 5. 技能库

如果你的 Agent 需要持续复用成功工作流，那么单独做一层技能库非常有价值。

它可以是 Markdown、结构化 JSON，或者某种标准 skill 文件。关键不是格式，而是它存的是“怎么做”，而不是“发生过什么”。

这是很多系统从“会搜索”迈向“会积累能力”的关键一步。

## 六、实现中最容易被低估的三个模块

讲完主干架构，再说三个很少被认真写、但实际上决定系统上限的模块。

### 1. 冲突处理

新事实和旧事实冲突时，怎么办？

例如：

- 用户以前偏好英文，现在明确要求中文
- 项目以前用 npm，后来迁移到 pnpm
- 某 API 规则已经变了，但旧经验还在库里

这里不能简单“最新覆盖最旧”，也不能永远保留两个版本不处理。更合理的做法通常是：

- 记录版本和时间
- 标记来源
- 给置信度
- 保留必要的历史
- 对当前态做显式裁决

如果没有这套机制，你的记忆系统会越来越像一个堆满过期便签的冰箱门。

### 2. 压缩

压缩不是日志裁剪，而是状态提炼。

当上下文过长时，成熟系统不会只做简单截断，而是把历史工作重新组织成几个高密度块：

- 已完成什么
- 哪些路径已验证失败
- 当前剩余风险是什么
- 下一步最值得做什么

近来大家强调 `compaction`，本质上就是在解决这个问题。Agent 要长时间运行，必须学会把自己的过去压缩成未来还能用的状态。

### 3. 遗忘

没有遗忘机制，记忆系统迟早会坏。

低价值、过期、未验证、很久没命中的内容，应该逐步降权甚至删除。否则你每次召回都在和历史垃圾竞争。

一个比较务实的遗忘策略通常会综合：

- 时间衰减
- 命中次数
- 成功复用次数
- 来源可信度
- 是否已被新记忆覆盖

很多人害怕删记忆，但对工程系统来说，**不会遗忘，比会遗忘更危险。**

## 七、如果你现在就要做，一个最小可用实现应该怎么搭

不是每个团队都需要一步到位做成“全自动认知内核”。对大多数人来说，更现实的是先做一个最小可用版本。

下面给一套我认为很实用的 MVP 路线。

### 第一步：先做事件主账本

先别急着做 fancy memory。

第一步只做一件事：把用户消息、工具调用、关键结果、失败原因，统一记成结构化事件，放进 SQLite 或 Postgres。

这一层打好后，你至少有了：

- 可回放的历史
- 可评测的数据
- 后续抽取事实和技能的原料

### 第二步：加一个很小的事实记忆层

只保存少量高价值、稳定、跨会话的信息。

宁可少，不要脏。典型上限可以非常保守，例如一人几十条事实，而不是几千条。

如果你发现自己在往这里塞大量一次性内容，说明分层已经出了问题。

### 第三步：做按需召回，而不是默认全召回

让调度层先判断这轮是否需要历史，再去搜。

这一步会直接决定你的成本曲线。很多系统成本失控，不是因为模型太贵，而是因为根本没有“记忆选择器”。

### 第四步：把长历史压缩成状态摘要

任务超过一定长度后，不要继续把整段对话挂在上下文里。把它压成“当前状态卡片”，只保留高价值信息。

### 第五步：最后再做技能沉淀

只有当你看到某类任务已经重复出现，且系统确实能稳定做对，才值得把它固化成 skill 或 procedure。

否则你会过早地把噪音经验制度化。

## 八、不同场景下，最优实现并不一样

记忆系统没有单一标准答案，场景不同，最优实现差异非常大。

### 对话型助手

重点是偏好一致性、事实稳定性和低成本。通常不需要太重的过程层，更看重事实层和压缩。

### Coding Agent

重点是工作区状态、错误回放、工具示例、补丁历史和测试结果。这里文件系统和事件账本的重要性，通常高于大而全的向量库。

### 企业知识助手

重点是权限、时效、来源可信度和版本更新。这里关系型数据库、审计链路和权限边界非常关键。

### 自主任务型 Agent

重点是长任务恢复、状态快照、过程复用和失败反思。这里事件层、过程层和 compaction 会变得更核心。

所以真正成熟的做法不是背一个“最佳架构模板”，而是先判断：你的 Agent 更像助手、工具人、分析员，还是长期工作的执行体。

## 九、最后给一个判断标准：别问它记了多少，要问它下次是不是更会做事

判断一个 Agent 的记忆系统做得好不好，最不该问的问题是：

“它存了多少条记忆？”

更该问的是：

- 下次遇到类似问题，它是不是更快找到正确路径？
- 用户是不是更少重复背景信息？
- 长任务里它是不是更少迷路？
- 上下文更长后，它是不是还能保持稳定？
- 新经验是不是在增加能力，而不是增加噪音？

这背后其实对应着一条很重要的设计哲学：

**记忆系统的目标，不是让 Agent 背更多东西，而是让 Agent 在下一次任务里，更可靠地调用过去。**

如果只用一句话总结本文，那就是：

**Agent 记忆系统的实现重点，不在“存储了什么名词”，而在“你是否做对了写入、召回、压缩、冲突处理和遗忘”。**

只有这五件事站稳了，记忆才会从“概念能力”变成“工程能力”。

## 一个可直接执行的落地顺序

如果你接下来真要开工，我建议按下面的顺序做，而不是一上来就追求“大而全”：

1. 先做结构化事件流，保证一切可回放。
2. 再做一个很小的事实记忆层，只存高价值稳定信息。
3. 接着做按需召回和上下文打包，控制成本。
4. 然后补压缩、冲突处理和遗忘机制。
5. 最后才做技能沉淀，把成功流程变成过程记忆。

这样做的好处是，每一步都能单独验收，也能明确看到它是否真的提升了长任务成功率、偏好一致性和经验复用率。

从工程实践看，这比一开始就把“长期记忆”“知识图谱”“技能系统”全部铺开，更稳，也更容易做出真正有用的 Agent。
