# MiniMax M2.5 背后的机制：Forge 框架解析

> _本文基于 @neural_avb 的技术分析文章编译，深入解析 MiniMax M2.5 如何通过 Forge 框架解决大规模 RL 训练难题。_

最近 MiniMax M2.5 发布后引发了广泛关注。它不仅酷炫、快速、便宜，而且在编程方面表现出色。然而，关于它为何如此有效的文章并不多。它是如何训练出来的，以至于能与成本高出 10 倍的 Opus 模型相媲美？

MiniMax 团队已经发布了一篇极具技术深度的文章。但由于内容较为艰深，充斥着学术术语，本文旨在将其转化为更通俗易懂的版本。

## 核心问题 (The Main Problem)

他们解决的核心问题是：**如何让强化学习（RL）在大规模下有效运作，从而打造出优秀的 Agentic AI（代理式人工智能）。**

这不仅仅是关于训练 LLM 对查询生成单一回复，而是训练它们成为优秀的 Agent。优秀的 Agent 必须学会如何在执行任务时采取行动、使用任意工具、管理长上下文以及与复杂的环境进行交互。

训练 LLM 擅长 Agent 任务面临一系列挑战，MiniMax 将其称为“不可能三角”。

## 不可能三角 (The Impossible Triangle)

1.  **系统吞吐量 (System Throughput)**：需要快速处理海量的训练数据。
2.  **训练稳定性 (Training Stability)**：训练必须收敛，不能出现剧烈震荡。
3.  **Agent 灵活性 (Agent Flexibility)**：Agent 必须擅长各种不同的任务。

## “脚手架” (Scaffolds)

用户在各种各样的“脚手架”中运行 Agent——多 Agent 交互、记忆 Agent、代码沙箱等。模型需要在所有这些环境中都能良好运行。

**什么是脚手架？**
“脚手架是一个围绕基础语言模型的外部系统或包装器，定义了它如何作为 Agent 运作。”

**举例说明：**
Claude Opus 4.6 是一个训练好的 LLM。Claude Code 是一个用于编程的脚手架。Claude Code 负责编排规则、提示词（Prompts）、工具，并向 Opus 发起 API 请求。

## Forge 框架 (The Forge)

Forge 是 MiniMax 用于平衡“不可能三角”的 RL 框架。它试图解决“如何训练稳定的 Agent 以在各种脚手架中表现良好，并从海量训练数据中优雅地学习”的问题。

像大多数优秀的架构一样，Forge 也是一个三方架构：

1.  **Agent 端 (The Agent Side)**：决定做什么的外部代码：管理上下文、调用工具、链接推理步骤并生成轨迹。
2.  **中间件端 (Middleware Side)**：一个直通层，静默拦截所有 LLM 请求并将结果数据存储在数据池中。
3.  **引擎端 (Engine Side)**：包含 LLM 引擎和训练引擎。这是模型托管的地方，负责生成 Token 并执行权重更新。

**交互流程：**

- Agent 发起 LLM 调用。
- 调用通过中间件。
- 中间件将其路由到推理引擎进行生成。
- 同时，中间件静默地将所有内容收集到数据池中，供训练引擎稍后使用。
- LLM 引擎接收请求，生成 Token，并通过中间件发送回 Agent。
- 训练引擎从数据池中消费处理后的 Token 序列来更新策略。

**为何独特？**
特殊之处在于**中间件层**。其他框架通常只有 Agent 和训练/LLM 引擎。引入中间件和数据存储层，实际上将底层的训练-推理引擎与 Agent **解耦 (Decoupled)** 了。

**解耦的意义：**

- Agent 与推理引擎解耦（通过网关）。
- 生成与训练解耦（通过数据池）。
- Agent 逻辑与训练逻辑解耦（通过中间件）。
  Agent 层可以继续自行生成轨迹，而训练引擎则从数据池中异步获取数据进行训练。

## 训练策略

**如何确保模型学习多样化任务？**
他们直接使用广泛的脚手架阵列进行训练，而无需对 Agent 进行内部修改。这种方法有效地使模型能够在不同的脚手架（即环境）中泛化。

**CISPO 与混合训练：**
他们同时训练推理、问答（QA）和 Agent 任务，而不是一个接一个地训练，这避免了学习一个领域导致另一个领域性能下降的问题。
_注：CISPO 是 MiniMax M1 论文中介绍的一种 RL 技术。_

## 奖励机制 (Rewards)

RL 训练的奖励通常取决于环境（即环境根据模型表现分配奖励）。但 Forge 做了一些额外的技巧：

1.  **过程奖励 (Process Rewards)**：对中间步骤给予反馈——例如惩罚模型在轨迹中期进行畸形的工具调用。
2.  **完成时间奖励 (Completion Time Reward)**：模型不仅因为正确而获得奖励，还因为**快速**而获得奖励。这鼓励模型并行运行工具而不是串行运行，并高效地执行任务！这平衡了用户等待时间与模型质量。
3.  **Reward-to-go**：不再将相同的最终奖励分配给轨迹中的每个动作，而是每个动作获得从该点往后的奖励总和。这减少了噪声，帮助模型弄清楚哪些具体动作是关键的。

## Agent 层

Agent 本身不是 Forge 的可训练部分。它只是一个与脚手架/环境交互并通过中间件向 LLM 发送查询的通用 Agent。
Agent 提示 LLM 处理不同场景，并要求其生成轨迹。轨迹是该序列的完整记录：所有提示词、模型输出、选择的工具调用、返回的结果，以及最终任务是成功还是失败（奖励）。
LLM 本身是无状态的，Agent 提供提示/情境并维护其行动记录。

## 窗口化 FIFO (Windowed FIFO)

当 Agent 通过中间件发出请求时，数据池存储了所有关于 prompt-ids、completion-ids、rewards 等信息。训练器应该从这个缓冲区获取数据点进行训练。

有些请求通过一两次交互就早早结束了，但有些困难任务需要长时间的交互。LLM 在成功或失败之前会被调用多次。

为了解决这个问题，使用**滑动窗口 (Sliding Window)**：

- 在窗口内，任务可以乱序完成。
- 但窗口不会越过“掉队者”（Stragglers，即那些开始得早但尚未完成的任务，因为 Agent <-> LLM 的交互还在进行中）。

这平衡了“吞吐量”（每个周期处理的 Token 量）与“分布一致性”（简单任务 vs 困难任务）。

_（想象一个队列，训练器只能看到窗口内的任务。即使后面的任务完成了，如果窗口最前面的任务还没完成，窗口就不能向前滑动，确保模型不会只学习简单快速的任务。）_

## 前缀树合并 (Prefix Tree Merging)

在朴素的多轮对话中，消息是按顺序追加的。因此，许多训练样本共享很长的公共前缀。
与其冗余地独立处理每一个样本，不如将共享前缀合并成树结构，只计算一次。

例如，如果有三个样本包含相同的长前缀（蓝色部分）。朴素处理意味着蓝色部分要处理 3 次！这是计算浪费。
通过前缀合并，他们重用了蓝色上下文的计算结果来处理后续的序列。据报道，这在他们的测试中带来了 **40 倍** 的训练速度提升。

## 基于多 Token 预测的投机采样 (Speculative Decoding with MTP)

**投机采样 (Speculative Decoding)** 是 LLM 生成的一种加速技巧。通常 LLM 逐个生成 Token，这很慢。投机采样的思路是：使用一个更小、更快的“草稿”模型快速猜测接下来的几个 Token，然后让大模型在一次前向传递中验证它们。

**RL 训练中的问题**：
标准方法使用固定的草稿模型。但在 RL 训练期间，主模型的权重不断更新，行为不断变化。如果草稿模型保持不变，它的猜测会随着时间推移变得越来越不准确，接受率下降，加速效果消失。

**MTP (Multi-Token Prediction) Heads**：
MiniMax 在主模型上直接附加了额外的小型层（Heads）。主模型不仅预测下一个 Token，还同时预测接下来的 2、3 或更多个 Token。
因为这些 Heads 是模型的一部分，它们随着 RL 训练期间模型权重的更新而自然保持同步。他们还使用 Top-K KL 损失来微调这些 MTP Heads。

## 总结 (Takeaways)

1.  **解耦 (Decoupling) 是好的**：分离 Agent、推理和训练。
2.  **异步 (Async) 是好的**：提高效率。
3.  **混合环境 (Mixing Environments)**：能训练出多样化的模型。
4.  **中间奖励 (Intermediate Rewards)**：对长程任务至关重要。
5.  **中间件 (Middlewares)**：虽然是 60 年代的发明，但依然是神级架构。
6.  **DSA (DeepSeek Sparse Attention)**：依然有效。
