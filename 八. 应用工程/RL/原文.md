Forge: Scalable Agent RL Framework and Algorithm
Scaling RL for complex, real-world agents confronts a fundamental trilemma: balancing system throughput, training stability, and agent flexibility. These conflicting constraints have long impeded the...
Enough jazz, let's dance.

The Main Problem
The core problem they're solving is how to make RL work at massive scale for being good at Agentic AI.
It is is about training LLMs in a way so that they can become good agents.
Agents are not about generating one response from a query.
LLMs that are good at being agents must learn how to take actions, use arbitrary tools, manage long contexts, and interact with complex environments as it's performing tasks.
Training LLMs to be good at agentic tasks imposes a list of challenges. Minimax calls it the "Impossible Triangle".

The Impossible Triangle
System Throughput:
We need to process huge volumes of training data quickly
Training Stability:
Training should converge and not oscillate crazily
Agent Flexibility:
Diverse range of task where agentic AI is good at

"Scaffolds"
Users run agents in all kinds of scaffolds - multi-agent interactions, memory agents, coding sandboxes, etc. The model needs to work well across all of them.

"The Forge"
OOOOOOOHHHHHH THE FORGGEEEEE!
Forge is the RL framework Minimax uses to balance all 3 corners of the Impossible Triangle.

It tries to solve "how do you train stable agents to perform well in a diverse range of scaffolds and learn from huge volumes of training data cutely?"
Like most amazing architectures, Forge is also a 3-side architecture.
The Agent Side: The external code that decides what to do: manages context, calls tools, chains reasoning steps, and generate trajectories
Middleware Side: A passthrough that silently intercepts all LLM requests and stores the resulting data in a data pool.
Engine Side: Contains the LLM engine and the training engine. This is where the model is hosted, it generates tokens and performs weight updates too.
Q. How do they interact with each other?

- The agent makes LLM calls.
- The call passes through the middleware
- Middleware routes them to the inference engine for generation
- Also quietly, the Middleware collects everything into a data pool that the training engine can consume later.
- The LLM engine receives the requests, and generates tokens, sends it back to agent via middleware
- The training engine consumes processed token sequences from the data pool to update the policy.

Q. Why is it so special?
The special part actually comes from this middleware layer. Other frameworks use an agent and a training/LLM engine. Introducing this middleware and data storage layer basically decouples the underlying training-inference engine from the agent.

Q. Why is the train engine fetching from data pool? Why not just train it after the LLM finished generating?
This decouples the training of the LLM from the data generation by the agents. The Agent Layer can continue generating trajectories on its own.

Q. You talk about decoupling, but I dont understand what it means?
Decoupling basically means that we are seperating multiple concerns and solving them within their own modules.
Agent is decoupled from the inference engine (via the gateway)
Generation is decoupled from training (via the data pool)
Agent logic is decoupled from training logic (via the middleware)

Q. How do we ensure models are learning diverse tasks
They simply train using an extensive array of scaffolds without requiring internal modifications to the Agent. This approach effectively enables the model to generalize across diverse scaffolds, a.k.a. environments.
CISPO with mixed training: They train on reasoning, QA, and agent tasks all at once rather than one after another, which avoids the problem where learning one domain degrades performance on another.
CISPO is an RL technique introduced in the Minimax M1 paper: https://arxiv.org/abs/2506.13585

Q. What kind of rewards do the RL training happen?
Mostly RL training rewards depend on the environment. That is, the environment decides how rewards are assigned depending on how well the model is doing. However, they do some additional tricks during RL:
Process rewards: They give feedback on intermediate steps - like penalizing the model for making malformed tool calls mid-trajectory.
Completion time reward: The model is rewarded for being correct and for being fast, encouraging it to run tools in parallel rather than sequentially and be efficient in performing tasks! This balances the time users wait vs how good the model is.
Reward-to-go: Instead of assigning the same final reward to every action in the trajectory, each action gets the sum of rewards from that point forward. This reduces noise and helps the model figure out which specific actions mattered.

The Agent Layer
So here is the thing: the Agent is not a trainable part of Forge.

It is just a general agent that interacts with the scaffold/environment and sends queries to the LLM (through the middleware).
Basically the agent prompts the LLMs with different scenarios, and asks it to generate trajectories. A trajectory is then the full record of that sequence: all the prompts, the model's outputs, the tool calls it chose, the results that came back, and ultimately whether the task succeeded or failed (the reward).

The LLM itself is stateless, the Agent provides it prompts/situations and maintains a record of it's actions.

The Agent Layer is evaluated with both white-box and black-box architectures.

Q. Ahh so Forge is an Agentic AI product?  
"No, it's a framework to train LLMs. It:
Takes a base model
Throw it into an enormous variety of agent scaffolds and environments during RL training
The model learns to perform well across all of them
Ship that single model (as M2.5)
It is a training recipe that can be used to both create agentic products (Minimax-Agent) and use it in other scaffolds (like Claude Code)"

Windowed FIFO
"There are sliding windows..."
As the Agent makes requests through the middleware, the data-pool stores all the information about the prompt-ids, completion-ids, rewards, etc. The trainer is supposed to grab data points from this buffer and train. But remember that the agent actually sends requests from a diverse set of scaffolds.
Some requests end early with one or two back-and-forth interactions between agent and the LLM, but some harder tasks require long-interactions. The LLM gets called multiple times before succeeding or failing at the task.
To address this basically use a sliding window — within the window, tasks can complete out of order, but the window won't advance past "stragglers" (jobs that started early but havent finished, because Agent <-> LLM interaction is still going on).
This balances "throughput" (amount of tokens processed per cycle) with "distributional consistency" (easy vs difficult tasks).

You have a batch of 8 tasks (numbered 0–7) submitted for generation. The window size is 4, so the trainer can only see tasks 0, 1, 2, 3. Tasks 4–7 exist but are invisible to the trainer right now, even if some of them finish quickly.

Bottom picture:
Time has passed. Tasks 0 through 10 have all been consumed by training except task 7, which is a "straggler" - maybe a complex slow task. So the window is now anchored at 7, and the trainer can see 7, 11, 12, 13 (window size 4). Tasks beyond 13 are invisible.

"... merging"
In naive multi-turn dialogues, messages are sequentially appended. Therefore, many training samples share long common prefixes. Instead of redundantly processing each one independently, they merge shared prefixes into a tree structure, computing them only once.
In above picture, we got three examples containing same long prefix (blue). Processing naively will mean we process each of these cases independently. This means we process the blue section 3 times! More compute!
In the below section (prefix merging), they reuse the computation already done for the blue context, for processing seq2 and seq 3.
This yields a reported 40x training speedup on their harness.
Speculative decoding with Multi-Token Prediction
Q. This is too much.
"Persevere"
Speculative Decoding is a speed trick for LLM generation. Normally, an LLM generates one token at a time, which is slow because each token requires a full forward pass through the model.

The idea is: use a smaller, faster "draft" model to quickly guess the next several tokens, then have the big model verify them all at once in a single forward pass. If the guesses are right (which they often are), you just saved several sequential steps. All these are known concepts, not introduced here.
The problem during RL training: The standard approach uses a fixed draft model. But during RL, the main model's behavior is constantly changing as its weights get updated. The draft model stays the same, so its guesses become increasingly wrong over time. The acceptance rate drops and the speedup disappears.
MTP heads (Multi-Token Prediction heads) are small additional layers attached directly to the main model. Instead of a separate draft model, the main model itself has extra output heads that predict not just the next token but the next 2, 3, or more tokens simultaneously. Because these heads are part of the model, they naturally stay in sync as the model's weights update during RL training. They also use Top-K KL loss to fine-tune these MTP heads.
