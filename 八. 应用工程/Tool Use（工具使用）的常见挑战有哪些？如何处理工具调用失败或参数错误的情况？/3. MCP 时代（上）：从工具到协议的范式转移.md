# MCP 时代（上）：从工具到协议的范式转移

在上一篇文章中，我们讨论了如何设计一个健壮的 Agent Tool。然而，随着 **Model Context Protocol (MCP)** 的横空出世，整个 Tool 开发的游戏规则正在发生剧变。

很多开发者会有疑问：有了 MCP，是不是以后都不用写 Tool 了？
答案是：**Tool 依然要写，但写法从“定制裁缝”变成了“标准件制造”。**

本文将深度拆解 MCP 时代下，工具开发模式的**第一个核心变化**：架构解耦与资源革命。

---

## 变化一：架构解耦——从“内置插件”到“独立服务”

### 以前：Agent 与 Tool 是连体婴

在 MCP 出现之前，工具通常是硬编码在 Agent 项目里的。

- 如果你用 Python 写 LangChain Agent，你就得写一个 Python 函数。
- 如果你用 Node.js 写 AutoGen Agent，你就得重写一遍 JS 函数。
- **痛点**：复用性极差。你为公司内部 ERP 系统写的查询工具，没法直接给 Cursor 或 Claude Desktop 用。

### 现在：Tool 是独立的远程资源 (Remote Resources)

MCP 引入了 **Client-Host-Server** 架构。你写的不再是一个简单的函数，而是一个标准的 **MCP Server**。

- **标准化**：它通过 JSON-RPC 协议与外界沟通。
- **跨平台**：一旦你把 ERP 系统封装成了一个 MCP Server，那么：
  - 你的 Python 脚本可以连它。
  - Claude Desktop App 可以连它。
  - IDE (如 Cursor, VS Code) 也可以连它。
- **开发思维转变**：你不再是为某个 Agent 写插件，而是在为你的数据源开发一个**通用的 AI 驱动程序**。

---

## 变化二（Part 1）：Resources——直接的数据管道

在传统开发中，我们把所有能力都塞进 `Tools` 里。但在 MCP 协议中，能力被大幅扩展。其中最重要的一个新原语就是 **Resources（资源）**。

### Passive Context Injection（被动上下文注入）

以前，如果你想让 AI 监控一个日志文件，Agent 必须每隔几秒调用一次 `read_log()` 工具（轮询）。这既浪费 Token，又反应迟钝。

现在，MCP 引入了 **Resources**。它定义了一套标准的 URI（如 `postgres://db/users/schema`）。

- **被动注入（Passive）**：用户可以直接把这个 Resource “贴”到聊天窗口里。MCP Client 会自动读取它的最新内容，并将其作为 System Prompt 的一部分（或 Context）直接喂给模型。
- **实时更新**：如果 Resource 的内容变了（比如日志有了新的一行），MCP Server 可以通知 Client 更新 Context。
- **优势**：Agent 不需要“思考”去调用什么查询函数，数据就像“背景知识”一样，天然地存在于它的上下文里。它只需要专注于处理逻辑，而不是搬运数据。
