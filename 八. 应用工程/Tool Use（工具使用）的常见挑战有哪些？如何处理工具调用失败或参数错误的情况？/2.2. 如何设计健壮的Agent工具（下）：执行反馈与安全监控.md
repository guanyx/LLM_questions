# 给 AI 用的工具怎么造？（下）—— 执行反馈与安全监控

*(承接上篇：在完成了接口设计与输入防御后，我们进入工具的运行时阶段)*

当工具被 Agent 成功调用后，真正的挑战才刚刚开始。如何处理执行结果？如何防止 Agent 造成破坏？如何监控 Agent 的行为？

本篇将深入探讨工具设计的**后三个关键步骤**：执行反馈、安全兜底与可观测性。

---

## 第三步：执行与反馈（Execution & Feedback）—— 报错也是对话

当工具执行完毕（无论成功还是失败），返回给模型的内容至关重要。

### 1. 失败时的“建设性报错”

不要返回 `KeyError: 'data'` 或 `500 Internal Server Error`。模型看不懂 traceback。
你要把异常捕获住，翻译成“人话”（其实是“模型话”）：

- ❌ `Error: list index out of range`
- ✅ `Error: No order found with ID 12345. Please verify the customer_id and try again.`

### 2. 成功时的“信噪比控制”

如果工具查到了 100 条订单，直接把 100 条 JSON 全部扔回给 LLM？

- **后果**：Context Window 瞬间爆炸，模型注意力被稀释，费用飙升。
- **Robust 做法**：
  - **截断**：只返回前 5 条，“...and 95 more items”。
  - **摘要**：在工具内部做一次简单统计，返回“Found 100 orders. The most recent one is...”。
  - **字段过滤**：只返回模型真正需要的字段（如 id, status），过滤掉无关的元数据。

---

## 第四步：安全兜底（Safety Net）—— 别让 AI 删库跑路

### 1. 只读与副作用分离

将工具分为 **Read-Only（只读）** 和 **Side-Effect（副作用）** 两类。

- 只读工具（查询、搜索）可以让 Agent 自动执行。
- 副作用工具（删除、转账、发邮件）必须引入 **Human-in-the-loop**（人工确认）。

### 2. Dry Run（预演模式）

对于高风险工具，设计一个 `dry_run` 参数。

- 当 `dry_run=True` 时，工具不真正执行操作，而是返回“If executed, this would delete file X”。
- Agent 可以先调用 Dry Run，把结果展示给用户确认，用户点头后再调用 `dry_run=False`。

---

## 第五步：可观测性（Observability）—— 打开 Agent 的黑盒

不要让 Agent 在黑盒里盲跑。你需要知道它到底调用了什么、耗时多久、为什么失败。

### 1. 链路追踪（Tracing）

使用 LangSmith、Arize Phoenix 或 LangFuse 等现代工具，把每一次工具调用的 Input、Output、Latency 和 Token Usage 记录下来。

- **Debug 价值**：当 Agent 表现不佳时，你可以回放它的思考过程，发现“哦，原来它卡在了这一步”。

### 2. 语义级监控

仅仅监控 `HTTP 200` 是不够的。Agent 可能成功调用了工具，但因为工具返回了“无结果”，导致任务失败。
你需要监控的是**语义层面的成功率**。

- 比如：工具调用成功率 vs 任务完成率。

---

## 结语：好的工具是 Agent 的外骨骼

构建 Robust Tool 不是代码技术问题，而是**交互设计问题**。

你需要时刻把自己代入 Agent 的视角：

- “这个函数名我能看懂吗？”
- “如果我参数填错了，你会告诉我正确答案吗？”
- “结果太长了，我记得住吗？”

当你的工具能够像一位耐心的导师一样，引导模型输入正确参数、包容它的细微错误、并给出清晰的反馈时，你的 Agent 就会表现出惊人的智能与稳定性。
