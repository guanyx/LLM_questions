# 给 AI 用的工具怎么造？（上）—— 接口设计与输入防御

*(承接前文：我们讨论了 Agent 如何使用工具，现在视角反转，讨论如何为 Agent 设计工具)*

很多开发者在设计 Agent 工具时，往往会犯一个习惯性错误：**把给人类用的 API 直接扔给 AI。**
人类程序员会看文档、会调试、会理解隐喻，但 LLM（大语言模型）本质上是一个概率预测器。它不看文档，它只看 Prompt；它不调试，它只尝试。

因此，**Robust Tool（健壮工具）的设计哲学**，核心在于：**降低模型的认知负荷，提升系统的容错能力。**

本篇将聚焦于工具设计的**前两个关键步骤**：接口定义与输入防御。

---

## 第一步：接口定义（The Interface）—— 命名即 Prompt

对于 LLM 来说，函数名和参数名不仅仅是符号，它们本身就是 Prompt 的一部分。

### 1. 拒绝“程序员式”缩写

人类喜欢简洁，但 LLM 喜欢语义。

- ❌ **Bad:** `get_data(id, type)`
  - 模型困惑：什么 data？什么 id？type 是字符串还是枚举？
- ✅ **Good:** `search_customer_orders(customer_id, order_status)`
  - 模型秒懂：这是查订单的，需要客户 ID 和订单状态。

### 2. Description 是“操作手册”而非“功能简介”

很多工具的 description 只有一句 "Search orders"。这是远远不够的。
优秀的 description 应该包含：

- **功能边界**：能做什么，*不能*做什么。
- **参数格式**：比如日期必须是 `YYYY-MM-DD`。
- **返回值预期**：会返回 JSON 还是纯文本？如果没有结果会返回什么？

> **实战技巧**：在 Description 里写几个 Example（Few-Shot），能显著提高调用成功率。例如：“Example: customer_id='12345', status='SHIPPED'”。

---

## 第二步：输入防御（Input Defense）—— 宽进严出

永远不要相信 LLM 传进来的参数。它可能会传错类型、甚至传一个不存在的参数。

### 1. 优先使用 Native Structured Output

现在的主流模型（如 OpenAI GPT-4o, Claude 3.5）都原生支持了 Structured Outputs（JSON Schema 约束）。

- **过时做法**：让模型输出一段 JSON 字符串，然后用正则表达式去扣，或者用 Pydantic 去 parse（如果失败了再重试）。
- **前沿做法**：直接在 API 调用层开启 `response_format: { type: "json_object" }` 或更严格的 Schema 约束。这相当于在模型生成的“喉咙”处就卡住了非法格式，从概率上杜绝了类型错误。
- **Pydantic 的角色**：它不再是“事后诸葛亮”的校验器，而是“事前诸葛亮”的 Schema 生成器。

### 2. 模糊输入的“归一化”

模型对格式很不敏感。

- **场景**：你要求日期格式 `YYYY-MM-DD`。
- **模型行为**：它可能传 `"2023/01/01"` 或者 `"Jan 1, 2023"`。
- **Robust 做法**：不要直接报错！在工具内部引入一个轻量级的解析层（如 `dateparser`），尝试把各种怪异的格式统一转成标准格式。只有当真的无法解析时，才抛出错误。

### 3. 枚举（Enum）优于字符串

如果你需要模型选择“状态”，千万不要用自由文本。

- ❌ `status: str` -> 模型可能填 "finished", "done", "Completed"。
- ✅ `status: Enum['PENDING', 'SHIPPED', 'DELIVERED']` -> 模型只能三选一，出错率几乎为零。
