# 如何处理工具调用失败？构建反脆弱的 Agent 系统

*(承接上篇：Tool Use 的核心冲突与三大“死亡谷”)*

面对 Tool Use 中的各种挑战，一个成熟的 Agent 系统不能指望模型“永远正确”，而是要设计一套机制，让它在犯错时能**自我修正**。这不仅仅是技术问题，更是系统设计哲学。

---

## 一、 破局之道：如何处理失败与错误

### 1. 从“报错”到“反思”（Self-Correction）

传统程序的逻辑是：遇到 Error -> 抛出异常 -> 程序崩溃。
Agent 的逻辑应该是：**遇到 Error -> 这是一个新的观察（Observation） -> 思考原因 -> 重新尝试。**

- **错误即反馈**：当工具调用失败时，不要对用户说“系统错误”，而是把错误的详细信息（比如 `Invalid Date Format`）作为一条新的 Prompt 喂回给模型。
- **自我修正循环**：模型看到错误信息后，会意识到：“哦，原来这个字段不能填字符串，应该填数字。”然后它会生成一个新的、修正后的调用请求。
- **机制**：这就像教小孩子拼图，拼不进去时，告诉他“形状不对”，他自己会换个方向再试，而不是直接大哭跑开。

### 2. 前置校验与“受限解码”（Guardrails & Constrained Decoding）

不要完全信任 LLM 生成的参数。在请求真正发给 API 之前，设立一道“安检门”。

- **受限解码（Constrained Decoding）**：这是一种更底层的技术。在模型生成 Token 的时候，就强制它只能输出符合正则表达式或 JSON Schema 的字符。如果模型想输出一个非法字符，系统直接把那个 Token 的概率抹零。这从根源上杜绝了语法错误。
- **业务规则拦截**：如果参数只有 A、B、C 三个选项，而模型填了 D，直接在代码层面驳回，不给 API 制造麻烦。

### 3. 承认无知与人机协作（Human-in-the-loop）

当模型发现参数缺失，或者不确定该用哪个工具时，最优雅的处理方式不是“硬猜”，而是**反问用户**。

- **场景**：用户说“帮我订票”。
- **错误做法**：Agent 直接去订一张它认为最可能的票。
- **正确做法**：Agent 调用一个特殊的“提问工具”，回复用户：“请问您要去哪里？出发时间是？”
- **核心逻辑**：把“向用户提问”也视为一种工具能力（Clarification Tool）。

## 二、 进阶思考：从“孤岛工具”到“通用协议”

如果我们把视线拉高，Tool Use 的进化正在经历三个阶段：

1.  **静态工具（Static Tools）**：
    也就是现在的主流模式。开发者写死 API 文档，模型照着文档填空。这像是在**填表格**。

2.  **通用协议互联（MCP - Model Context Protocol）**：
    这是当下的前沿趋势。以前每个 App 的工具都是孤岛，现在通过标准化的协议（如 Anthropic 提出的 MCP），让 Agent 可以即插即用地连接本地文件、GitHub、Slack 等任何数据源，而无需为每个工具单独写适配代码。这像是在**插 USB 设备**。

3.  **计算机使用（Computer Use / GUI Agents）**：
    未来，也许我们不需要强迫 LLM 去适应程序员的 API。最新的模型（如 Claude 3.5 Computer Use）已经学会了像人一样看屏幕、移动鼠标、点击按钮。这意味着所有为人类设计的软件，瞬间都变成了 Agent 的工具。这才是真正的**数字实体（Digital Entity）**。

---

## 结语

Tool Use 的本质，是**确定性的逻辑世界**向**概率性的语义世界**的一次妥协与握手。

在这个过程中，错误不仅是不可避免的，甚至是有价值的。一个优秀的 Agent 系统，不是从不犯错，而是拥有强大的**鲁棒性（Robustness）**——它能在参数错误中学会规则，在调用失败中寻找替代方案，最终像一个真正的智能体一样，在充满不确定性的环境中，顽强地达成目标。
