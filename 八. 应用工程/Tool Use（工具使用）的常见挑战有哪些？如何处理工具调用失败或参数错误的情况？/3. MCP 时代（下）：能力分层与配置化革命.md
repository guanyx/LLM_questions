# MCP 时代（下）：能力分层与配置化革命

_(承接上篇：我们讨论了 MCP 带来的架构解耦与 Resources 革命，现在我们将深入探讨能力分层的其他方面以及连接方式的变革)_

MCP 不仅仅是把 Tool 变成了 Server，它还重新定义了 Agent 与工具交互的全部可能性。

---

## 变化二（Part 2）：能力分层——不仅仅是 Tool

除了 Resources，MCP 还引入了 Prompts 和 Sampling，以及保留了传统的 Tools，构成了完整的能力矩阵。

### 1. Prompts：预设的交互模板

以前，Prompt Engineering 是写在 Agent 端的。
现在，MCP Server 可以自带 Prompts。

- **场景**：你的 MCP Server 是一个代码审查工具。你可以内置一个名为 `code-review` 的 Prompt 模板。
- **优势**：服务端不仅提供能力（Tool），还教客户端如何使用这些能力（Prompt）。这让工具的**可解释性**和**易用性**大幅提升。

### 2. Sampling：反向控制（Human-in-the-loop）

这是一个极其前沿的变化。MCP 允许 Server 请求 Client 对模型进行“采样”（Sampling）。

- **场景**：Agent 在执行一个危险操作前，MCP Server 可以主动弹窗，要求 Client（比如 Claude Desktop）展示一个确认框，或者要求用户输入更多信息。
- **意义**：这让 Tool 不再是被动的“受气包”，它拥有了与用户直接对话的能力，极大地增强了安全性与交互性。

### 3. Tools：依然是执行者

当然，对于需要副作用（Side Effects）的操作，比如“重启服务器”、“发送邮件”，依然通过 Tools 来实现。但因为有了 Resources 的分担，Tools 可以更专注于**“做动作”**，而不是**“搬运数据”**。

---

## 变化三：连接方式——从“代码引用”到“配置驱动”

### 以前：Import 也是一种负担

你需要手动 `import my_tool`，然后把它加到 `tools=[...]` 列表里。每次增加新工具，都得改 Agent 的代码并重新部署。

### 现在：即插即用的配置化

MCP Server 通常是一个独立的进程（比如一个 Docker 容器或本地命令行）。
Agent 连接工具，只需要一行配置：

```json
"mcpServers": {
  "my-erp": {
    "command": "uvx",
    "args": ["mcp-server-erp"]
  }
}
```

- **动态扩展**：Agent 可以在运行时动态发现新的 MCP Server，而无需重启或修改核心代码。
- **权限隔离**：因为 Tool 跑在独立的进程里，你可以通过系统级的权限控制（如沙箱），防止 AI 做出越权操作。

---

## 结语：从“造轮子”到“造标准”

MCP 并没有消灭 Tool 开发，它消灭的是**重复的适配工作**。

在 MCP 时代，作为开发者，你的目标不再是“帮这个 Agent 写个工具”，而是“为这个业务系统写一个 MCP Server”。
当你完成了这个 Server，你不仅服务了当下的 Agent，也为未来所有兼容 MCP 的 AI 实体（无论是 GUI Agent 还是 CLI Agent）铺平了道路。

**这，才是 AI 基础设施真正的样子。**
