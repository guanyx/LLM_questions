# work.md 是如何被抽象成可工作内容的

## 一、先说结论

`work.md` 不是“这个同事擅长什么”的简历摘要，而是“当模型被要求代替他干活时，应该遵守哪些工作规则”的操作说明书。

它之所以能变成可工作的内容，不是因为模型真的学会了这个人的能力，而是因为项目先把原材料拆成一组稳定的工作维度，再把这些维度重写成可执行的约束。最终，模型拿到的不是散乱聊天记录，而是一份结构化工作协议。

## 二、work.md 抽象的输入是什么

这一层的输入主要来自三类材料：

- 他主动写的技术文档、方案、接口说明、runbook
- 他在评审、故障处理、需求讨论中留下的文字记录
- 用户手工补充的基础身份信息，比如公司、职级、岗位

这里最关键的一点是：项目并不直接问“他厉不厉害”，而是只看能落到工作行为上的证据。也就是说，只有那些能回答“他平时怎么做事”的材料，才有资格进入 `work.md`。

## 三、第一步：把原材料压成固定维度

在分析阶段，项目会先把原材料映射到几个固定槽位里，而不是直接自由总结。核心槽位包括：

- 职责范围：他负责哪些系统、模块、业务线
- 技术规范：他默认使用什么技术栈、接口规范、命名习惯
- 工作流程：他接需求、写方案、做 CR、处理线上问题时的步骤
- 输出风格：他写文档是结论前置还是背景前置，喜欢列表还是长文
- 经验知识库：他明确说过的原则、踩坑结论、重复出现的技术判断

这一层非常像把非结构化语料抽成表单。项目真正想做的，不是“模仿一个人”，而是“把一个人的做事约束变成一组可复用字段”。

## 四、第二步：按岗位重排重点

`work.md` 的抽象不是一套模板打天下，而是会根据岗位类型改变提取重点。

比如后端工程师会更强调：

- 接口设计
- 错误码、分页、幂等
- 数据库和事务边界
- Code Review 的典型阻塞点
- 监控、发布、线上排查流程

如果换成其他岗位，关注点会切到各自的专项维度：

- 前端工程师
  - 技术栈、状态管理、样式方案
  - 组件拆分原则
  - 首屏、懒加载、包体积等性能关注点
  - 接口调用和错误处理方式
  - ESLint、Prettier、测试覆盖、可访问性、响应式、兼容性
- 算法工程师 / ML 工程师
  - 问题定义方式
  - 实验设计习惯、基线选择、ablation 设计
  - 离线指标与在线指标的权衡
  - 常用模型、训练框架、数据处理规范
  - 模型上线流程、实验报告写法、常引用的方法论
- 产品经理 / 技术产品经理
  - PRD 结构和详细程度
  - 用户故事与需求边界定义方式
  - 与研发对齐的评审方式和修改流程
  - 优先级排序框架
  - 数据驱动与直觉判断的比例
  - 原型、埋点、竞品分析等输出物偏好
- 设计师
  - 设计系统和组件库使用习惯
  - 标注方式与交付规范
  - 对 pixel-perfect 的要求程度
  - 从需求到方案的推进步骤
  - 走查、验收、处理开发还原度问题的方式
- 数据分析师
  - 漏斗、同期群、A/B 测试等分析框架
  - SQL 写法风格
  - 图表类型和可视化偏好
  - 报告里结论和数据的比例
  - 处理数据异常和口径争议的方式

这样做的意义在于，`work.md` 不是一份通用的“优秀员工画像”，而是一个和岗位绑定的能力配置文件。

## 五、第三步：只保留可执行表达

这是 `work.md` 能真正“工作”的关键。

项目不会满足于“他很重视代码质量”“他比较严谨”这种形容词，因为这种描述对模型没有直接约束力。它会尽量把结论改写成可以执行的规则，并且会按不同岗位落成不同类型的约束。

如果是后端工程师，常见的可执行表达会包括：

- 接口统一返回 `{code, message, data}`
- 分页接口必须明确页码、页大小和总数语义
- 涉及重复提交的写操作默认要求幂等
- 事务边界要显式控制，不能把整条调用链无边界包进事务
- 做 Code Review 时优先检查 N+1 查询、事务边界、并发安全
- 处理线上问题先看监控，再缩小变更范围，再定位异常链路
- 数据库操作先看索引和扫描范围，再讨论 ORM 写法是否优雅

如果是前端工程师，常见的可执行表达会包括：

- 超过一定复杂度的页面逻辑必须拆成组件和 hooks，而不是继续堆在单文件里
- 首屏关键内容优先加载，非关键模块延迟加载
- 接口错误必须给出明确用户反馈，不能直接静默吞掉
- 状态归属要清晰，本地状态、全局状态、服务端状态不要混用
- 新组件默认先检查可访问性、响应式和兼容性
- 提交前必须过 lint 和基础测试，不能把格式化问题留到 CR

如果是算法工程师 / ML 工程师，常见的可执行表达会包括：

- 新方案必须先给 baseline，再讨论更复杂模型
- 实验报告要同时写清数据集、指标、对照组和 ablation
- 离线指标提升不足以直接推动上线，必须说明和在线目标的关系
- 训练数据处理流程要可复现，不能靠手工临时清洗
- 模型上线前先确认特征一致性、推理延迟和回滚方案
- 讨论模型效果时，先区分数据问题、特征问题和模型问题

如果是产品经理 / 技术产品经理，常见的可执行表达会包括：

- PRD 必须先写目标、范围、非目标，再展开细节
- 需求评审前先对齐边界条件和异常流程，不能只讲主链路
- 优先级判断先看目标贡献，再看实现成本和依赖关系
- 需求变更必须记录原因和影响范围，不能口头改完就算
- 埋点需求要和主需求一起定义，不能等上线前补
- 原型、文案、状态流转和验收口径必须在评审前准备齐

如果是设计师，常见的可执行表达会包括：

- 新页面先复用设计系统组件，不从零画一套
- 交付稿必须附状态说明、尺寸标注和交互说明
- 视觉稿不仅给正常态，还要给 loading、空态、错误态
- 验收时先看结构和交互，再抠像素级还原
- 发现开发还原偏差时，先判断是设计表达不清还是实现偏差
- 组件级改动要同步检查整套设计系统的一致性

如果是数据分析师，常见的可执行表达会包括：

- 分析问题前先统一口径、时间窗口和人群范围
- 报告先给结论，再给支撑数据和拆解过程
- SQL 查询默认先校验去重逻辑、关联条件和时间过滤
- 异常数据必须先判断采集问题还是业务问题，不能直接下结论
- 做 A/B 分析时先确认实验分流、样本量和显著性假设
- 图表选择要服务于结论表达，不能为了“好看”堆复杂图形

一旦写成这种格式，模型拿到的就不再是印象，而是流程和规范。

## 六、第四步：把经验结论变成可调用知识

除了流程和规范，`work.md` 还会保留“经验知识库”这一层。

这层的作用不是让模型显得懂很多，而是让它在回答问题时优先复用这个同事已经反复表达过的判断。例如某人长期坚持“先保证幂等，再谈性能优化”，或者“函数职责不单一就要拆”，这些都能变成默认判断基线。

从 Agent 工程角度看，这一步相当于把隐性经验压缩成低成本、可复现的策略缓存。

## 七、第五步：装配成工作能力说明书

在 builder 阶段，前面抽出来的槽位会被重新装配成 `work.md`。它的结构通常包括：

- 职责范围
- 技术规范
- 工作流程
- 输出风格
- 经验知识库
- 工作能力使用说明

最后这一节尤其重要，因为它明确告诉模型：当用户让你写代码、写文档、做 Code Review、处理需求、回答技术问题时，要分别优先调用哪一部分规则。

也就是说，`work.md` 不只是知识陈列，更像一份任务路由表。

## 八、为什么它最后会变成“可工作的”

因为这个项目做了三次压缩：

- 第一次压缩：把原始材料压成固定维度
- 第二次压缩：把维度结论改写成可执行规则
- 第三次压缩：把规则装配成任务可调用的能力说明书

所以 `work.md` 的本质，不是“关于某人的描述”，而是“替某人干活时应该遵循的工作协议”。

这也是为什么它看起来像知识总结，但实际作用更接近一个轻量级的执行规范层。

## 九、一句话总结

`work.md` 的抽象过程，本质上是把“这个人过去怎么工作”的证据，压缩成“模型现在应该怎么工作”的规则集合。
