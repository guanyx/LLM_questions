     
**结论**

- 这个项目的“蒸馏同事”本质上不是模型蒸馏，也不是微调一个新模型。
- 它做的是一套 **基于原始材料的结构化人格/工作能力提取 + Prompt 组装**。
- 最终产物不是权重文件，而是一个新的可调用 Skill：把“这个人怎么做事”和“这个人怎么说话”固化成可执行的提示词与规则。

**整体原理**

- 它把“同事”拆成两个正交部分：
- **Work Skill**：工作能力、技术规范、职责边界、流程、经验库
- **Persona**：性格、说话风格、决策模式、人际行为、雷区
- 运行时先让 Persona 决定“态度和口气”，再让 Work Skill 决定“怎么把事做出来”。

这其实是一个很典型的“人设层 + 能力层”双层代理设计。

**它是怎么运行的**

- 整个仓库本身就是一个元 Skill，入口是 `SKILL.md`
- 当你在 Claude Code 里执行 `/create-colleague` 时，它会按预设流程工作
- 流程大致是：
- 先收集基础信息：姓名、公司/职级/岗位、MBTI、标签、主观印象
- 再导入原材料：飞书、钉钉、Slack、邮件、PDF、截图、Markdown、直接粘贴
- 然后走两条分析链路：
- 一条提炼工作方法，生成 `work.md`
- 一条提炼人格行为，生成 `persona.md`
- 最后把两者拼成完整的 `SKILL.md`，放到 `colleagues/{slug}/` 下供后续调用

所以它不是“让模型记住一个人”，而是“替模型写出一份像这个人的操作手册”。

**“蒸馏同事”的核心机制**

- 这个项目最关键的不是采集，而是 **结构化约束**。
- 它不是简单把聊天记录扔给模型总结，而是先规定好要抽取哪些维度。

Work 侧重点是：
- 负责什么系统/模块
- 技术栈和代码规范
- 接口设计习惯
- Code Review 关注点
- 需求处理和线上故障处理流程
- 明确表达过的经验结论

Persona 侧重点是：
- 高频词、口头禅、黑话
- 句式长短、正式程度、emoji/标点习惯
- 决策优先级：效率、流程、数据、人情、政治
- 面对催进度、被质疑、背锅时的反应
- 对上/对下/对平级的行为差异
- 不能踩的红线和回避方式

也就是说，它把“一个人”分解成一组可观测特征，再把这些特征写回 Prompt。

**为什么它看起来像“真的像这个人”**

核心在三个点：

- **标签先验**
  - 用户手工输入的标签优先级高于文件分析
  - 比如“甩锅高手”“字节范”“完美主义”这类标签，不是停留在形容词，而是被翻译成具体行为规则
  - 例如“字节范”会变成：先问 context、先问 impact、强调对齐、直接打断信息不足的讨论
  - 例如“甩锅高手”会变成：先模糊责任边界、被追责先说需求没说清

- **原材料证据化**
  - prompt 明确要求“不要推断，有依据才写”
  - 没证据就标“原材料不足”
  - 有原话就尽量引用原话
  - 这让它不是纯脑补，而是尽量从文本中抽证据

- **分层控制**
  - Persona 不是一段笼统描述，而是分成 Layer 0 到 Layer 5
  - Layer 0 是硬规则，优先级最高
  - Layer 2/3/4 分别控制表达、决策、人际行为
  - Correction 层还能后续纠偏
  - 这相当于给“像不像本人”加了一个可维护的控制系统

**它并不是怎么“训练”的**

可以明确说：它没有做传统意义上的训练。

它没有看到这些东西：
- LoRA / SFT 微调
- 向量检索驱动的 RAG 主架构
- embedding 建索引后再召回回答
- 模型参数更新

它主要做的是：
- 采集原始语料
- 调脚本做格式清洗/解析
- 用分析 prompt 提炼结构化结论
- 用 builder prompt 生成最终 Skill 文本
- 以后交互时直接把这些文本当系统规则使用

所以更准确的说法是：

- 这是 **Prompt 蒸馏**
- 也是 **专家知识压缩**
- 还是一种 **人格/工作流模板化**

但不是机器学习意义上的“知识蒸馏”。

**数据采集层是怎么做的**

它支持两类输入：

- **自动采集**
  - 飞书：消息、文档、表格
  - 钉钉：文档、表格，消息部分靠浏览器方案补
  - Slack：通过 API
- **手动导入**
  - PDF、图片、飞书 JSON 导出、邮件、Markdown、纯文本

这里的设计思路很实用：
- 自动采集解决规模问题
- 手动导入解决权限和覆盖问题
- 两者混用提升还原度

尤其飞书这部分，它甚至把 OAuth、user token、chat_id、open_id 这些采集细节都写进了 Skill 的执行流程里，说明它不是停留在“概念 demo”，而是认真把企业 IM 数据接入当成一等公民。

**生成产物长什么样**

最终每个同事会有一个目录，大致包含：

- `work.md`
- `persona.md`
- `SKILL.md`
- `meta.json`
- `versions/`
- `knowledge/`

其中：
- `work.md` 是工作能力规范
- `persona.md` 是人格行为说明书
- `SKILL.md` 是最终可调用入口
- `meta.json` 存同事画像、版本、来源
- `versions/` 做版本快照
- `knowledge/` 存原材料

这说明它把“同事”当成一种可演化的软件资产，而不只是一次性生成文本。

**为什么要拆成 Work + Persona**

这是这个项目最聪明的地方。

如果只做 Persona：
- 会很像这个人说话
- 但不一定能按这个人的方法做事

如果只做 Work：
- 能复现工作规范
- 但回答会很像普通 AI，不像“他本人”

拆开后有几个好处：
- 能力和口气解耦
- 更新时更容易增量维护
- 冲突更容易定位
- 也能单独调用 `/slug-work` 或 `/slug-persona`

这是一种很工程化的设计，而不是“来一大段人设 Prompt”。

**它如何持续进化**

项目设计了两种进化方式：

- **追加文件**
  - 新文档/新消息进来后，先判断更新 work 还是 persona
  - 只做增量 merge，不覆盖原结论
  - 如果有冲突，提示用户决定
- **对话纠正**
  - 用户说“这不像他”“他不会这样”
  - 系统就把这句话转成 Correction 记录
  - 追加到对应层里，后续立即生效

这一步很重要，因为“蒸馏人”最难的是第一次不可能完全准确。
它没有试图一次生成完美人设，而是设计成 **可纠偏、可演化、可回滚** 的系统。

这其实比一次性生成更接近真实知识管理。

**它像什么，不像什么**

它更像：
- 给某个同事写一份超细的“AI 代理配置文件”
- 把 tacit knowledge 显式化
- 把工作习惯和说话习惯编码成 Prompt 规则

它不像：
- 数学意义上的知识蒸馏
- 一个真正拥有长期记忆和自我更新能力的独立智能体
- 一个从原始数据端到端学出人格的训练系统

**它成功的关键前提**

项目里其实已经点明了：原材料质量决定 Skill 质量。

优先级大致是：
- 他主动写的长文
- 决策类回复
- 对别人方案/代码的评论
- 日常消息

原因很简单：
- 长文最能暴露方法论
- 决策回复最能暴露判断框架
- 评论最能暴露标准和雷区
- 日常消息最能暴露语气和节奏

所以它蒸馏的不是“这人全部人格”，而是“这个人在工作场景下的可复现部分”。

**优点**

- 工程实现简单，几乎不依赖训练基础设施
- 可解释性强，生成结果能直接读、直接改
- 可持续进化，有版本管理和纠错机制
- 双层结构清晰，复用性比一坨 Prompt 高
- 很适合知识交接、角色替身、经验显性化

**局限**

- 高度依赖原材料质量和覆盖度
- 很容易学到“话术外壳”，却不一定学到真正深层能力
- Persona 比 Work 更容易拟真，Work 更容易失真
- 如果标签先验太强，可能把人刻板化
- 本质还是 prompt-based simulation，不是真正继承了人的隐性思维

**一句话总结**

- 这个 Skill 的工作原理是：**先收集同事留下的各种文本痕迹，再按“工作能力”和“人物性格”两条线做结构化提取，最后把结果固化成一个新的可调用 Skill。**
- 它实现“蒸馏同事”的方式不是训练模型，而是 **把同事的规范、经验、口头禅、决策习惯、人际反应，压缩成一套高约束 Prompt 与规则系统。**
