# work.md 包含什么内容才能像一个人那样工作的文生图提示词

Create a highly detailed technical infographic in hand-drawn sketch style on pure white background. The theme is "What must be inside work.md so an AI can work like a real colleague".

Use a single clear narrative: a completed `work.md` is opened layer by layer, revealing the essential sections that make an AI capable of working in a recognizable professional style. The image should explain the internal content structure of `work.md`, not the source material or extraction process.

**Main design concept:**
Use an "operational blueprint opened in sections" metaphor:

- Center = a large opened `work.md` blueprint as the hero visual
- Around it = six essential content modules expanded outward like engineering layers
- Bottom or right side = examples of how these modules drive concrete work output

The focus should be on showing what categories must exist inside `work.md`, and why those categories are enough to make the agent behave like a real worker rather than a generic AI.

**Layout structure:**

Vertical or centered infographic with 4 major sections, connected by clear engineering arrows.
Image-to-text ratio should be approximately 60% diagrams / 20% text / 20% whitespace.
Use compact labels, strong visual hierarchy, and enough white space so the image does not feel crowded.

**Title area (top, 15-20%):**

- Main title in medium-large bold font:
  "work.md 里到底要装什么，AI 才能像一个人那样工作"
- Subtitle below in smaller font:
  "把职责、规范、流程、风格和经验装进同一份工作蓝图，最终形成可执行的工作协议"

【一：work.md 的工作蓝图 - 里面到底装了什么、各模块分别负责什么】

- Center hero visual should be a large opened `work.md` blueprint card
- The blueprint should be divided into six clearly labeled blocks:
  · 职责范围
  · 技术规范
  · 工作流程
  · 输出风格
  · 经验知识库
  · 工作能力使用说明
- Add a small annotation:
  · "这不是简历摘要"
  · "这是替他干活时要遵守的工作协议"
- Show six expanded module cards around the central blueprint
- For each module, add concise sub-annotations:
  · 职责范围：系统、模块、业务边界
  · 技术规范：技术栈、接口规范、命名习惯
  · 工作流程：接需求、写方案、CR、排障
  · 输出风格：结论前置、列表化、文档颗粒度
  · 经验知识库：重复出现的判断、踩坑结论、默认原则
  · 工作能力使用说明：写代码、写文档、CR、排障时各调用哪部分规则
- Add a side note:
  · "少一层，模型就会少一类稳定行为"
- Add a second side note:
  · "每个模块解决一种不同的工作问题"
  · "模块之间不是重复，而是分工"
- Suggested diagram elements:
  · large blueprint document
  · section dividers
  · small icons for each module
  · outward expansion arrows from the blueprint
  · six module cards orbiting the blueprint
  · thin leader lines linking modules back to the main document
  · tiny icons: system map / API / flowchart / doc template / memory card / router
- Arrow label:
  · "模块齐全，work.md 才能覆盖完整工作行为"

【二：不同岗位被抽象成什么样 - work.md 不是空模板】

- Show 4 compact role slices arranged around or below the central blueprint
- Each role slice should look like a mini extracted profile card, showing that the same `work.md` structure will落成不同岗位特征
- Keep them concrete and visual, not abstract:
  · 前端工程师：
  - 组件拆分原则
  - 状态归属
  - 错误反馈
  - 可访问性 / 响应式
  · 产品经理：
  - 目标 / 范围 / 非目标
  - 优先级判断
  - 边界条件
  - 验收口径
  · 后端工程师：
  - 接口规范
  - 事务边界
  - 幂等 / 错误码
  - 线上排障
  · 算法工程师：
  - baseline
  - 指标定义
  - 数据处理规范
  - 上线检查
- Add a small annotation:
  · "同样是 work.md，不同岗位会长出不同的工作蓝图"
  · "结构相同，内容因岗位而异"
- Suggested diagram elements:
  · four mini role blueprint cards
  · tiny icons and checklist snippets on each card
  · thin arrows linking these role cards back to the six core modules
  · subtle labels showing which模块在该岗位更突出
- Arrow label:
  · "抽象出的不是统一模板，而是岗位化工作蓝图"

【三：work.md 如何驱动真实工作 - 不同任务会调用哪些具体规则】

- Show a compact task routing diagram instead of再解释内部模块
- Use 4 representative task cards:
  · 写代码
  · 做 Code Review
  · 写技术方案
  · 处理线上故障
- For each task, show both the模块 and the具体规则 that get pulled in:
  · 写代码
  - 调用：技术规范 + 经验知识库
  - 规则示例：统一返回结构 / 组件拆分 / baseline 先行
  · Code Review
  - 调用：技术规范 + 工作流程
  - 规则示例：N+1 / 并发安全 / 可访问性 / 边界条件
  · 技术方案
  - 调用：工作流程 + 输出风格
  - 规则示例：先写目标范围 / 再写异常流程 / 结论前置
  · 线上故障
  - 调用：工作流程 + 经验知识库
  - 规则示例：先看监控 / 再查发布 / 再缩小范围
- Add a tiny dispatcher note:
  · 工作能力使用说明 = 任务路由器
- Add a small annotation:
  · "work.md 不直接替你输出所有答案"
  · "它先决定当前任务应该按哪套规则执行"
- Suggested diagram elements:
  · dispatcher node
  · four task cards
  · selective routing arrows
  · tiny module badges attached to each task
- Arrow label:
  · "不同任务，调用不同模块组合"
