# work.md 和 persona.md 在真实场景中是如何协同完成工作的

## 一、先说结论

如果把“蒸馏同事”拆开看，`work.md` 解决的是“会不会做”，`persona.md` 解决的是“会怎么接”。但一个电子同事真正能工作，不是因为这两份文件各自存在，而是因为它们在任务运行时形成了一条前后衔接的执行链。

更直接一点说：

- `persona.md` 负责决定态度、语气、优先级和互动方式
- `work.md` 负责决定方法、规范、步骤和产出内容
- 两者装配完成后，真实任务不是“先随机说两句，再随机干活”，而是先由 Persona 做任务入口判断，再由 Work 接管任务执行，最后回到 Persona 包装输出

所以，这个系统的关键不只是“抽象出了两层”，而是“定义了两层的协作顺序”。

## 二、先把两者的分工再压缩成一句话

面对一个真实场景时，可以把两者理解成：

- `persona.md` 是前台控制层
- `work.md` 是后台执行层

前台控制层要回答的问题是：

- 这个任务我愿不愿意接
- 我要先追问，还是直接回答
- 我是强势一点、敷衍一点，还是认真拆解
- 被质疑时我要解释、反问，还是先甩锅

后台执行层要回答的问题是：

- 这件事应该按什么流程做
- 用什么规范产出结果
- 哪些点是默认要检查的
- 最终结果应该长成什么结构

如果没有 `persona.md`，系统会做事，但不像这个人。
如果没有 `work.md`，系统会像这个人说话，但不一定真能把事做对。

## 三、完整协同流程：从收到任务到给出结果

装配完成后，一个真实任务通常会经历六步：

### 1. 接收任务

用户抛来一个真实请求，比如：

- 帮我看一下这个接口设计
- 这个线上故障怎么排查
- 帮我写一份技术方案
- 这个需求值不值得做

这时系统拿到的还只是任务文本，不会立刻进入纯执行。

### 2. Persona 先做入口判断

这一步由 `persona.md` 先接管。它会隐含回答四个问题：

- 这个人遇到这种任务时，第一反应是什么
- 他会不会先补 context
- 他会用什么态度进入对话
- 他会不会在开始前先划清边界

比如一个“字节范 + 强势 + CR 很严格”的 Persona，可能会先问：

- 这个接口的 impact 是什么
- 调用量、时延目标、失败场景你先说清楚
- 背景不清楚，先别讨论实现

而一个“差不多就行 + 创业公司派”的 Persona，可能会直接进入：

- 先做最小可行版本
- 复杂的先别上
- 能跑通主链路再说

也就是说，任务一进来，先决定的不是“答案是什么”，而是“以什么姿态进入解题”。

### 3. Persona 决定互动策略

在真正做事前，`persona.md` 还会进一步决定互动方式：

- 信息不足时，是温和追问还是直接打断
- 需求模糊时，是主动补全还是把问题抛回去
- 遇到冲突时，是先妥协、先质疑，还是先找责任边界
- 面对上级、平级、下级时，说话方式是否切换

这一步非常重要，因为它决定了同样一个任务，系统是以“咨询顾问”的姿态进入，还是以“强势 reviewer”的姿态进入，还是以“会先自保的同事”的姿态进入。

## 四、Work 接管：把任务变成可执行流程

当 Persona 把态度和沟通入口定下来后，`work.md` 才真正开始接管任务主体。

它会把任务映射到自己内部已经抽象好的能力模块：

- 如果是写代码，就调用技术规范和代码风格
- 如果是写方案，就调用方案结构和输出格式偏好
- 如果是做 Code Review，就调用 CR 检查重点
- 如果是处理故障，就调用线上问题处理流程
- 如果是回答技术问题，就优先调用经验知识库里的判断

这一步的关键在于，`work.md` 不只是“存知识”，而是带有任务分发能力。它像一个轻量级的执行手册，会判断当前任务属于哪类工作，再把对应规范调出来。

## 五、Work 产出中间结果，Persona 负责最终包装

`work.md` 负责把任务做出内容，但最终给用户看到的结果，通常还要重新经过 `persona.md` 的风格包装。

也就是说，最后对外输出时，系统会同时满足两件事：

- 内容层面像这个人的工作方法
- 表达层面像这个人的说话方式

举个简单对比。

同样是发现接口设计有问题：

- 纯 Work 风格可能会说：返回结构不统一，缺少错误码定义，分页语义不清晰，建议补齐幂等策略
- 加上 Persona 之后，输出可能会变成：先别急着写，这个接口 impact 是什么？返回结构为什么不是统一的 `{code, message, data}`？分页语义也没定义清楚，幂等怎么保证

前一句更像规范摘要，后一句更像“这个人本人在说话”。

## 六、真实场景一：做 Code Review 时怎么协同

假设用户说：“帮我看一下这个接口设计。”

协同流程大致是：

- `persona.md` 先判断
  - 这个人是温和型 reviewer，还是高压型 reviewer
  - 他会不会先问业务背景和 impact
  - 他批评问题时是直接指出，还是分级表达
- `work.md` 再执行
  - 检查返回结构、错误码、分页、幂等、事务、性能风险
  - 检查是否符合他的接口规范和 CR 重点
  - 给出修改建议和阻塞项
- 最终输出再回到 `persona.md`
  - 用他的口头禅、句式和正式程度说出来
  - 如果他平时说话比较冲，结果就不会像标准机器人那样中性

所以在 CR 场景里：

- `work.md` 决定“看什么”
- `persona.md` 决定“怎么说”

## 七、真实场景二：处理线上故障时怎么协同

假设用户说：“线上报错暴涨了，帮我判断应该先查哪里。”

协同流程会变成：

- `persona.md` 先接入
  - 有些人格会先追问最近是否有变更
  - 有些人格会先确认告警级别和影响面
  - 有些人格会先问“这块是不是你们刚动过”
- `work.md` 再给出排查框架
  - 先看监控指标
  - 再看最近发布记录
  - 再缩小异常范围
  - 再定位具体链路、依赖和错误模式
- 输出阶段再由 `persona.md` 包装
  - 强势型人格会像在现场指挥
  - 背锅侠型人格会先接住问题再分析
  - 甩锅型人格会先明确“是不是这次改动引入的”

这里可以明显看到，人格不是装饰品，它会改变排障对话的组织方式。

## 八、真实场景三：写技术方案时怎么协同

假设用户说：“帮我起草一份新服务的技术方案。”

这时通常是：

- `persona.md` 决定起手方式
  - 是先对齐背景和目标
  - 还是直接给方案骨架
  - 是强调 impact、资源和风险
  - 还是先把边界讲清楚
- `work.md` 决定正文结构
  - 方案怎么分章节
  - 技术选型怎么展开
  - 异常流程怎么写
  - 发布、监控、回滚要不要单列
- 最终再回到 `persona.md`
  - 让整份方案保留这个人的文风
  - 比如结论前置、喜欢列点、喜欢先讲方法论，或者喜欢直接下判断

所以写方案时：

- `persona.md` 定义写作姿态
- `work.md` 定义正文骨架

## 九、真实场景四：面对模糊需求时怎么协同

假设用户说：“我想做一个推荐模块，你帮我看看。”

这种场景最能看出两层配合。

很多普通 AI 会直接开始出方案，但这个系统会先经过 Persona：

- 如果这个人习惯先问目标，会先问推荐什么、给谁用、衡量指标是什么
- 如果这个人习惯先划边界，会先问这是不是他职责范围
- 如果这个人有强烈的风险意识，会先问数据、依赖和上线约束

然后 Work 才会进入：

- 根据岗位能力给出模块拆分
- 给出推荐策略、实验方式、埋点、监控或接口设计建议
- 形成一个符合该岗位习惯的初步方案

这说明 `persona.md` 不只是控制语气，它还在任务入口处控制“问题是如何被定义的”。而问题定义方式，往往正是一个同事最有辨识度的部分。

## 十、用一句更工程化的话来描述它们的配合关系

如果用更工程化的语言来讲：

- `persona.md` 像任务前置策略层
- `work.md` 像任务执行规则层
- 最终输出像策略层重新接管后的响应渲染层

也可以理解成：

- Persona 负责路由任务的互动方式
- Work 负责路由任务的执行方式
- 两者合起来才构成完整代理

## 十一、为什么必须这样拆，而不是混成一份大 Prompt

因为真实工作里，“怎么做事”和“怎么互动”虽然高度相关，但不是同一种信息。

如果混成一份大 Prompt，会出现几个问题：

- 规则之间容易互相覆盖
- 更新时难以判断该改工作规范还是改人格行为
- 输出会变得既不像人，也不稳定
- 很难单独复用某一层能力

拆开之后，系统才有可能做到：

- 能力升级不改人格
- 人格纠偏不动工作规范
- 同一个 Work 可以搭不同 Persona
- 同一个 Persona 也可以在不同 Work 上复用

这正是它具有工程感的地方。

## 十二、一句话总结

`work.md` 和 `persona.md` 的协同，本质上不是“两份文档同时存在”，而是一次任务在运行时经历了“人格定入口、能力做执行、人格包输出”的三段式编排。
