# Vibe Coding 的最佳工程实践

本文聚焦“工程化 Vibe”：让 AI 产出更快，但把质量、可维护性、安全性用流程锁住。

## 0. 团队共识：Vibe Coding 宣言

在团队推广 Vibe Coding 前，建议达成以下共识，防止“效率工具”变成“烂代码生成器”：

1.  **AI 是手，人是脑**：AI 负责实现，人负责定义验收标准。
2.  **无验证，不提交**：未经测试或人工验证的代码，禁止合入主干。
3.  **回滚优于修补**：AI 陷入混乱时，立即回滚重来，不要试图通过对话“修修补补”。
4.  **意图显性化**：所有的 Prompt 必须包含明确的工程约束（上下文、边界、风险）。

## 1. 核心心法：Thinker & Doer 双模驱动

2026 年的 Vibe Coding 更接近**“推理模型做决策 (Thinker) + 编码模型做执行 (Doer)”** 的组合拳（同一产品线里也可能同时包含两种能力）：

- **Thinker（推理模型，例如 DeepSeek-R1、OpenAI o1/o3 类）**：负责“慢思考”。用来设计 API、拆解任务、排查复杂 Bug、补齐测试用例与边界。
- **Doer（编码模型，例如 Claude Sonnet、GPT-4o 类）**：负责“快执行”。用来补全代码、写 CRUD、做局部重构、生成文档与示例。

**最佳实践**：

- 遇到复杂需求，先用 Thinker：“请设计接口/数据结构，并拆解成 3 个可执行的 Step（每步给验收标准）。”
- 拿着拆解好的 Step，交给 Doer：“按 Step 1 实现，不做无关重构，补齐测试并给出验证命令。”

## 2. 先定边界：哪些任务适合交给 AI

优先选择“可验证、低耦合、失败可回滚”的工作：

- 修复明确的 bug（有复现步骤/日志/测试样例）
- 小范围重构（限定文件与验收标准）
- CRUD、表单校验、适配器/胶水代码、脚手架代码
- 文档驱动的改动（OpenAPI/Schema/接口契约明确）

谨慎或避免把 AI 当成决策者的工作：

- 核心架构与边界划分（分层、依赖方向、核心抽象）
- 安全/权限/加密等高风险逻辑（可以用 AI 辅助，但必须人主导）
- 性能敏感路径、并发一致性、资金/交易逻辑（必须可解释、可证明、可回滚）

## 3. 用“可控循环”替代“凭感觉循环”

工程化 Vibe 的最小闭环建议固定为 6 步：

- **明确验收**：输出“可运行、可验证”的验收标准（测试、接口行为、边界条件、错误码、性能上限）。
- **提供上下文**：入口文件、相关模块、现有模式、约束条件（不新增依赖/不改公共接口/遵循现有风格）。
- **让 AI 先给计划**：要求先列“改哪些文件、怎么改、风险点、如何验证”，再执行。
- **小步提交变更**：一次只做一个可合并的切片（slice），保证随时可回滚。
- **强制验证**：本地/CI 跑完测试、lint、typecheck，再进入 review。
- **人类把关**：审查接口契约、错误处理、安全、依赖与迁移；必要时让 AI 解释关键决策，但最终以人类判断为准。

这套循环的目标不是“让 AI 更聪明”，而是“让错误更早暴露、让回滚更便宜”。

## 4. 拆解任务：把一次对话变成一个可合并切片

拆解标准用三条就够：

- **范围可枚举**：明确说出允许触碰的文件/目录；超出范围必须先停下来说明原因。
- **行为可描述**：用输入/输出/错误分支描述行为，不用“差不多”“类似”。
- **验证可自动化**：至少一条自动化验证（单测、集成测、类型检查、静态分析、契约测试）。

实践上，把“大功能”拆成这样的序列更稳：接口/Schema → 核心逻辑 → 错误处理 → 可观测性（日志/指标） → 性能与边界 → 文档与示例。

## 5. “任务卡”提示词：把需求写成工程约束

用固定结构给 AI 下发任务，比“想到哪说到哪”稳定很多。建议每次都给出下面这些字段（按需裁剪）：

- **目标**：一句话说明要做什么，不写实现方案。
- **现状入口**：相关文件路径、关键函数/类名、调用链入口。
- **约束**：不新增依赖/不改公开 API/遵循现有分层/保持向后兼容/性能上限等。
- **验收标准**：必须新增或更新哪些测试；哪些命令必须通过；给出 2–3 个关键用例（含边界与错误分支）。
- **变更范围**：允许改哪些文件；不允许碰哪些模块（例如 auth、billing）。
- **风险点**：安全、数据迁移、并发一致性、回滚策略。
- **输出要求**：先给变更计划与风险清单；实现后给验证步骤与回滚方式。

可以把上面的字段固化成一个可复制的“任务卡”：

```text
目标：<一句话目标>

现状入口：
- 入口：<文件路径>#<符号>
- 相关：<文件路径>#<符号>（可选）

约束：
- 不新增依赖；不改公开 API；遵循现有目录与分层
- 变更限制在：<允许修改的目录/文件列表>

验收标准：
- 用例：<输入> -> <输出>；<边界>；<错误分支>
- 必须通过：<test 命令>、<lint 命令>、<typecheck 命令>

输出要求：
1) 先给 3–6 步改动计划 + 将修改的文件清单 + 风险点
2) 再实现；不要做无关重构/格式化
3) 最后给验证步骤与回滚方式
```

## 6. 验证优先：让测试成为 AI 的“护栏”

工程化 Vibe 里，“不读代码”可以有限度存在，但“不验证”不允许存在。优先级建议如下：

- **先补可复现**：先把 bug 复现写成测试/脚本，再让 AI 修。
- **先补边界**：空值、超长、非法枚举、权限不足、超时、重试、幂等。
- **先补契约**：对外接口（HTTP、RPC、SDK）用契约测试或示例驱动固定行为。
- **把验证写死**：每次让 AI 输出“我运行了哪些命令，结果是什么”；CI 里把这些命令固定下来。

## 7. Review 清单：把注意力放在“高价值风险”

AI 生成的代码最常见的问题不是“语法”，而是“工程约束被悄悄破坏”。Review 建议重点看：

- **接口与语义**：函数/接口的输入输出、错误语义、兼容性是否被改变。
- **错误处理**：异常吞没、返回值混乱、重试/超时策略是否一致。
- **依赖变化**：是否引入新库、版本是否合理、license 与安全扫描是否过关。
- **安全基线**：鉴权鉴别、注入、SSRF、路径遍历、敏感信息日志、CORS 配置。
- **层次与耦合**：是否把数据访问塞进 UI 层、把网络调用塞进领域层等。

## 8. 上下文工程：给“足够少但足够用”的信息

- **显式引用**：直接贴文件路径与关键符号名，让模型能定位到正确的“模式”。
- **MCP (Model Context Protocol)**：在“需要让模型访问真实系统/数据源”的团队里，MCP 正在成为通用连接层。通过 MCP Server 让 AI 直接读取数据库 Schema、Git 历史或云端日志，能减少手工复制粘贴导致的遗漏与过期信息。
- **带上失败证据**：复现步骤、输入样例、日志、堆栈、期望与实际差异。
- **固定约束语句**：反复强调团队风格与限制（不加依赖、保持 API、遵循现有目录结构）。
- **把 MCP 当生产基础设施**：最小权限、白名单、审计日志、密钥轮换、隔离环境；禁止把 MCP 当“随便装个插件”。

## 9. 版本控制：用回滚能力换速度

- **原子提交**：一次提交只包含一个意图；确保每个提交都能独立通过测试。
- **检查点提交**：做大改动时先提交可编译/可运行的中间态，方便回滚与二分定位。
- **拒绝大爆炸合并**：避免“几百个文件一次性改完”；把 AI 的输出切片化。

## 10. 常见反模式（会把项目带进维护地狱）

- **“Accept All” 常态化**：跳过 diff 与验证，等同于把代码库当黑盒运行。
- **让 AI 设计架构**：AI 可以给候选方案，但边界、依赖方向、演进策略必须人拍板。
- **无复现修 bug**：只有一句“报错了，修一下”，模型只能瞎猜，改动会扩散。
- **不断加依赖图省事**：短期省时间，长期增加供应链风险与维护成本。

## 11. 推荐工具栈

工欲善其事，必先利其器。工具选型的第一原则是：**合规与可控优先，其次才是“能生成多少代码”**（尤其是公司代码、客户数据、密钥与日志）。

- **IDE/编辑器（按团队约束选一个主力即可）**：
  - **AI-first IDE**：Cursor / Windsurf / Trae 等。这一类的优势是多文件改动与计划执行更顺滑；风险点通常在数据权限、外部传输与可控性。
  - **VS Code 生态**：GitHub Copilot +（Cline / Continue / 其他 Agent 插件）。优势是可控、可替换、团队一致性更强，适合平台化治理。
  - **JetBrains 生态**：如果团队以 IntelliJ/PyCharm/GoLand 为主，优先保证“编辑体验一致 + AI 作为增强”，不要为了 AI 牺牲主力工作流。
- **模型组合**：
  - **Thinker**：选择“推理稳定、能给出结构化计划与风险点”的模型（R1 / o 系列等作为代表）。
  - **Doer**：选择“指令遵循强、代码风格一致、改动可控”的模型（Claude Sonnet / GPT-4o 类作为代表）。
- **终端工具**：Aider（适合 CLI 重度用户；用 Git 作为变更边界，天然利于回滚与审查）。
- **基础设施与质量闸门**：
  - **MCP Servers**：把 Git、Issue、Schema、日志等“事实来源”接入模型，但必须带权限控制与审计。
  - **SAST/依赖安全**：Semgrep / Sonar / Snyk / Trivy / Dependabot（任选组合），用于兜底“AI 误删鉴权/引入高危依赖”等风险。
  - **Lint/Test/Type**：选择“速度快、默认严格、CI 易固化”的组合。前端可用 Biome + Vitest；Python 可用 Ruff + Pytest；Go 可用 golangci-lint + go test。

## 总结

工程化 Vibe 的关键不是“更会聊天”，而是把软件工程里最贵的部分（质量、安全、可维护）变成硬约束：先验收、后生成；先切片、后扩展；先验证、后合并。这样才能把“速度”变成团队的长期优势，而不是技术债的加速器。
