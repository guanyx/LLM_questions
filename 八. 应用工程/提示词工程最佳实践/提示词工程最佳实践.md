如果你把提示词当成写作比赛，你会得到偶尔灵验的答案。

如果你把提示词当成效率工程，你会得到稳定可复用的交付。

这就是我最近看了网上最火的四份提示词教程之后得出的最核心结论。

很多人现在最耗时间的动作不是"不会用模型"，而是每次都从零写提示词，写得越来越长，返工越来越多，最后把成本花在重试而不是产出上。

所以这篇只做一件事：教你怎么高效编写提示词，让一次命中率上升、返工轮次下降、模板复用率提升。

##

一、先定义高效：不是写得多，而是写得可执行

高效提示词可以用一个很工程化的公式理解：

高效 = 一次命中率 × 可复用率 ÷ 返工轮次

附录中的四份白皮书的共同点都落在这个公式上，细节表述不一样，但底层逻辑一致：复杂任务要拆分、输入信息要分舱、动作语言要明确、示例要做双向约束、参数要按任务容错率配置。

后面的每一条原则，都直接对应这个公式里的某个变量：有的提升命中率，有的降低返工轮次，有的让模板可以反复使用\`。

如果你的提示词不能被同事直接复用，不能在下周同类任务里继续使用，不能让返工从 4 轮降到 1-2 轮，那它就不是高效提示词，只是一次性对话。

##

二、从 Mega Prompt 到流水线：效率提升的第一步

低效写法是把意图识别、资料提取、结构组织、风格控制一次性塞给模型。模型每一步都可能引入噪声，前面一处偏差会污染后面全部输出。

高效写法是把任务拆成 3 段流水线：

- 第 1 段：确定任务背景，引入上下文。
- 第 2 段：确定任务执行动作，只处理逻辑顺序。
- 第 3 段：确定任务输出方式，以及相关边界限制。

这样做的价值是可定位、可回滚、可调优：错了只改当前节点，不用整段重写，这一步通常就是效率差距最大的地方。

##

三、使用 XML 标签进行边界管理

很多文章讲提示词却不把 XML 标签写出来，等于讲"结构化隔离"但不给可执行结构。

下面是一段作为参考的 XML 骨架，用来隔离规则、上下文、示例和输出格式：

xml

```xml
<prompt>
  <system_role>你是资深AI编辑，目标是输出可直接发布的中文专栏内容</system_role>
  <goal>将输入资料整理为结构清晰的深度文章</goal>
  <instructions>
    1. 先提取事实，再组织结构，最后生成正文
    2. 不编造数据，不遗漏关键结论
    3. 严格遵循输出格式
  </instructions>
  <context>
    在此粘贴原始资料、访谈记录、会议纪要等
    ⚠️ 注意：上下文要剪枝，不是越全越好。无关信息会稀释关键内容的权重，建议只放与任务直接相关的段落。
  </context>
  <examples>
    <good>示例：结论明确 + 证据对应 + 结构完整</good>
    <bad>示例：空话开场 + 无证据 + 结构跳跃</bad>
  </examples>
  <constraints>
    字数 1800-2200；术语首次出现给一句解释；不使用营销腔
  </constraints>
  <output_format>
    标题
    开头（问题定义）
    正文（3-5段逻辑递进）
    结尾（行动建议）
  </output_format>
</prompt>
```

这里面的每一个 XML 标签都代表了一个语义，通过语义把不同的提示词部分拆开，对 AI 来说也更容易理解。

这套结构的核心是把"必须遵守的规则"从"可参考的资料"里分离出来，避免上下文污染，也能明显降低提示词注入导致的跑偏。

##

四、长上下文的高效排序：资料在前，问题在后

四份白皮书里有一个容易被忽略但很实用的共识：在长文本任务里，把资料放在前面，把问题放在最后面，通常比"问题先写"更稳。

原因不是玄学，而是模型的生成机制决定的。模型是从左到右逐字生成的，对尾部信息更敏感。问题放尾部，等于在最后一刻收束模型注意力——此时问题与生成内容之间的"距离"最短，能有效缓解中间内容的遗忘问题。

所以做文档问答、代码库分析、长报告摘要时，不要只想着"加更多提示词"，先把输入顺序排对，往往收益更大。

##

五、动作语言要精确：用"动词 + 对象 + 约束"写提示词

低效提示词常见表达是"看一下""优化一下""写得专业一点"，这些词没有执行边界，模型只能猜。

高效写法统一成一个句式：

- 动词：提取、对比、归纳、重写、推演。
- 对象：观点、数据、段落、方案、代码。
- 约束：字数、格式、证据、口吻、禁止项。

示例：

> 对比 A/B 两个方案在成本、风险、收益上的差异，各给 2 条证据，输出为三列表格，结论不超过 100 字。

你会发现这类提示词没有华丽修辞，但执行命中率显著更高，这才是高效。

##

六、示例要双向：Good/Bad 同时给，返工会明显下降

当任务有严格格式或质量要求时，只给一个正确示例不够，因为模型只知道"最好是什么样"，不知道"不能走哪些路径"。

高效做法是给 3-5 组双向示例：

- Good：标准输出示范。
- Bad：常见错误输出。
- Why Bad：错误原因说明。

有两点需要注意：

第一，示例组数不要超过 5 组，过多反而会让模型过度拟合示例的格式，而忽略掉你的实际指令。

第二，这个方法在事实提取、分析报告、代码重构等高约束任务里收益最明显，几乎是最省时间的保险丝。

##

七、正向指令优先，负向规则做补充

你告诉模型"不要啰嗦"，模型仍然要先猜什么叫啰嗦，你告诉它"每点不超过 30 字，只保留结论和依据"，它就能直接执行。

所以高效提示词的顺序应该是：先写要做什么（正向动作），再写不要做什么（负向边界）。

这不是文字偏好，而是符合模型顺向生成逻辑的工程写法。模型先建立正向框架再设禁区，比一开始就说"不要"更容易被执行。

##

八、参数不是玄学：按任务容错率配置 Temperature / Top-P

所有大模型在接入时都可以设置模型温度。如果直接使用 Claude Code 之类的产品可能没有接触过，可以跳过这一节。

四份白皮书对参数的共识很清楚：参数要服务任务，不要凭感觉拉满，不同温度带来不同效果，实用快表如下：

任务类型 Temperature Top-P 事实核查、代码生成、数学推导 接近 0 收紧 常规总结、业务文稿、流程文本 0.2 - 0.5 中度约束 创意写作、头脑风暴、营销文案 0.7 - 0.9 放宽

要严谨的时候一定不要提高温度，否则返工一定增加，这个坑很多人都踩过。

##

九、把提示词做成资产：从"会写"到"可规模化"

最后一个常被忽略的高效点是资产化。

不要把提示词散落在聊天记录里，而要按场景沉淀模板库，例如信息处理模板、决策分析模板、内容生产模板，每个模板固定骨架，只替换变量。

更进一步，模板应该像代码一样做版本管理：记下每次修改的原因和效果对比。否则迭代出来的"好版本"很容易因为某次随手改动而悄悄失效，你甚至不知道是从哪一次开始变差的。

查理·芒格曾经说过：只要把 80 ～ 90 个重要模型学透并能运用，你就具备了通达世事者 90% 的认知底盘。

把这句话套用在 AI 提示词资产化上：只要迭代出 30-50 个自己实际需要的提示词，就能解决 9 成以上的重复性工作。

##

十、可直接执行的 10 分钟自检清单

发布前快速过一遍：

- 目标是否单一，还是把三件事塞进一个请求
- 是否使用 XML 或明确分块，而不是全部混写
- 动词是否可执行，是否带对象与约束
- 是否有 Good/Bad 双向示例（组数不超过 5 组）
- 参数配置是否匹配任务容错率
- 这条提示词下周能否直接复用

提示词高效编写的本质不是写得多，而是把任务表达做成可执行、可验证、可复用的系统。

##

附：四份指南的交集（更普适的提示词指南）

如果把四份指南“求交集”，它们其实都在强调同一件事：把提示词当成一份可执行的规格书，而不是一次性灵感写作。

这份交集版指南的目标是：同样一类任务，下次直接复用；同一个提示词，换人也能跑通；同一份输入，返工轮次明显下降。

### 1）先把成功标准写清楚：你到底想要什么“可交付”

很多返工不是模型不行，而是你没把“验收口径”写出来。通用写法是：

- 目标：一句话说明要解决的问题
- 受众：给谁看（决定用词、细节深度、口吻）
- 成功标准：什么算对、什么算不合格（能量化就量化）

把“写得更专业”替换成可验收指标，例如“用 5 条要点解释，每条不超过 25 字，包含 1 条反例”。

### 2）把提示词拆成固定分舱：规则、资料、动作、输出

四份指南都在强调“结构化隔离”：让模型分清哪些是必须遵守的规则，哪些只是参考资料，避免互相污染。

建议固定成 4 块（格式不限，XML/Markdown/标题分块都可以）：

- 规则（必须遵守）：事实边界、禁止项、风格限制、缺失信息如何处理
- 资料（只当输入）：原始文本、链接摘要、数据表、背景说明
- 动作（要做什么）：按顺序列出动词任务（提取/对比/归纳/改写/评审）
- 输出（要怎么交付）：字段、顺序、长度、是否需要表格/列表/段落

### 3）复杂任务别写 Mega Prompt：用“计划 → 执行 → 质检”的流水线

当任务包含多步推理或多种输出时，一次性塞进一个请求往往会把噪声串起来。更稳的交集做法是把任务拆成流水线：

- 计划：先列出要处理的步骤或提纲（只要结构，不要正文）
- 执行：严格按计划产出内容（只做规定动作）
- 质检：用自检清单复核（缺项就补，越界就删）

你可以把每一段当成可替换节点：错了只改当前节点，不用整段重写。

### 4）示例优先用来“锁格式”：Good/Bad 双向约束最省返工

共同共识是：当输出格式、口吻、颗粒度很重要时，给示例比加形容词更有效。

- Good：你希望的标准输出长什么样
- Bad：你最不想要的错误长什么样
- Why Bad：一句话说明为什么不合格（让模型知道禁区边界）

示例不宜太多（通常 1–3 组就够），否则模型容易“学格式不学任务”。

### 5）长上下文要先排顺序再加字：资料在前，问题在后

交集结论是：长文任务里，把资料放前面，把指令/问题放后面更稳；另外，“资料剪枝”往往比“追加提示词”更有收益。

可复用的输入组织方式：

- 先放资料：按来源或主题分段，必要时加小标题
- 再放指令：用清晰动作语句收束
- 最后放输出格式：让模型在生成前一刻对齐交付形态

### 6）动作语言用“动词 + 对象 + 约束”：把模糊词全部翻译成规则

四份指南都在反对“看一下/优化一下/写得专业点”这类不可执行表达。通用替换法：

- 动词：提取 / 对比 / 归纳 / 重写 / 评审 / 生成
- 对象：段落 / 观点 / 证据 / 表格 / 方案 / 风险点
- 约束：数量、字数、格式、证据要求、禁止项、优先级

### 7）不确定性要提前约定：缺信息怎么办、能不能编

通用且非常省返工的两条规则：

- 禁止编造：没有证据就写“资料未提供，无法判断”
- 缺失处理：需要补充的信息用“待确认清单”列出来（并说明为什么需要）

当任务涉及事实核查、法律、医疗、财务等高风险领域，这一条几乎是必备底座。

### 8）参数按容错率配置：先收紧，再放宽；一次只调一个旋钮

交集共识是：参数不是玄学，它决定了“稳”还是“散”。实用做法：

- 低容错（事实、代码、数学）：先把随机性收紧
- 中容错（总结、业务文稿）：适度开放
- 高容错（创意、脑暴）：再放宽

调参建议一次只动一个变量（先从温度开始），否则你很难知道是哪一项导致结果变差或变好。

### 9）把提示词做成资产：模板化、变量化、可评测

四份指南隐含的工程化落点是：提示词要能像代码一样复用与迭代。

- 模板化：固定骨架，只替换变量（主题、受众、字数、资料）
- 版本化：记录每次修改的原因与效果（返工轮次、命中率变化）
- 小评测：准备 5–10 条代表性输入做回归测试，防止“悄悄变差”

### 10）一页纸通用骨架（可直接复制改变量）

你可以把下面这份骨架当成“默认起手式”，按任务删减字段：

> 角色：你是谁（决定视角与口吻）
>
> 目标：要完成的交付是什么
>
> 受众：给谁看（可选）
>
> 资料：粘贴原始材料（只放相关段落）
>
> 动作（按顺序）：1）… 2）… 3）…
>
> 约束：必须…；禁止…；缺失信息则…
>
> 输出格式：字段/顺序/字数/是否需要表格或清单
>
> 自检：是否覆盖所有要点；是否有证据对应；是否越界编造
