# 微调的硬件准备

你这个理解是对的，但还差一个比“模型大小、训练时长、数据规模”更关键的变量：**你到底怎么微调**。

同样是 14B 模型，如果你做的是 LoRA，和做全参数微调，硬件需求可能差一个数量级；同样是一张 48GB 的卡，如果你训练的是短上下文分类任务，和训练长上下文、多轮对话任务，体验也完全不是一回事。

所以，评估微调硬件时，最稳妥的思路不是直接问“需要几张卡”，而是先把问题拆成 5 个层次：

- 你做的是哪种微调
- 基座模型有多大
- 训练时的上下文长度和 batch 要多大
- 你希望多快训完
- 你是不是要上多卡、分布式、长期稳定复现

如果这 5 个问题没想清楚，单看参数量，硬件预算很容易失真。

**先说结论：硬件不是只看模型大小**
很多人第一次做微调，会天然认为“模型越大，显卡越大”。这当然没错，但只说了一半。

真正决定硬件需求的，通常是这几部分：

- **权重本身**：模型参数要先放进显存
- **梯度**：训练时不是只前向，还要反向传播
- **优化器状态**：比如 Adam 这类优化器会额外保存状态，占不少内存
- **激活值**：中间层结果要留着反传，序列越长，这部分越吓人
- **吞吐目标**：你是能跑就行，还是希望一天内训完

你可以把它类比成搬家：

- 模型权重像是“固定家具”
- 梯度和优化器状态像是“施工时额外占地的工具和材料”
- 激活值像是“施工现场临时堆放的箱子”

很多时候，真正把显存顶满的，不是那套固定家具，而是施工现场的临时堆放物。

这也是为什么现在做微调，**上下文长度和有效 batch，经常比想象中更影响硬件规划**。

**第一步：先判断你做的是哪种微调**
这是硬件评估里最关键的一步。因为训练方式一变，预算量级就会变。

常见可以先粗分成 4 类：

- **LoRA / DoRA / Adapter 类微调**
  - 只训练少量新增参数
  - 对显存最友好
  - 现在大多数业务微调，默认先考虑这一类
- **QLoRA / 量化后再做适配器微调**
  - 在 LoRA 的基础上继续压低显存
  - 更适合单卡试验和成本敏感场景
  - 代价通常是吞吐更低、调试更细、上限和稳定性要更谨慎评估
- **全参数微调**
  - 真正把整个模型都训起来
  - 对显存、通信、稳定性要求都高很多
  - 一般只有模型规模不大，或者你有很明确的必要性时才考虑
- **偏好优化 / DPO / RLHF 后续阶段**
  - 如果建立在 SFT 模型之上
  - 硬件需求通常和前面的训练方式有关
  - 但因为训练流程更复杂，工程开销会继续上升

一句话记住：

- **先决定训练方式，再决定硬件**

不是反过来。

**第二步：基座模型大小决定“地板”，上下文长度决定“天花板”**
模型大小决定的是你至少要准备多少资源；但真正把你从“能跑”推到“跑不动”的，往往是上下文长度。

这背后最关键的术语有两个：

- **显存（VRAM）**
  - 就是 GPU 自己的高速工作区
  - 显存不够，训练通常直接失败，或者被迫把 batch 和长度压得很小
- **激活内存**
  - 就是前向传播过程中临时保存的中间结果
  - 序列越长、batch 越大、层数越多，这部分越容易爆

很多人会低估第二个。

比如你训练一个 7B 模型：

- 如果只是短输入、短输出、LoRA、batch 很小
  - 一张消费级高显存卡就有机会跑起来
- 但如果你把上下文拉长很多，或者想要更像样的吞吐
  - 需求会迅速往上跳

所以硬件规划时，至少要同时看 4 个量：

- 模型参数规模
- 上下文长度
- 有效 batch size
- 训练方式

这 4 个量缺一个，预算都可能不准。

**第三步：训练时长其实是在问“你要多高的吞吐”**
很多人说“我想估硬件”，其实隐含的意思是：

- 我希望多久训完

这会直接影响你是选：

- 单卡慢慢跑
- 多卡并行
- 还是直接上高带宽、高显存的数据中心卡

因为训练时长本质上近似取决于两件事：

- 总共要处理多少 token
- 你每秒能吞多少 token

第一项跟数据规模和 epoch 有关，第二项跟硬件、精度、并行方式、序列长度都有关系。

这也是为什么“数据量不大”不一定意味着“硬件要求低”：

- 如果你要求一天内出结果
- 或者你要频繁调参、反复重训

你需要的不是“勉强能跑”，而是“有足够吞吐”。

现实里，很多团队真正买的不是“最低可运行硬件”，而是“能在业务节奏里完成迭代的硬件”。

## 第四步：给你一个够用的硬件判断框架

如果你想要的是一种**能快速落判断**的方法，我更建议用“**先定基线，再看增压，再看值不值得升级**”的方式，而不是只列概念。

也就是分三层想：

- **先用训练方式 + 模型大小，定一个基线档位**
- **再用上下文长度 + batch，看要不要上调档位**
- **最后用吞吐目标 + 数据规模，判断升级值不值**

这样你不是在背知识，而是在做一个很像产品决策的分档判断。

### 第 1 层：先用“训练方式 × 模型大小”定基线档位

你可以先把项目粗分成 4 档。

- **A 档：轻量试验**
  - 常见组合：QLoRA + 7B～8B
  - 硬件直觉：先追求“跑起来”
  - 常落在：16GB～24GB 起步，24GB 更从容
- **B 档：常规微调**
  - 常见组合：LoRA + 7B～14B，或 QLoRA + 13B～14B
  - 硬件直觉：已经不是“能不能跑”，而是“跑得别太难受”
  - 常落在：24GB～48GB，部分场景希望到 48GB
- **C 档：高压训练**
  - 常见组合：LoRA + 30B～32B，或较小模型但想做更激进的全参数微调
  - 硬件直觉：高显存开始变成前提，不再只是优化项
  - 常落在：80GB 档，或者多卡配合
- **D 档：企业级 / 研究级**
  - 常见组合：70B+，或中大模型的全参数微调
  - 硬件直觉：瓶颈已经不只是显存，还有通信、稳定性和总成本
  - 常落在：多张高显存卡，甚至完整集群

先别急着算细节，先问自己一句：

- **我这个项目的基线档位，落在 A / B / C / D 的哪一档？**

这一步一旦定下来，硬件范围就已经收窄很多了。

### 第 2 层：再用“上下文长度 + batch”判断要不要上调一档

基线档位只是起点，不是最终答案。真正会把你从“能跑”推到“跑不动”的，往往是：

- 上下文长度
- 有效 batch size

你可以直接这么判断：

- **不加压**
  - 短上下文
  - 小 batch
  - 可以按基线档位先估
- **上调一档**
  - 上下文明显变长
  - 或者 batch 想做得更像样
  - 这时显存和激活内存压力都会明显上来
- **上调两档**
  - 长上下文 + 还想保留较高吞吐
  - 或者模型本来就不小，还要更大的有效 batch
  - 这时很容易从“单卡勉强跑”直接跳到“高显存卡 / 多卡”

一个很实用的记法是：

- **模型大小决定地板**
- **上下文和 batch 决定你会不会突然撞天花板**

所以同样一个 14B 模型：

- 短上下文、小 batch，可能还在 B 档附近
- 一旦拉长上下文，再想保吞吐，可能就被推到 C 档

### 第 3 层：最后用“吞吐目标 + 数据规模”判断升级值不值

前两层解决的是：

- 能不能装下
- 会不会爆显存

第三层解决的是：

- **值不值得为了更快出结果而升级硬件**

你可以把吞吐目标分成 3 种：

- **能跑就行**
  - 偏验证型项目
  - 只要能把链路跑通，慢一点也能接受
- **当天要出结果**
  - 开始对吞吐敏感
  - 这时更强硬件的价值会明显上升
- **要频繁重训 / 高频调参**
  - 这时买的已经不是“最低可运行硬件”
  - 而是“能跟上业务节奏的硬件”

再把数据规模放进来，你就能判断升级值不值。

- **数据规模小**
  - 不一定直接逼你升级显存
  - 但如果要反复试验，时间成本仍然可能很高
- **数据规模中等**
  - 会明显影响总训练时长和 checkpoint 数量
  - 更值得为吞吐买单
- **数据规模大**
  - 训练时长、迭代成本、存储和恢复训练压力都会上来
  - 这时更强硬件常常不只是“更舒服”，而是“更划算”

所以真正实用的判断，不是“数据大不大”，而是：

- **数据规模会不会让慢训练变得不可接受**

### 你可以直接拿去用的“硬件分档卡”

以后你接到一个微调任务，可以直接按这张卡走：

- **先定基线档**
  - QLoRA + 7B～8B → A 档
  - LoRA + 7B～14B / QLoRA + 13B～14B → B 档
  - LoRA + 30B～32B / 小模型全参数 → C 档
  - 70B+ / 中大模型全参数 → D 档
- **再看是否上调**
  - 长上下文 → +1 档
  - 更大 batch → +1 档
  - 长上下文 + 还要保吞吐 → 可能 +2 档
- **最后判断是否值得升级**
  - 只是验证方向 → 不一定升
  - 当天要结果 → 倾向升
  - 高频调参 / 多轮重训 / 数据规模大 → 更值得升

如果你愿意把它压成一句话，可以这样记：

- **先用训练方式和模型大小定基线，再用上下文和 batch 看会不会爆，最后用吞吐和数据规模决定要不要升级。**

## 第五步：别只盯着 GPU，配套资源也会卡你

上面这张“分档卡”解决的是主判断，但真正落地时，还要补一层系统检查。

真正做过训练的人很快会发现，GPU 只是最显眼的部分，不是全部。

至少还要看下面这些：

- **CPU 内存**
  - 数据加载、预处理、分布式训练控制、缓存都要吃内存
  - GPU 很强但 CPU 内存太小，会让整个流程变得很别扭
- **存储容量**
  - 基座模型、数据集、checkpoint、日志、多个实验版本都会占空间
  - checkpoint 往往比很多人预期大
- **存储带宽**
  - 如果数据读取慢，GPU 可能会“等数据”
  - 特别是在多卡和频繁 checkpoint 的场景下更明显
- **GPU 之间的通信**
  - 多卡不是简单地“数量乘上去”
  - 如果卡间通信差，扩卡收益会明显打折
- **训练框架的成熟度**
  - 同一套硬件，在成熟训练栈和临时拼装训练栈上，体验能差很多

你可以把这理解成：

- GPU 决定你有没有发动机
- 存储、内存、通信决定你这辆车能不能稳定跑高速

**一个很容易踩坑的误区：数据规模不是线性决定硬件**
你提到“数据规模”，这个变量当然重要，但它更直接影响的是：

- 训练总时长
- checkpoint 数量
- 迭代成本

它不一定线性决定“单次训练能不能放进显存”。

真正决定“能不能放进显存”的，通常是：

- 模型大小
- 训练方式
- 上下文长度
- batch

而数据规模更像是在决定：

- 你要跑多久
- 要不要多卡提速
- 总成本会不会失控

所以一个很实用的纠偏是：

- **模型大小决定能不能装下**
- **上下文和 batch 决定会不会爆显存**
- **数据规模决定你要跑多久、花多少钱**

**近两年很值得更新的 4 个判断**
如果你看的是更旧一点的微调经验，下面这几件事会明显改变硬件判断。

**1）适配器微调已经从“技巧”变成了默认起点**
过去很多人一提微调，脑子里还是“把整个模型训起来”。但现在在大多数业务场景里，更主流的起点其实是：

- LoRA
- QLoRA
- 其他参数高效微调方法

这直接改变了硬件门槛。很多以前必须上多卡高显存的尝试，现在可以先在更小的硬件上验证方向。

**2）长上下文越来越常见，激活内存成了更显眼的瓶颈**
这两年很多任务不再满足于短问答，而是：

- 长文摘要
- 多文档问答
- 工单串联
- 多轮对话

这会让“激活内存”比过去重要得多。也就是说，哪怕模型大小没变，任务一换，硬件判断也要重做。

**3）高端数据中心 GPU 的优势越来越不只是显存，还有训练效率**
近两年业内越来越重视：

- 更高带宽显存
- 更强张量计算能力
- 更适合混合精度训练的硬件能力

这意味着高端卡的价值，不只是“能装下”，还包括：

- 吞吐更高
- 多卡效率更好
- 工程稳定性更强

对于频繁迭代的团队，这些收益有时比“单卡能不能勉强跑”更重要。

**4）消费级卡更适合原型验证，但生产训练仍然更偏向数据中心环境**
现在确实有越来越多团队会用消费级 GPU 做：

- QLoRA 实验
- 小规模 LoRA
- 数据流程打通

但一旦进入：

- 长上下文
- 多卡并行
- 高吞吐
- 长时间稳定训练

数据中心卡和更成熟的服务器环境通常还是更省心。

这不是因为消费级卡“不能做”，而是因为整体工程边界更紧。

## 几个常见误区

**误区一：数据只有几千条，所以随便一张卡都行**
不一定。

如果你训的是大模型、长上下文，哪怕样本条数不多，显存压力也可能很大。数据少，只是说明你总时长可能没那么夸张，不代表单步训练就轻松。

**误区二：显存够装下模型，就等于能训练**
也不对。

训练不是推理。除了模型本体，还要考虑梯度、优化器状态、激活值。很多“能加载模型”的环境，一到训练就爆。

**误区三：为了效果最好，优先全参数微调**
现实里常常不是。

如果你的目标是风格对齐、格式约束、垂直任务学习，LoRA 一类方法往往先值得试。全参数微调通常要有更强的必要性，而不是默认选项。

**误区四：多加几张卡，速度就能线性翻倍**
很少这么理想。

多卡训练会引入通信成本、同步成本、调试复杂度。卡多了，不代表一定更省事。

## 现实世界里，怎么做最稳

如果让我给一个通用建议，我会这样分。

### 如果你是第一次做微调

- 优先从 LoRA / QLoRA 开始
- 优先用较小模型验证数据和评测链路
- 优先确认“值不值得训”，不要一开始就追求豪华硬件

### 如果你已经知道任务有价值，而且会反复迭代

- 优先提升显存余量和训练吞吐
- 优先减少“每次试验都要排很久”的等待成本
- 优先考虑更成熟的多卡和存储环境

### 如果你要做长上下文、较大模型、稳定业务训练

- 不要只按参数量估预算
- 把上下文长度和 batch 当成一等公民
- 提前把通信、checkpoint、恢复训练都纳入规划

**给你一个最实用的硬件评估清单**
你接到微调任务后，可以先写这样一页纸：

- **训练方式**
  - LoRA / QLoRA / 全参数 / 偏好优化
- **模型规模**
  - 7B / 14B / 32B / 70B+
- **上下文长度**
  - 常规还是长上下文
- **有效 batch**
  - 追求能跑，还是追求吞吐
- **目标时长**
  - 几小时、几天，还是一周内可接受
- **硬件形态**
  - 单卡 / 多卡 / 高显存卡 / 集群
- **配套资源**
  - CPU 内存、存储、checkpoint 空间、卡间通信
- **风险点**
  - 是显存风险、吞吐风险，还是工程稳定性风险

这张纸一写出来，硬件准备基本就清楚了一大半。

**一句话总结**
微调的硬件准备，表面上看是在估 GPU，实际上是在平衡 4 件事：

- 模型有多大
- 训练怎么做
- 你想多快拿到结果
- 你能接受多复杂的工程方案

先把训练方式、上下文长度和吞吐目标想清楚，再去看 GPU 型号和卡数，判断会准得多。

如果你继续往下学，我建议下一步顺着看这三个方向：

- 不同微调方式为什么会带来完全不同的显存需求
- 长上下文为什么会把训练成本抬得这么快
- 多卡训练为什么不是“卡数翻倍，速度就翻倍”
