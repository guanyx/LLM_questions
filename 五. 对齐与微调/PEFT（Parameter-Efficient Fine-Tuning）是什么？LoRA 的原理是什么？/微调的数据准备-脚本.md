# 微调的数据准备：信息图脚本

## 使用说明
- 本脚本拆成 2 张信息图
- 第 1 张覆盖“第一步到第三步”
- 第 2 张覆盖“第四步”
- 文案风格：专业、直接、适合知识型内容账号
- 目标：让读者在 30 秒内形成“接到微调需求后如何快速决策”的整体框架

---

# 信息图 1：先判断目标，再确定方案和数据形状

## 封面标题
老板让你微调一个模型  
你第一时间该怎么规划？

## 副标题
先别急着问要准备多少数据  
先把这 3 步判断清楚

## 版式建议
- 竖版信息图
- 结构：顶部标题区 + 中部 3 个纵向模块 + 底部收束区
- 每个模块都尽量保留原始信息，不做过度抽象
- 视觉重点：让读者顺着一张图看完“目标 → 手段 → 数据形状”

## 开场文案
接到“把某个基座模型微调成 xxx 效果”这种任务后，  
第一反应不应该是先去收数据，  
而是先依次判断：

- 目标到底属于哪一类
- 这件事是不是该靠微调解决
- 如果要做，每条数据应该长什么样

## 模块一：先把目标翻译成训练目标

### 模块标题
第一步：先把目标翻译成训练目标

### 核心文案
老板说的通常是业务语言，不是训练语言。  
你要先把它翻译成下面几类之一：

### 具体内容
- 风格对齐：更像某种口吻、品牌语气、客服语气
- 格式约束：稳定输出 JSON、表格、固定字段
- 任务学习：分类、抽取、改写、摘要、排序
- 复杂生成：SQL、代码、长文生成、多轮决策
- 知识增强：更懂公司知识、行业知识、产品知识
- 偏好对齐：更稳、更短、更礼貌、更少废话

### 收束提示
这一步最关键，因为不同类别直接决定数据长什么样。

### 视觉建议
- 直接做成 6 行分类卡片或 2 列 3 行卡片矩阵
- 不用额外隐喻，重点是把 6 类训练目标完整列出来
- 模块标题和分类内容之间用清晰分隔线

---

## 模块二：先判断是不是该微调

### 模块标题
第二步：先判断“是不是该微调”

### 核心文案
很多需求看起来像微调，实际上不一定。  
你可以直接用这个判断表：

### 判断表文案
- 如果目标是学风格、学格式、学固定任务：优先考虑微调
- 如果目标是补最新知识、企业知识、频繁变化知识：优先考虑 RAG / 检索，不要先想着 SFT
- 如果目标是让输出更符合偏好：先做 SFT，必要时再做偏好优化
- 如果目标是复杂工具调用、多步决策：可以微调，但要预期需要更高质量数据和更细评测

### 中间提问
一个很实用的问题是：

- 这个需求是想让模型“会一种行为”
- 还是想让模型“记住一批知识”

### 收束文案
- 前者通常适合微调
- 后者很多时候更适合检索或继续预训练

### 视觉建议
- 直接做成“目标类型 → 推荐实现手段”的对照区块
- 中间单独高亮一句提问，作为模块视觉焦点
- 不需要复杂分流图，保留判断逻辑本身即可

## 模块三：立刻确定数据最小单元

### 模块标题
第三步：立刻确定“数据最小单元”

### 核心文案
一旦知道目标类型，你就能很快知道数据该长什么样。

### 数据形状文案
- 风格对齐
  - 数据形式：`用户输入 -> 理想回答`
  - 重点：语气、长度、结构一致
- 格式约束
  - 数据形式：`输入 -> 标准格式输出`
  - 重点：字段完整、边界输入、异常输入
- 分类 / 抽取 / 改写 / 摘要
  - 数据形式：`原始输入 -> 唯一标准答案`
  - 重点：标注一致性，减少多解
- SQL / 代码 / 复杂生成
  - 数据形式：`需求描述 + 上下文 -> 高质量目标输出`
  - 重点：覆盖 schema、边界条件、失败样本
- 知识问答
  - 如果决定微调
  - 数据形式：`问题 + 必要上下文 -> 标准回答`
  - 重点：别把“裸知识背诵”当主要路径，优先考虑带检索上下文
- 偏好优化
  - 数据形式：`同一输入 -> 好回答 / 差回答`
  - 重点：偏好标准要一致

### 收束提示
真正决定后续效率的，不是先收多少条，  
而是先把“每条样本长什么样”定义清楚。

### 视觉建议
- 直接做成“目标类型 → 数据形式 → 重点”的三列表
- 尽量完整保留每一类的数据形状说明
- 不额外插入 checklist，优先保留原始映射关系

---

## 信息图 1 结尾文案
先把目标分对类，  
再判断该用什么实现手段，  
最后定义每条数据的形状。

这三步想清楚之后，  
再去讨论要准备多少数据。

# 信息图 2：第四步，用评分法快速估起步量

## 封面标题
微调数据要准备多少？  
先别追求精确，先判断量级

## 副标题
用一个简单评分法，  
快速估出第一批该准备几百、几千还是几万

## 版式建议
- 竖版信息图
- 结构：方法说明区 + 4 项评分区 + 总分对应区间 + 注意事项
- 视觉重点：让读者一眼看到“打分 → 对应起步量”

## 开场文案
现实里最重要的不是一开始精确到“多少条”，  
而是快速判断第一批数据应该落在哪个量级。

---

## 模块一：评分维度

### 模块标题
给 4 个维度打分，每项 1～3 分

### 评分项文案
- 任务复杂度  
  1 分：风格 / 格式  
  2 分：明确任务，如抽取 / 摘要  
  3 分：复杂生成 / 多轮决策

- 场景分散度  
  1 分：输入很集中  
  2 分：有几类典型场景  
  3 分：输入非常发散

- 稳定性要求  
  1 分：演示可用  
  2 分：内部试用  
  3 分：线上高稳定

- 基座能力差距  
  1 分：基座本来就会  
  2 分：能做但不稳  
  3 分：明显不擅长

### 视觉建议
- 每个维度做成单独评分卡片
- 卡片内保留 1/2/3 分三级说明
- 强调“这是起步估算，不是精确公式”

---

## 模块二：总分对应起步量

### 模块标题
打完分后，看总分落在哪一档

### 分档文案
- 4～5 分：先做 300～1000 条
- 6～8 分：先做 1000～5000 条
- 9～10 分：先做 5000～10000 条
- 11～12 分：按 1 万条以上规划，并准备更严格评测

### 强调文案
这不是科学公式，  
但非常适合接需求后的第一轮规划。

### 视觉建议
- 用阶梯图或分档条展示
- 从“几百条级”到“上万条级”逐级上升

---

## 模块三：对应的快速建议表

### 模块标题
不同目标下，第一批数据怎么准备

### 快速建议表
- 风格对齐
  - 数据：高质量问答对
  - 起步：300～1000
  - 风险：风格一致但业务正确性未必提升
- 格式约束
  - 数据：输入到标准结构化输出
  - 起步：500～2000
  - 风险：边界输入覆盖不足时容易漏字段
- 分类 / 抽取 / 摘要 / 改写
  - 数据：单任务监督样本
  - 起步：1000～5000
  - 风险：标注口径不一致
- SQL / 代码 / 复杂生成
  - 数据：需求到目标输出，包含难例
  - 起步：3000～10000
  - 风险：只覆盖简单样本，线上复杂 case 崩
- 行业问答
  - 数据：问题 + 上下文 + 回答
  - 起步：若无检索，慎重微调
  - 风险：把知识更新问题误判成微调问题

### 强调文案
这个表的作用，不是替你精确算出样本量，  
而是让你先快速落到一个合理区间。

### 视觉建议
- 直接做成“目标类型 / 数据 / 起步 / 风险”的四列表
- 起步这一列用颜色或粗体做视觉焦点
- 风险一列用浅色提示框，增强实用感

---

## 信息图 2 结尾文案
别一开始就问：  
到底是 3000 条，还是 8000 条？

先问：  
这个任务属于几百条级、几千条级，还是几万条级。

先把量级判断对，  
再进入训练和扩容。
